# 多路径调度论文审计

## 为什么这次不能再凭直觉调权重

FlowWeave 已经真实淘汰五种自定义调度：轮询、最低 RTT、预计最早送达、交付速率加权和 ACK-ECF。

最近失败的“交付速率加权”使用 `拥塞窗口 / 平滑 RTT` 估计线路能力。它在平衡 15+15 Mbit/s 时表现很好，但在 8+25 Mbit/s 异构线路中把两路数据分成接近 1:1，五轮全部不合格。根因是：

- 拥塞窗口表示“拥塞控制当前允许多少数据在路上”，不是线路真实带宽。
- 短连接仍在慢启动，窗口大小强烈受 ACK 时机和调度器自身影响。
- 两路都有流量不等于两路能同时按时完成；慢路上的低序号数据会挡住快路已经到达的数据。
- 如果调度器永远要求“有路能发就立刻发”，它会为了不空闲而把数据塞进一条预计更晚到达的路，反而降低应用层速度。

所以，下一代不能只回答“哪条路权重大”，而要回答：

> 这一小段数据放到哪条路，预计最早何时到达？如果目前只有更差的路可发，等待会不会反而更快？

## 标准给我们的边界

IETF 的 Multipath QUIC 草案 `draft-ietf-quic-multipath-21` 明确说明：

- 标准只定义多路径怎样建立、验证和管理，不替应用规定调度算法。
- 同一条 QUIC Stream 分散到多条路径后，应用交付会受到所用路径中最大延迟的影响，因为 Stream 必须按序交付。
- 选路是实现本地决策，可以从故障切换一直做到同时聚合容量。

这意味着 FlowWeave 可以研究自己的调度，但不能更改 TLS、路径验证或 QUIC 的按序语义来“制造更好看的吞吐量”。

### BLEST 的关键判断

BLEST 设快路为 `F`、慢路为 `S`，用 RTT 比例和快路拥塞窗口估算：在慢路一个 RTT 内，快路大约能发送多少数据 `X`。如果慢路新数据和已有在途数据会占满连接级发送窗口，使快路的 `X` 放不下，就跳过慢路并等待快路。

它最重要的不是某个常数，而是这个原则：

> 只有当慢路不会阻止快路完成更多有用工作时，才使用慢路。

### QAware 的关键公式

QAware 将路径 `k` 看成服务速率为 `μ_k` 的队列，队列里已有 `n_k(t)` 个包。新包的条件期望完成时间为：

```text
T_k = (n_k(t) + 1) / μ_k = (n_k(t) + 1) × S_k
```

其中 `S_k = 1/μ_k` 是 ACK 反馈得到的平均服务时间。论文使用指数平滑：

```text
Ŝ_k ← α × Ŝ_k + (1 - α) × 当前服务时间
```

论文取 `α = 0.8`。它还指出，仅靠 RTT 会慢半拍，因为 RTT 反映的是先前已经发生的排队；本地队列量能更快揭示调度器刚刚制造的负载。

### Peekaboo 的轻量学习层

Peekaboo 没有让深度神经网络直接控制每个包，而是在“等快路”和“用当前慢路”之间使用 LinUCB 在线学习。它的即时奖励来自：

```text
r = 数据包字节数 / 从发送到 ACK 的时间
```

LinUCB 对每个动作估计：

```text
预计奖励 = 当前特征·已学参数 + 探索系数×不确定度
```

第二项让尚未充分尝试的动作仍有机会被探索。论文进一步用随机调整避免确定性模型在高抖动网络中一直做同一个错误决定。

这个思路很先进，但不应成为 FlowWeave 的第一层：没有一个正确、可解释的完成时间模型时，学习器只会更快地学习测试场景里的偶然现象。

## 已淘汰候选：确认交付完成时间优先（ACK-ECF）

### 每条路径只维护三个可解释量

1. `d_i`：传播时间代理，第一版取该路径最小 RTT 的一半。最小 RTT 比平滑 RTT 更少包含排队延迟。
2. `q_i`：当前仍在途的字节数，直接使用 NoQ 已有的逐路径在途统计。
3. `b̂_i`：实际确认交付率。按路径累计 ACK 确认的字节，用时间窗口计算字节/秒，再做指数平滑。

对一个大小为 `L` 的新包，预计到达时间为：

```text
A_i = d_i + (q_i + L) / b̂_i
```

调度器选择 `A_i` 最小的路径。这个式子同时包含三件事：

- 路有多远；
- 路上已经排了多少；
- 这条路真实交付得有多快。

长期持续发送时，各路径的 `A_i` 会被自动拉近；为了让完成时间接近，分配量会自然趋向实际带宽比例，而不是固定 1:1。

### 没有可靠速率样本时

- 新路径先获得有限的探测数据，而不是永久按 `拥塞窗口 / RTT` 猜带宽。
- 探测数据属于正常传输，全部计入基准；不提供免费的隐藏预热。
- 一条路如果一直没有 ACK，拥塞窗口会自然填满，其他路径继续前进，不能因“等待学习”卡死连接。

### 等待规则

如果预计最早到达的路径暂时被拥塞窗口或 pacing 挡住，而当前能发送的另一条路预计更晚到达，则返回“暂时不发”，等待优选路径的 ACK 或 pacing 定时器唤醒。

只有当备用路径的预计到达时间不晚于优选路径的保守上界时，才立即改走备用路径。这样把 BLEST/ECF 的等待思想加入完成时间模型，同时避免无限等待。

### 第一版明确不做

- 不引入神经网络、离线训练模型或新的运行时依赖。
- 不修改 TLS 1.3、拥塞控制算法、路径验证或 QUIC Stream 按序语义。
- 不做 FEC 或关键包双发；那属于 C 组高丢包目标。
- 不根据测试场景写死 8:25、15:15 或任何带宽比例。
- 不把控制包 ACK 当作应用聚合成功；调度状态与测量仍要区分控制流量和 STREAM 数据。

## 实现前必须通过的确定性测试

1. 相同延迟、8:25 交付率时，长期分配接近 8:25，而不是 1:1。
2. 一条低延迟慢路与一条高延迟快路组合时，预测到达序列不会持续让低序号数据落后。
3. 路径速率变化后，有限时间内会转向新状态，不背负无法消除的历史积分。
4. 控制包不会污染应用调度的探测或速率状态。
5. 优选路径短暂阻塞时会等待；确认等待更差时会使用另一条 Available 路。
6. Available 存在时不使用 Backup，单路径和 NoQ 默认行为不变。
7. 错误、极小或极大的时间与字节数不会除零、溢出或恐慌。

确定性测试通过后仍不代表算法成功。它必须在原来的五对固定种子中，同时满足平衡和异构场景至少 4/5 轮比最佳单路快 15%，才允许进入长测。

## 真实实验结论：公式缺少可观测的容量输入

ACK-ECF 完成了上述确定性测试、真实 ACK 接线、控制包隔离和全量回归，再运行 40 场固定五种子短筛；原始数据保存在 `benchmark-results/2026-07-11-ack-ecf-screening.csv`。

- 平衡 15+15 Mbit/s：5/5 轮过线，相对最佳单路中位 161.2%。
- 异构 8+25 Mbit/s：0/5 轮过线，相对最佳单路中位 95.5%，最差 74.7%。
- 异构场景较慢线路承担中位约 44.0% 的首次数据，而真实容量比例只约为 `8 / (8 + 25) = 24.2%`。
- CPU 中位也从 NoQ 默认的 20.9% 增至 33.5%，单数据报重评分没有换来收益。

失败不在完成时间式本身，而在 `b̂_i` 的来源。若路径没有被喂满，ACK 字节/时间测到的是调度器分给它的 offered load（投喂量），只是容量的下界，不是容量上限：

```text
调度器先接近 1:1 分包
        ↓
两条路各自只收到接近 1:1 的待发送数据
        ↓
ACK 观测也接近 1:1
        ↓
估计器反过来认为 1:1 就是线路能力
```

这是一种自证循环。确定性测试直接把 8:25 的真实速率喂给公式，所以能证明“若容量已知，分配会收敛”；它没有证明“容量能从被调度器限流的 ACK 中辨认出来”。因此该候选按合同完整删除，没有运行长测。

下一路线必须把三种状态分开：应用没有数据、调度器没有给这条路足够数据、线路真的已饱和。只有第三种状态的样本才有资格估容量。优先研究受控 packet-pair / packet-train、pathChirp 和 BBR delivery-rate sampler；若仍需学习层，奖励也必须建立在可辨认的饱和探测上，不能继续平滑同一份循环反馈。

## 2026-07-13：合格服务量 + 有界 GSO 量子 v1 预注册

下一候选固定命名为 `qualified-service-gso-v1`，结果文件在预注册时不存在：

```text
benchmark-results/2026-07-13-b-qualified-service-gso-v1-smoke.csv
```

它使用 BBR3 作为逐路径拥塞控制，但不读取 BBR3 私有 `max_bw`、Startup、ProbeBW 或 filled-pipe 状态，也不读取 `tc` 配置。协议只使用正常应用 STREAM 数据、目标路径构包结果、拥塞/节奏阻塞和全局首次确认字节；不发送免费探针，不把重传或控制帧计为服务量。

固定算法合同如下：

1. 第二条 Available 路验证后开始 2 秒训练，恰好覆盖基准允许的预热；训练 STREAM 字节和 UDP 线速必须单独记录，不能隐藏。
2. 训练和缺少合格样本时，以 GSO batch 为单位在 Available 路之间轮换；一次选择最多产生 NoQ 现有上限的 10 个 UDP 段。选中路径受 cwnd 或 pacing 阻塞时立即尝试另一条合格路径，不绕过目标路径约束。
3. 每 `250 ms` 关闭一个路径窗口。窗口只有同时具备“发送过首次 STREAM 数据”“仍有首次 STREAM 待发送时发生过至少一次 cwnd 或 pacing 阻塞”“收到该路径包的全局首次 STREAM 确认”三项证据，才是合格饱和窗口。
4. 合格窗口样本为该路径的全局首次确认 STREAM 字节数除以窗口实际时长；首个样本直接初始化，之后固定用 `S_i <- (S_i + sample_i) / 2`。应用受限、没有调度投喂、只有控制包或只有重传的窗口不得更新 `S_i`。
5. 两条路径都有至少一个合格样本后，使用平滑加权轮换选择下一个 GSO batch；权重按 `S_i / ΣS` 计算并限制在 `[10%, 90%]`。每个 batch 后更新平滑轮换积分，不在单个 UDP 数据报之间重新排序。
6. 任一路没有合格样本时继续 batch 级轮换，不把下界冒充容量；路径关闭、Backup、单路径、A 组恢复和普通 NoQ 默认配置继续保持原行为。

实现前必须新增确定性测试，覆盖平衡与 `8:25` 合格样本的 batch 比例、未饱和窗口不更新、唯一 ACK 字节不双计、单路无行为变化、Backup 不抢 Available、被阻塞路径让行、路径关闭清场、最多 10 段的 batch 边界，以及默认配置完全不启用候选。

smoke 仍固定首种子、两个场景、release、2 秒预热、5 秒测量和 16 KiB 应用块。每个场景固定六项：Cubic 单路一/二、候选配置下的 BBR3 单路一/二、BBR3 NoQ 默认双路、BBR3 候选双路。只有同时满足以下条件才允许预注册五种子 formal：

- 候选双路在两个场景都比四项单路最佳值快至少 15%，且不比同场 BBR3 NoQ 双路慢超过 5%；
- 候选双路两条路径各承担至少 10% 首次 STREAM 数据、都保持开放，UDP/应用线速比 `<=1.20`；
- 候选配置下的两个 BBR3 单路相对 Cubic 同线路都保留至少 95%；
- 候选双路的每条路径都至少出现 2 个合格窗口，并记录训练首次字节、唯一确认字节、cwnd/pacing 阻塞、合格/不合格窗口、batch/segment 选择、重传、CPU 和 RSS；

## B 传感器门控：`cwnd-cohort-sensor-v1` 预注册

`qualified-service-gso-v1` 已证明“ACK 到达窗 + pacing 压力”不能识别异构路径服务率。下一步不是调度候选，而是一个默认关闭、完全不参与路径选择的纯观测传感器。它的唯一目的，是在接入任何 batch 权重前证明：相同线路会得到近似相同的服务率，`25 Mbit/s` 线路会显著快于 `8 Mbit/s` 线路。

### 不可变观测定义

- 名称固定为 `cwnd-cohort-sensor-v1`，不得复用 ACK-ECF、ACK 信用 v1 或 `qualified-service-gso-v1` 的状态、开关和结果文件。
- 发送 cohort 固定为 `250 ms`，按连接内首次新鲜 STREAM 字节的发送时刻对齐。每个首次发送的 STREAM 字节范围只登记一次，并绑定其原始 `PathId` 与发送 cohort；普通重传不创建新来源。
- 只有协议构包时仍存在 STREAM 数据待发，并且 `path_congestion_check` 因 `bytes_in_flight + MTU >= cwnd` 返回真实 `PathBlocked::Congestion`，该发送 cohort 才获得饱和资格。pacing 暂停、GSO batch 已满、应用暂时无数据、反放大和路径状态限制都不能晋级样本。
- 任意原包或重传首次确认某个 STREAM 字节范围时，该唯一确认量必须归还给首次发送来源的 cohort；重复 ACK 和同一范围的其他副本不得重复计数，ACK 到达时所在的时间窗和 ACK 返回路径都不得改变归属。
- cohort 在发送窗关闭后继续等待固定 `1 s` 的有界 ACK drain，然后不可逆结算。结算后的首次确认只计入 `late_acked_bytes`，不得回写服务率；未结算来源字节和迟到字节必须可审计。
- 合格服务率定义为 `qualified_acked_bytes / (qualified_cohorts * 250 ms)`。只有同时含首次发送字节和真实 cwnd-full 事件的已结算 cohort 进入分母；零样本不得回退成发送量、pacing 率、cwnd 或线路配置值。
- 传感器默认关闭；关闭时不改变 NoQ 官方低 Path ID 调度、不保留 cohort 字节范围，也不改变拥塞控制、pacing、丢包恢复、TLS、线速或应用可见行为。

## B 传感器门控：`persistent-backlog-cohort-v1` 预注册

## B 传感器门控：`declared-backlogged-epoch-v1` 预注册

这个门控仍只验证单路服务观测，不实现双路权重。它与上一版的决定性区别是：样本集合由应用在观察任何本窗网络结果之前一次性声明，协议只能计数，不能决定哪些内部窗口有资格。

### 不可变观测定义

- 名称固定为 `declared-backlogged-epoch-v1`，唯一结果文件固定为 `benchmark-results/2026-07-13-b-declared-backlogged-epoch-v1-smoke.csv`。运行前必须确认文件不存在；无论通过、算法失败还是基础设施失败，都不得覆盖、补行、改名或以同一合同重跑。

- 估计没有 `eligible` 掩码：

```text
K = {0, 1, ..., 19}
S_i = sum(k in K, U_i[k]) / (20 * 250 ms)
```

  `F_i[k]=0`、没有 cwnd-full、处于 pacing pause、没有构包机会、没有 ACK 到达或协议队列暂时为空，都不能删除该 cohort；该窗只会以零服务自然留在分子中。cwnd、pacing、ACK 到达窗、构包检查和 STREAM 队列状态只允许写审计列，不得影响 `K` 或分母。

- 传感器默认关闭；关闭时不保留 epoch、offset 或 ACK 归属状态。开启也不得修改 NoQ 官方低 Path ID 调度、拥塞控制、pacing、恢复、TLS、包边界或线速。

## B 双路候选：`staged-fresh-service-gso-v1` 预注册

下一候选把“制造可辨认样本”和“消费权重”分成两个固定阶段，不再从测量期窗口后验挑样本。名称固定为 `staged-fresh-service-gso-v1`，唯一 smoke 文件固定为 `benchmark-results/2026-07-13-b-staged-fresh-service-gso-v1-smoke.csv`；运行前必须不存在，运行后无论结果如何都不得重跑或补行。

### 固定训练和调度定义

- 应用仍使用单条确定性无限记录 STREAM。连接建立并打开两条 Available 路径后，前 `2 s` 只用于训练，不计入候选吞吐。
- 训练被预先切成八个 `250 ms` fresh-STREAM 独占槽，固定顺序为 `1,2,2,1,2,1,1,2`。每条路径恰好获得四槽、总计 `1 s`；槽序不依赖 RTT、ACK、cwnd、pacing 或发送结果。
- 训练槽内只有指定路径可承载首次 STREAM 数据；ACK、PATH_ACK、流控和其他控制帧仍按普通协议规则发送。指定路径暂时受 cwnd/pacing 阻塞时，不把该槽的首次数据改投另一条路径，也不延长或补跑该槽。
- 令 `T_i` 为路径 `i` 在自己的四个固定槽内首次发送的 STREAM 字节除以严格 `1 s`。训练结束一次性冻结：

```text
raw_i = T_i / (T_1 + T_2)
w_1 = clamp(raw_1, 0.10, 0.90)
w_2 = 1 - w_1
```

  若总训练字节为零则只允许回退 `50/50` 以完成审计，该项训练护栏必然失败；测量期不得继续学习、滑动 EWMA 或读取 `tc` 配置。
- 后续固定 `5 s` 测量期使用冻结权重，在 fresh-STREAM GSO batch 边界执行有界平滑加权轮转。每次只在当时可发送 fresh 数据的 Available 路径集合中累加权重并选择当前信用最高者；被选路径扣除当次可用权重总和，信用绝对值封顶为一个总权重，避免阻塞路径积累无界突发。单个 batch 最多沿用 NoQ 现有 10 段 GSO；不逐 UDP 包重评分。
- 候选默认关闭；关闭时普通 NoQ 低 Path ID 顺序、单路行为、恢复、TLS 和包边界必须不变。训练统计与权重不能泄漏到下一连接。

### 唯一 smoke 合同

- 仍只跑首种子和锁定的平衡/异构场景。每场固定运行 `Cubic / 仅线路一`、`Cubic / 仅线路二`、`Cubic / NoQ 双路`、`Cubic / staged-fresh-service-gso-v1 双路`，release、`2 s` 预热/训练、`5 s` 测量、`16 KiB` 记录，逐记录完整性校验。
- 基础护栏：MPQUIC 成功、全部配置路径最终开放、业务完整、writer 全程存活、测量至少 `5 s`、无发送/接收错误；单路模式不得启用候选状态。
- 训练护栏：两路 `T_i>0`；平衡 `min(T_1,T_2)/max(T_1,T_2)>=75%`；异构 `T_2/T_1>=2.0`。训练方向失败时，即使偶然吞吐过线也不得通过。
- 调度护栏：候选吞吐必须至少为同场最佳单路的 `115%`，且不得低于同场 NoQ 双路的 `95%`；两条路径各自首次 STREAM 字节至少占候选总首次字节的 `10%`；UDP 总字节/应用字节不得超过 `1.10x`。
- 两场全部通过才允许另建五种子、20 秒/64 MiB formal 文件。任一护栏失败立即停止，不调参数、不运行 formal；保留 CSV/hash 并完整删除候选调度状态、枚举、统计和脚本入口。已通过的 declared epoch 纯观测传感器是否保留，由失败机理单独决定，不能拿候选失败反向否定其已固化的单路门控。

### 唯一 smoke 结果：外生槽无偏，但 250 ms 训练估计的是瞬态而非稳态

唯一首种子 smoke 已运行并在异构场景失败。平衡场景训练为 `16.815/18.925 Mbit/s`，冻结权重 `47.0%/53.0%`；候选测量吞吐 `28.046 Mbit/s`，相对最佳单路 `192.9%`、相对 NoQ `101.0%`，两路 fresh 份额约 `50/50`，阶段通过。

异构场景中，独立单路为 `7.077/23.114 Mbit/s`，NoQ 双路为 `30.821 Mbit/s`。候选的固定训练槽却只测得 `8.801/10.991 Mbit/s`，快慢比 `1.249x`，没有达到预注册的 `2.0x`，因而冻结成错误的 `44.5%/55.5%`。测量期的 work-conserving 发送仍自然形成约 `25/75` fresh 分配并取得 `28.896 Mbit/s`，是最佳单路的 `125.0%`、线速 `1.070x`，但只保留 NoQ 的 `93.8%`，低于 `95%` 护栏。

固定槽集合没有条件抽样偏差，失败来自估计对象不对：每个 250 ms 独占槽同时包含 Cubic 恢复/增长、RTT 反馈相位和切槽后的短暂发送状态。50 ms 路径在四个离散短槽中无法表现 5 秒稳态的 `23+ Mbit/s`，所以：

```text
E[短槽 fresh service_i] != steady service_i
```

外生取样解决了“挑窗口”的偏差，却不能把非平稳短探测自动变成稳态容量。不得延长同名训练、合并槽、改用 ACK 或重跑。证据 `benchmark-results/2026-07-13-b-staged-fresh-service-gso-v1-smoke.csv` 为 44 列、9 行、3420 字节，SHA-256 `eeb7a38564d6f087021606d8e65a5fbdb0e24c2ad78a3cc3628e0deaa134bc64`；formal 不运行，候选代码按合同删除，已通过的 declared epoch 传感器保留。

同一文件中的非候选 NoQ 连续 writer 基线是新的只读线索：平衡/异构分别为 `27.757/30.821 Mbit/s`，相对最佳单路约 `190.9%/133.3%`，两路 fresh 最小份额约 `47%/23%`，线速约 `1.025x/1.026x`。它不因 staged 候选失败而失效，但也只有一个种子；若继续，必须用新名称预注册 NoQ + Cubic 连续 writer 的五种子正式 B 门控，不能把这一个基线直接宣称为 B 胜出。

## B 默认调度正式门控：`cubic-noq-continuous-formal-v1` 预注册

首种子线索把待检验假设收缩为：旧 B 长测的逐请求/响应负载反复打开 STREAM、等待响应并结束连接内事务，使默认低 Path ID 遍历主要暴露短流启动与响应串行化；当应用像真实大文件或隧道一样在一条 STREAM 上持续积压时，NoQ 默认的 work-conserving 行为可能已经可以聚合两条 Available 路。该假设不需要新的调度器，因此本轮禁止修改 NoQ 路径选择、拥塞控制、pacing、恢复或 GSO。

名称固定为 `cubic-noq-continuous-formal-v1`，唯一结果文件固定为 `benchmark-results/2026-07-13-b-cubic-noq-continuous-formal-v1.csv`。运行前文件必须不存在；开始后按每个参赛项立即落盘。无论通过、失败或基础设施中断都不得覆盖、补行、删行、改名或用同一合同重跑。

### 固定负载与矩阵

- 场景仍只有锁定的平衡 `15+15 Mbit/s, 20+20 ms, 0.1%+0.1%` 与异构 `8+25 Mbit/s, 15+50 ms, 0.1%+0.1%`，使用既定五对 netem 种子。
- 每轮只运行 `Cubic / 仅线路一`、`Cubic / 仅线路二`、`Cubic / NoQ 默认双路`。三项顺序按轮次旋转并在奇数索引轮反转；每项前重置同场同轮 qdisc 和种子。
- 每项只打开一条双向 STREAM。发送端在同一 STREAM 上连续写入带单调序号、固定 `16 KiB` 载荷和摘要的确定性记录；接收端逐记录验证顺序、内容和摘要。writer 从预热前开始，到测量结束后才收到停止信号，不允许在预热与测量之间关闭或重开 STREAM。
- 连接和 writer 先预热 `2 s`，之后按墙钟连续测量至少 `20 s`，满足 `BENCHMARK.md` 的“至少 64 MiB 或稳定发送 20 秒”分支。吞吐分子只统计接收端在测量窗口内完成并验证的整条记录载荷；发送端入队字节、窗口外 drain 和最终响应都不能计入吞吐。
- 测量窗口起止快照逐路径首次 STREAM、重传、UDP、路径状态、CPU 和 RSS。业务完整性在停止 writer、发送 FIN 并收到服务端最终记录数/字节数响应后闭合；这个收尾只验证全流，不回写窗口吞吐。

### 逐项与逐轮门槛

- 每项必须 MPQUIC 协商成功、模式/调度/拥塞控制与合同一致、writer 在测量起止都未完成、测量不少于 `20 s`、窗口内至少收到一条完整记录、最终全流顺序/内容/摘要和记录数完全一致，且所有配置路径最终开放。
- 每项 UDP 总字节/窗口内验证的应用载荷必须 `<=1.10x`。该护栏同时防止异常重传换吞吐，以及发送端队列在测量后 drain 才完成却被提前算作窗口业务。
- 双路项必须严格达到当轮两条单路中较快者的 `115%`，并且两条路径各承担至少双路窗口内首次 STREAM 字节的 `10%`；两个条件分别逐轮判断，不能用五轮中位数掩盖失败轮。
- 一个场景只有在五轮中至少四轮同时通过双路 `115%`、双路 `10%+10%` 和全部基础/线速护栏时才通过。两个场景都通过，才称 FlowWeave B 自身门槛通过，并允许预注册 Hysteria 2.9.3 BBR/Brutal 的公平 B 对照；任一场景不足 `4/5` 则 B 仍失败，不得把首种子或中位数单独宣称为胜出。

### 代码与回退边界

- 新代码只能增加通用持续 writer 测量器、报告、确定性测试、独立 CSV writer 和脚本入口；不得增加 scheduler 枚举、路径权重、训练状态或 NoQ Proto 调度补丁。
- 代码前/网络前测试必须覆盖配置边界、同一 STREAM 跨越预热与测量、接收时间窗只计完整验证记录、writer 起止存活、停止后的 FIN/最终响应闭合，以及不完整轮次不能凭空构造最佳单路或阶段通过。
- 编译、全量单测、Clippy、脚本语法和差异检查全部通过后才允许运行正式矩阵。若实现或基础设施门槛失败，先修测量器并使用全新合同/文件；若真实指标失败，保留 CSV/hash 和机理，不修改本合同补救。
