# 多路径调度论文审计

## 为什么这次不能再凭直觉调权重

FlowWeave 已经真实淘汰四种自定义调度：轮询、最低 RTT、预计最早送达和交付速率加权。

最近失败的“交付速率加权”使用 `拥塞窗口 / 平滑 RTT` 估计线路能力。它在平衡 15+15 Mbit/s 时表现很好，但在 8+25 Mbit/s 异构线路中把两路数据分成接近 1:1，五轮全部不合格。根因是：

- 拥塞窗口表示“拥塞控制当前允许多少数据在路上”，不是线路真实带宽。
- 短连接仍在慢启动，窗口大小强烈受 ACK 时机和调度器自身影响。
- 两路都有流量不等于两路能同时按时完成；慢路上的低序号数据会挡住快路已经到达的数据。
- 如果调度器永远要求“有路能发就立刻发”，它会为了不空闲而把数据塞进一条预计更晚到达的路，反而降低应用层速度。

所以，下一代不能只回答“哪条路权重大”，而要回答：

> 这一小段数据放到哪条路，预计最早何时到达？如果目前只有更差的路可发，等待会不会反而更快？

## 标准给我们的边界

IETF 的 Multipath QUIC 草案 `draft-ietf-quic-multipath-21` 明确说明：

- 标准只定义多路径怎样建立、验证和管理，不替应用规定调度算法。
- 同一条 QUIC Stream 分散到多条路径后，应用交付会受到所用路径中最大延迟的影响，因为 Stream 必须按序交付。
- 选路是实现本地决策，可以从故障切换一直做到同时聚合容量。

这意味着 FlowWeave 可以研究自己的调度，但不能更改 TLS、路径验证或 QUIC 的按序语义来“制造更好看的吞吐量”。

## 核心论文给出的数学工具

| 工作 | 核心方法 | 对 FlowWeave 的启发 | 当前取舍 |
|---|---|---|---|
| BLEST（2016/2018） | 估算慢路上的数据会占住发送窗口多久；若这段时间内快路本可发送的数据放不下，就宁可等待快路 | “等待”是合法且必要的调度动作；不能为了让每条路忙碌而制造队头阻塞 | 采用等待思想，不照搬 MPTCP 发送窗口公式 |
| ECF（2017） | 比较“现在用慢路”和“等最快路恢复”谁先完成，并用切换余量避免来回抖动 | 候选路径不只包含当前能立刻发的路；被拥塞窗口暂时挡住的快路也可能是更优选择 | 采用预计完成时间和等待判断 |
| QAware（2018） | 把每条子流视为并行服务队列，选择 `arg min (n_k+1)×S_k`；`n_k` 是排队量，`S_k` 是 ACK 反馈得到的平均服务时间 | 选路应同时看“这条路已经排了多少”和“这条路实际服务多快”，而不是只看 RTT | 将公式连续化为“在途字节 / 实际交付率” |
| DEMS（2017） | 按块感知数据边界，让各子流尽量同时完成；必要时用少量冗余换完成时间 | 真正的聚合目标是各路协同完成，而不是平均分包 | 采用“平衡预计完成时间”，暂不加入冗余 |
| SA-ECF（2018） | 在 MPQUIC 中按 Stream 完成时间和优先级调度，避免慢路拖延单个 Stream | QUIC 可以比 MPTCP 更细地按 Stream 决策 | 第一版先服务单条代理字节流，多 Stream 优先级留到代理成型后 |
| Peekaboo（2020） | 用 LinUCB 上下文多臂老虎机在线学习“等快路/用慢路”；特征含 CWND、在途包、发送窗口和 RTT，奖励来自 `包大小 / ACK 时间` | 学习层最适合修正等待决策和探索概率，而不是替代全部传输逻辑 | 先做可解释确定性内核；动态场景证明需要后再加 LinUCB |
| MARS（2023）、MAMS（2024） | 分别使用多智能体深度强化学习、移动性与无线跨层预测 | 动态无线环境最终需要更丰富状态 | 当前没有无线驱动信息和训练集，不在第一版引入 |

### BLEST 的关键判断

BLEST 设快路为 `F`、慢路为 `S`，用 RTT 比例和快路拥塞窗口估算：在慢路一个 RTT 内，快路大约能发送多少数据 `X`。如果慢路新数据和已有在途数据会占满连接级发送窗口，使快路的 `X` 放不下，就跳过慢路并等待快路。

它最重要的不是某个常数，而是这个原则：

> 只有当慢路不会阻止快路完成更多有用工作时，才使用慢路。

### QAware 的关键公式

QAware 将路径 `k` 看成服务速率为 `μ_k` 的队列，队列里已有 `n_k(t)` 个包。新包的条件期望完成时间为：

```text
T_k = (n_k(t) + 1) / μ_k = (n_k(t) + 1) × S_k
```

其中 `S_k = 1/μ_k` 是 ACK 反馈得到的平均服务时间。论文使用指数平滑：

```text
Ŝ_k ← α × Ŝ_k + (1 - α) × 当前服务时间
```

论文取 `α = 0.8`。它还指出，仅靠 RTT 会慢半拍，因为 RTT 反映的是先前已经发生的排队；本地队列量能更快揭示调度器刚刚制造的负载。

### Peekaboo 的轻量学习层

Peekaboo 没有让深度神经网络直接控制每个包，而是在“等快路”和“用当前慢路”之间使用 LinUCB 在线学习。它的即时奖励来自：

```text
r = 数据包字节数 / 从发送到 ACK 的时间
```

LinUCB 对每个动作估计：

```text
预计奖励 = 当前特征·已学参数 + 探索系数×不确定度
```

第二项让尚未充分尝试的动作仍有机会被探索。论文进一步用随机调整避免确定性模型在高抖动网络中一直做同一个错误决定。

这个思路很先进，但不应成为 FlowWeave 的第一层：没有一个正确、可解释的完成时间模型时，学习器只会更快地学习测试场景里的偶然现象。

## 下一候选：确认交付完成时间优先（工作名 ACK-ECF）

### 每条路径只维护三个可解释量

1. `d_i`：传播时间代理，第一版取该路径最小 RTT 的一半。最小 RTT 比平滑 RTT 更少包含排队延迟。
2. `q_i`：当前仍在途的字节数，直接使用 NoQ 已有的逐路径在途统计。
3. `b̂_i`：实际确认交付率。按路径累计 ACK 确认的字节，用时间窗口计算字节/秒，再做指数平滑。

对一个大小为 `L` 的新包，预计到达时间为：

```text
A_i = d_i + (q_i + L) / b̂_i
```

调度器选择 `A_i` 最小的路径。这个式子同时包含三件事：

- 路有多远；
- 路上已经排了多少；
- 这条路真实交付得有多快。

长期持续发送时，各路径的 `A_i` 会被自动拉近；为了让完成时间接近，分配量会自然趋向实际带宽比例，而不是固定 1:1。

### 没有可靠速率样本时

- 新路径先获得有限的探测数据，而不是永久按 `拥塞窗口 / RTT` 猜带宽。
- 探测数据属于正常传输，全部计入基准；不提供免费的隐藏预热。
- 一条路如果一直没有 ACK，拥塞窗口会自然填满，其他路径继续前进，不能因“等待学习”卡死连接。

### 等待规则

如果预计最早到达的路径暂时被拥塞窗口或 pacing 挡住，而当前能发送的另一条路预计更晚到达，则返回“暂时不发”，等待优选路径的 ACK 或 pacing 定时器唤醒。

只有当备用路径的预计到达时间不晚于优选路径的保守上界时，才立即改走备用路径。这样把 BLEST/ECF 的等待思想加入完成时间模型，同时避免无限等待。

### 第一版明确不做

- 不引入神经网络、离线训练模型或新的运行时依赖。
- 不修改 TLS 1.3、拥塞控制算法、路径验证或 QUIC Stream 按序语义。
- 不做 FEC 或关键包双发；那属于 C 组高丢包目标。
- 不根据测试场景写死 8:25、15:15 或任何带宽比例。
- 不把控制包 ACK 当作应用聚合成功；调度状态与测量仍要区分控制流量和 STREAM 数据。

## 实现前必须通过的确定性测试

1. 相同延迟、8:25 交付率时，长期分配接近 8:25，而不是 1:1。
2. 一条低延迟慢路与一条高延迟快路组合时，预测到达序列不会持续让低序号数据落后。
3. 路径速率变化后，有限时间内会转向新状态，不背负无法消除的历史积分。
4. 控制包不会污染应用调度的探测或速率状态。
5. 优选路径短暂阻塞时会等待；确认等待更差时会使用另一条 Available 路。
6. Available 存在时不使用 Backup，单路径和 NoQ 默认行为不变。
7. 错误、极小或极大的时间与字节数不会除零、溢出或恐慌。

确定性测试通过后仍不代表算法成功。它必须在原来的五对固定种子中，同时满足平衡和异构场景至少 4/5 轮比最佳单路快 15%，才允许进入长测。
