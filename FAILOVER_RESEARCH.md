# PTO 对冲补发：快速换网研究合同

## 真正要解决的问题

当前 FlowWeave 在主路径突然变为 100% 丢包后，仍能保持同一条 MPQUIC 连接，但应用数据通常约 4 秒后才恢复。主要时间花在：

1. NoQ 等待固定 3 秒的逐路径空闲上限；
2. 发送 `PATH_ABANDON`；
3. 再等待路径排空，才把未确认的 STREAM 数据放到备用路径重传。

用户真正感受到的是“数据停了多久”，不是“协议多快给路径判了死刑”。因此下一候选不尝试在一次丢包后准确判断线路已经死亡，而是把两个决定拆开：

- **快速数据面**：路径第一次触发 PTO 时，就在另一条路径对冲补发被卡住的数据。
- **保守控制面**：原路径继续探活，仍按现有规则决定是否最终放弃。

这使误判的最坏结果从“错误关闭一条活线路”降为“多发送一轮受限的重复数据”。

## 标准与论文依据

2026 年 3 月的 `draft-ietf-quic-multipath-21` 第 5.7 节明确指出：逐路径 PTO 到期时，QUIC 的谨慎丢包规则可能造成显著交付延迟；多路径端点可以在其他可用路径上重传被延迟包的内容。第 5.6 节同时警告，一般性重复会增加网络负担，因此本候选只在逐路径 PTO 后触发一次，而不是无条件双发。

相关研究给出的边界：

- Peekaboo 用 LinUCB 在线学习动态选择调度动作，但每轮重新学习需要额外 4 MiB、约 0.5～8 秒，不符合 FlowWeave 的短流公平合同。
- HOP 用未来容量分布和 FEC 优化迟到概率，但论文明确面向持续数秒的流量，容量估计也需要先制造排队；它不适合作为当前短流恢复机制。
- QUIC-FEC 与 FlEC 证明“按延迟风险使用冗余”是有效方向，但完整编码框架复杂。本阶段先验证更小、更容易审计的跨路径重传。
- φ accrual、SPRT 等失效检测可以输出连续怀疑度，但都需要额外采样假设。QUIC 已有按每条路径 RTT 与波动计算的 PTO，本候选先复用标准信号，不再制造第二套计时模型。

2026-07-12 继续复核先进论文后，结论没有变成“引入更复杂的模型”，反而把边界收得更清楚：

- SingleR 证明，在固定额外请求预算下，“选一个时刻、最多补发一次，并用概率控制预算”在其简化模型里可优于多次重发策略。映射到本候选，第一版取 `h = PTO`、`q = 1`：只有已经进入尾部风险的在途 STREAM 数据才补一次，正常网络额外 UDP 流量门槛负责约束误触发。
- 2026 年生产级 MRC 把快速丢包恢复、路径/端口健康探测和最终故障处理做成相互独立的原语，支持“先恢复数据，再慢慢判断路径”。但 MRC 依赖数据中心多路径喷洒、ECN、硬件 NIC 与显式路径信息，不能把它的微秒级数字直接套到公网 QUIC。
- 2026 QUIC FEC 草案把 FEC 框架与具体编码方案分开，规定 Repair ACK 不得改变 RTT、拥塞窗口或 pacing，并建议只保护关键数据。这支持 C 组以后做选择性冗余，但草案安全章节仍是 `TBD`，当前不能绑定线协议。
- MP²-QUIC 把模型预测控制与 Band-Sparse Network Coding 结合，适合视频和持续反馈场景；论文闭源、状态模型复杂，而且仍需要可信的路径输入。它保留为 C 组远期参考，不进入当前 A 组实现。

## 数学模型

设：

- `T₁`：原路径剩余交付时间；
- `T₂`：备用路径交付时间；
- `h`：等待多久后开始对冲；
- `L`：被对冲的数据量；
- `λ`：重复流量相对延迟的成本权重。

对冲后的完成时间为：

```text
T(h) = min(T₁, h + T₂)
```

理想目标是：

```text
min  E[T(h)] + λ × L × P(T₁ > h)
 h
```

SingleR 使用的一个直观预算关系是：

```text
B = q × P(T₁ > h)
```

`B` 是允许的补发比例，`q` 是到达时刻 `h` 后实际补发的概率。当前 A 组把“唯一一次真实换网不能抽签失败”放在首位，因此使用 `q = 1`，再用 `h = PTO` 和正常网络 `1.05×` UDP 门槛控制成本。若未来需要服务大量独立短请求，才考虑在跨请求层面用预算控制 `q`，不在单次故障里随机放弃恢复。

第一版不在线拟合 `T₁` 的完整分布，直接使用 QUIC 已有的逐路径 PTO：

```text
h = PTO
PTO = smoothed_rtt + max(4 × rttvar, timer_granularity) + max_ack_delay
```

PTO 会随路径延迟和抖动调整。它不是“路径死亡证明”，但已经表示继续只在原路径等待的尾延迟风险明显上升。

## 候选状态机

每条路径只增加一个内部、非线上的数据调度状态：

```text
Healthy --首次数据 PTO且存在其他可用路径----> Suspect
Suspect --对冲后发出的本路径探针得到新 ACK--> Healthy
Suspect --现有逐路径空闲规则到期------------> Abandoned
```

`Suspect` 的行为：

- 不再承载新的应用数据；
- 允许 PING、IMMEDIATE_ACK 和现有 PTO 探针，继续检测恢复；
- 把该路径仍在途的 STREAM 范围加入全局重传队列一次；
- 备用路径仍受自己的拥塞窗口、pacing 和防放大规则约束；
- 如果所有路径都可疑，必须允许至少一条路径继续发送，不能制造“自己把全部路封死”的死锁。

实现测试发现，“任意新 ACK”这个原始表述不够严格：黑洞前已经在路上的旧 ACK 可能在对冲之后才到达。如果它能解除 `Suspect`，新数据会立即重新送回黑洞，真实 smoke 中候选曾因此从约 1.48 秒恶化到约 5.87 秒。最终状态记录“触发对冲时的下一数据包号”，只有该边界之后发送的本路径探针被确认，才允许恢复普通数据调度。确定性回归 `stale_pre_hedge_ack_does_not_reactivate_suspected_path` 覆盖了这个边界。

`pto_hedge_bytes = 0` 也不能直接解释为“没有恢复价值”：纯 FIN 是必须交付的流结束标记，但没有载荷字节。确定性测试保留了 FIN-only 对冲，禁止未来用一个看似简单的字节判断误删正确行为。

不发送新的帧类型，不改变 TLS、密钥、连接 ID、路径状态信令或对端实现要求。

## 必须先修的正确性地基

NoQ 当前默认同一段 STREAM 数据只会同时存在一个有效副本，因此收到 STREAM ACK 时会直接从全连接 `unacked_data` 中减去整个范围。如果原包和对冲包都得到 ACK，旧逻辑可能重复扣账。

实现对冲前，`SendBuffer::ack` 必须返回“这次 ACK 新确认了多少字节”，而不是假设整段都是第一次确认。需要覆盖：

- 完全重复 ACK；
- 部分重叠 ACK；
- 乱序 ACK；
- 原包先到；
- 对冲包先到；
- FIN 在两个副本中出现。

只有这层确定性测试通过，才允许接入 PTO。

## 预先锁定的短筛

所有候选和 NoQ 默认基线使用相同网络参数、相同随机种子，并在每轮开始前重置队列。

### 黑洞恢复

- 主路径：20 ms、0.1% 丢包、20 Mbit/s；传输中改为 100% 丢包。
- 备用路径：80 ms、1% 丢包、20 Mbit/s。
- 固定五组种子：1101～1105 与 2201～2205。
- 原连接先正确传输 128 KiB，再断主路并传输 256 KiB。

候选通过条件：

1. 5/5 轮保持同一连接并完整校验数据；
2. 五轮恢复时间全部低于 1 秒，因此五样本 P95 也低于 1 秒；
3. 每轮备用路径在断路后都承载有效数据；
4. 保留每轮主路径丢失、备用路径发送量和总额外 UDP 字节；
5. 同轮记录 NoQ 默认基线，不允许只引用历史最好或最差值。

### 正常网络误触发

- 仍使用 20 ms / 0.1% 与 80 ms / 1% 两条线路；不制造黑洞。
- 五种子各传输 2 MiB。

候选通过条件：

1. 5/5 数据正确、连接和两条路径均可继续使用；
2. 吞吐量中位数相对同轮 NoQ 默认不得倒退超过 5%；
3. UDP 总发送量中位数不得超过同轮 NoQ 默认的 1.05 倍。

### 高丢包抗误判

- 路径一：20 ms、8% 丢包；路径二：40 ms、2% 丢包。
- 五种子各传输 2 MiB。

候选通过条件：

1. 5/5 数据正确且不能因为暂时连续丢包关闭全部路径；
2. UDP 总发送量中位数不得超过同轮 NoQ 默认的 1.25 倍；
3. 必须报告发生过多少次对冲以及对冲字节，不能把重复流量藏在普通重传里。

短筛未同时通过三组条件，就删除实现和开关，只保留原始 CSV 与失败原因。短筛通过后，才进入 `BENCHMARK.md` 已锁定的 30 秒双向正式 A 组测试。

### 原来的数学模型缺了什么

短筛使用的模型是：

```text
T(h) = min(T_original, h + T_backup)
```

它只描述“同一份工作何时出现第一个完成副本”，没有描述有序持续流前方还有多少缺口。更符合正式实验的下界是：

```text
G >= h_detect + h_feedback + S_backup(B_blocking)
```

其中：

- `G`：用户看到的最大断流；
- `h_detect`：发送端真正进入怀疑态所需时间；
- `h_feedback`：健康路径的 ACK/PATH_ACK 能重新到达发送端所需时间；
- `B_blocking`：接收端下一个可交付记录之前，仍缺失的有序字节；
- `S_backup(B)`：备用路在自身拥塞窗口、RTT、丢包和 pacing 下消化这批阻塞字节所需时间。

只把某一时刻仍留在 `sent_packets` 里的 STREAM 帧搬一次，既不能保证 `h_detect` 有界，也不能保证覆盖所有 `B_blocking`。正式数据正好给出两种反例：

1. 5 轮中 2 轮故障后对冲数为 0，说明 STREAM 帧可能先被普通时限丢包逻辑移出当前已发送包表，等 PTO 到来时已经没有可触发候选状态的帧；路径继续把新数据送入黑洞。
2. 另 3 轮虽然搬了 `0.37～0.94 MiB`，仍停顿 `2.80～5.57 s`，说明一次快照没有完整覆盖有序交付缺口，或进入怀疑态/PTO 本身已经太晚。

反向结果还暴露第三项：MPQUIC 草案允许 PATH_ACK 在任意路径返回，并只“建议”同路 ACK 有利于 RTT 测量。若健康备用路收到的数据仍经主路返回确认，主路黑洞会同时切断数据发送端的反馈。此时备用路明明能送达数据，却会因收不到确认而耗尽拥塞窗口，甚至被误判超时。多路径恢复因此必须同时设计数据面和反馈面。

### 论文复核后的约束

- `Feedback-Based Path Failure Detection and Buffer Blocking Protection for MPTCP` 把快速失效判断（FPF）和接收缓冲阻塞保护（BBP）分开，实测目标也是避免故障或弱路径造成传输中断。它支持我们把“路径是否可疑”和“有序数据缺口如何尽快补齐”拆成两个状态，而不是只做一次包副本。
- BLEST 用可用发送窗口、慢路在途字节和快路在慢路 RTT 内可发送的字节估算 HoL 风险。它不直接解决黑洞，但提醒我们必须显式记录 `B_blocking`，不能用“对冲了多少字节”代替“还缺多少可交付字节”。
- MPQUIC draft 21 第 3.1、5.4 节允许 PATH_ACK 跨路径返回，同时指出同路返回有助于正确 RTT 测量；正式反向结果说明，FlowWeave 还要增加故障韧性约束：只要收到某路径上的 ack-eliciting 数据，就必须保证该路径本身或另一条已证实存活的路径能及时带回确认。
- MRC 2026 的协议与生产论文把“可靠性探测”和正常 ACK 超时拆开，还为反向控制包维护至少覆盖每个网络平面的已验证路径集合；入站活跃但没有出站数据时，每个 RTT 主动探测反向路径。它直接支持“反馈面是独立状态机”，但其可编程 NIC、SRv6、ECN 和微秒级数据中心环境不能直接移植到公网 QUIC。
- MPQUIC 拥塞窗口预留研究能给紧急数据留出立即发送空间，但论文同时观察到吞吐与 cwnd 增长下降。当前证据尚未证明备用路是“没有发送额度”，因此不能先引入预留机制掩盖状态机缺口。

### 有向反馈闭环

对于“数据从路径 `i` 发出、PATH_ACK 从路径 `j` 返回”，控制环延迟不是路径 `i` 的单一 RTT，而是：

```text
L_ij = d_i→ + ack_delay + d_j←
```

因此路径健康必须至少区分三项证据：

1. `i` 上收到新数据，只证明数据去程可达；
2. `PATH_ACK[i]` 从 `j` 返回，只证明组合 `(i, j)` 可达；
3. 同路 `PATH_ACK[i]` 才同时检验 `i` 的返回方向并形成稳定 RTT 样本。

本轮先实现最小可审计规则：开放路径同路 ACK 优先；路径 abandoned 或没有可用 CID 时跨路回退。关闭路径时还必须把已有延迟 ACK 立即置为可发送，因为它自己的 MaxAckDelay 定时器会同时停止。确定性回归要同时证明“开放 Backup 同路返回”“关闭后跨路返回”“abandoned 路径不再发送自己的 ACK”。

非对称故障的下一层逃生机制暂不实现。只有修复后实验仍出现“去程有数据、同路 ACK 回不来”时，才研究 MRC 式反向路径探测或有界 ACK 对冲，并单独计算额外 UDP 成本。

### 时间线诊断结论

三个代表场景的原始数据保存在 `benchmark-results/2026-07-12-pto-hedge-diagnostic.csv`，算法未作任何改变：

```text
路径状态保留：继续遵守 MPQUIC 的 3 PTO。
业务数据恢复：进入 abandoned 时立即跨路对冲，不等待最终删除。
提前怀疑时钟：锚定 ACK 进展，不因同一路径继续发新包而顺延。
```

### 数学与论文复核：ACK 进展纪元不是第二套 PTO

RFC 9002 的 QUIC PTO 会在每次发送或确认 ack-eliciting 包时重启，截止为：

```text
D_quic(t) = s_last(t) + PTO
```

这对单路径谨慎探测是合理的，但种子 1103 已证明，只要黑洞路径仍能被调度器零星投喂，`s_last(t)` 就会持续前移，使标准 PTO 永远追在最后一个新包后面。

经典 TCP 的 RFC 6298 使用了不同的计时边界：第一个未确认数据启动 RTO；后续继续发新数据不会重启它；只有真正确认新数据的 ACK 才重启。RFC 7765 的 RTO Restart 甚至会继续保留“最早未确认包已经等了多久”，避免一次部分 ACK 又白等完整 RTO。FlowWeave 不照搬 TCP 的丢包宣告和拥塞窗口重置，只借用这个更适合“是否还有进展”的纪元边界：

```text
e_i = 当前无 ACK 进展纪元的起点

无在途 STREAM -> 首个包含 STREAM 的 ack-eliciting 包：e_i = send_time
收到真正新确认且仍有在途包：e_i = now
收到真正新确认且在途 STREAM 归零：e_i = None
已有在途时继续发新包：e_i 不变

D_progress = e_i + PTO_i
```

因此，在没有新 ACK 进展时：

```text
h_suspect <= PTO_i + timer_granularity
```

而不是继续受最后一次发包时刻支配。这个时钟不替换 RFC 9002 的 LossDetection，不宣告包已丢失，也不改变公开 PATH_STATUS；它只在存在另一条已验证路径时触发一次受拥塞控制的跨路对冲，并在原路径安排一个探针。

RACK-TLP 的两个不变量也应保留：损失推断必须由“真正新确认”驱动；同一时刻最多只有一个额外探针在途。T-RACKs 更直接地验证了“最后一次新 ACK 的年龄”可以作为提前恢复信号，其计时器为 `β = α × RTT + rand(RTT)`；但论文也显示数据中心里 `α=1` 或 `5` 会因误触发恶化拥塞，且不存在可直接移植到公网的固定魔数。FlowWeave 因此只借其进展锚点，不采用伪造 ACK、固定 RTT 倍数或连续重试，而继续使用 QUIC 已有的 `SRTT + 4×RTTVAR + max_ack_delay` 作为保守等待预算。

φ-accrual 失效检测给出了以后可以校准、但现在不应先实现的连续怀疑度：

```text
A_i(t) = t - e_i
S_i(a) = P(健康路径的无进展间隔 > a)
φ_i(a) = -log10 S_i(a)
```

理论上可以按“停顿成本与重复流量成本的比值”选择 `φ` 阈值；现实中公网 RTT 分布会随换网、排队和省电状态改变，当前又没有足够健康样本训练稳定的 `S_i`。第一版只记录无量纲年龄 `A_i/PTO_i` 和误触发流量，若正常与高丢包实验表明单一 PTO 阈值不够，才用原始数据离线判断是否值得升级，不能先引入在线概率模型。

MRC 2026 进一步支持状态分离：可靠性探针不消耗数据序号、不改变连接前进状态，并独立于 ACK timeout；退化路径由后台探针恢复。其生产环境甚至会在一次非拥塞丢包后立刻避开路径，但这是因为数据中心能用 packet trimming、ECN、显式链路状态和大量等价 EV 区分故障，公网 QUIC 不能照搬“一丢就封”。Varuna 2026 则提醒恢复必须区分“未完成”和“只是确认丢了”；QUIC STREAM offset 天然具备幂等语义，但统计仍只能计算首次加入重传队列的范围，不能把重复副本重复记功。

种子 1103 的保守预测是：

```text
PTO_at_fault ~= 0.540 s
visible_backup_tail ~= 0.490 s
G_predicted  ~= 1.030 s
```

它只说明新时钟有机会把 3.71 秒压到门槛附近，不构成通过承诺；两项可能重叠，也可能被最早有序缺口、备用路冷启动和调度抖动放大。真实单场已经否定该乐观估算：虽然对冲提前到 `773.733 ms`，最大断流仍有 `3545.120 ms`。

### ACK 进展真实诊断：检测成功，服务失败

原始结果保存在 `benchmark-results/2026-07-12-ack-progress-reinjection-diagnostic.csv`，SHA-256 为 `acc908ddf6560b0ce440eaa79798ef368028aa43f5145b13a00abca622bcd917`。CSV 共 2 行，每行 109 列：

- 主路 ACK 进展超时/尝试/空尝试/对冲为 `1/1/0/1`，重新加入队列 `1,117,749` 字节；标准 PTO、PTO 对冲和 abandoned 对冲全部为 0。
- 超时和对冲均在故障后 `773.733 ms`，符合故障时 `523.260 ms` PTO 加已有 ACK 年龄的预期，证明后续发包没有继续推迟新截止。
- 最大断流从 `241.577 ms` 持续到 `3786.698 ms`，共 `3545.120 ms`；相比 abandoned 候选的 `3712.711 ms` 只改善 `167.591 ms`。
- 恢复后的下一条记录是 1452。每条记录在线上占 `16,400` 字节，故障前主路共首次发送 `23,815,667` 字节，而 `1452 × 16,400 = 23,812,800`；也就是说故障时只进入该前沿记录约 `2,867` 字节。

这给出新的集合分解：

```text
U_sender = 发送端未确认的 STREAM 范围
D_ack_lost = 接收端已收到、但确认没有回来的范围
M_missing = 接收端真正缺失的范围

U_sender = D_ack_lost ∪ M_missing
```

发送端当前只知道 `U_sender`，不知道真正阻塞有序交付的 `M_missing` 在其中什么位置。本轮把约 `1.12 MiB` 的 `U_sender` 按 offset 从小到大重新排队，而记录 1452 位于发送前沿；若前面的范围大多属于 `D_ack_lost`，真正缺口就会被大量无效重传挡住。Varuna 所说的“真正未完成”和“只是确认丢失”在这里成为实测主导项，而不是论文注脚。

这还形成一个简单下界。断流已经在 `241.577 ms` 开始，检测到 `773.733 ms` 才触发；若最大断流必须小于 1 秒，恢复数据最迟约应在故障后 `1241.577 ms` 可交付，只剩约 `467.844 ms`。20 Mbit/s 备用路在完全忽略 RTT、慢启动、1% 丢包和包头时，这段时间的线速上限约 `1.17 MB`（约 `1.12 MiB`），几乎等于整个重排集合。现实网络不可能稳定把整个集合都搬完后仍过线，因此下一步必须缩小服务集合，而不是继续调快同一个全量重排循环。

### 服务集合只读时间线：跨路径累计确认才真正解锁

算法完全不变的复跑保存在 `benchmark-results/2026-07-12-ack-progress-service-timeline-diagnostic.csv`，SHA-256 为 `9c72c07f3827899514bb947b2b9bb24e50088a09e08379ac56bcf520071373ae`；表头和数据行均为 112 列：

- ACK 进展超时/对冲在 `868.854 ms`，备用路首个 STREAM 重传也在 `868.854 ms`，证明恢复数据立即进入健康路，并没有等 PathIdle 才开始搬运。
- 备用路首次代其他路径发送 PATH_ACK 在 `3405.549 ms`；这与主路 LossDetection 首次未武装 `3306.044 ms` 相邻，符合路径进入 abandoned 后才允许跨路 ACK 的现有规则。
- 最大断流从 `299.587 ms` 延续到 `3649.614 ms`，共 `3350.027 ms`。跨路径 PATH_ACK 后约 `244.065 ms` 记录 1460 恢复，接近备用路一个反馈与排队周期。
- 备用路首次发送从未发过的新 STREAM 字节要到 `8822.129 ms`；因此 `3649.614 ms` 的恢复来自被剪枝后的重传集合，不是新数据绕过旧队列。

因果顺序现在完整：

```text
868.854 ms   早期检测并开始跨路重传
3405.549 ms  备用路首次带回主路累计 PATH_ACK
3649.614 ms  接收端有序记录恢复
8822.129 ms  备用路才到达全新的 STREAM 数据前沿
```

这排除了“备用路直到 3 秒后才开始工作”，也证明仅靠提前搬运 `U_sender` 不足。最小下一步不是新增 `R/H/W` 线协议，而是利用现有 MPQUIC 能力：恢复发送端在备用路放置一次 `IMMEDIATE_ACK` 请求；接收端在同一备用路的立即响应中，除确认备用路外，还带回其他路径尚保留的累计 PATH_ACK。该行为必须满足：

- 默认关闭；
- 正常情况下仍同路 ACK 优先；
- 只有一次明确恢复事件才武装一次跨路逃生；
- 不改变主路 PATH_STATUS、PathIdle 或 3 PTO 清场；
- 统计逃生请求和实际跨路 PATH_ACK，正常网络与高丢包都要计算额外 UDP。

### 有界跨路径 ACK 逃生：确定性实现边界

默认关闭的实现已经完成，但尚未跑真实 netem。为保持历史可复现，旧 ACK 进展变体不启用它，新实验变体才同时打开四项恢复原语。

线上的触发信号仍是标准帧组合，而不是新扩展：发送端只有在 PTO 或 ACK 进展确实成功重排 STREAM 数据后，才在同一已验证备用路排入一次 `IMMEDIATE_ACK`。接收端必须在同一个数据包里同时看到 `IMMEDIATE_ACK` 和 STREAM，才把其他路径的累计 PATH_ACK 指定到这条备用路；因此普通 MTU 探测、路径验证或单独的立即确认请求不会改变 ACK 路由。

确定性测试覆盖：

- 开关默认关闭时，恢复副本仍只得到自己的同路 ACK，不发生跨路逃生；
- 开关打开但没有恢复事件时，开放 Backup 仍同路确认；
- ACK 进展恢复时，主路尚未 abandoned，备用路已经能带回主路累计 PATH_ACK，原 STREAM 只完成一次；
- 标准 PTO 恢复也会在备用路发出一次请求，避免正向 1101 被 ACK 进展状态互斥漏掉；
- 逃生 ACK 只在指定备用路宣布可发送，避免“资格判断说有 ACK、构包时又抑制”形成空包循环。

NoQ 413 项全量测试、FlowWeave 10 项普通测试和两边 Clippy 均通过。

### 有界跨路径 ACK 逃生：真实 1103 结果

原始结果保存在 `benchmark-results/2026-07-12-cross-path-ack-escape-diagnostic.csv`，SHA-256 为 `c74826b8f13f067bc2c3b40e54e0700b53ea57c735a7a9db5afaf811704c2de3`；CSV 共 2 行，每行 116 列。

- ACK 进展超时、对冲和备用路首个重传同在 `783.550 ms`；发送端备用路记录 1 次逃生请求，接收端备用路记录 1 个逃生 PATH_ACK。
- 其他路径的累计确认在 `865.403 ms` 从备用路返回，相比只读时间线的 `3405.549 ms` 提前 `2540.146 ms`，直接证明新反馈路生效。
- 最大断流从 `161.115 ms` 到 `1167.468 ms`，共 `1006.353 ms`；相对相同算法但无逃生的只读轮 `3350.027 ms` 缩短 `2343.674 ms`。
- 跨路 PATH_ACK 到恢复记录之间约 `302.065 ms`；备用路首次新数据要到 `7469.798 ms`，因此快速恢复仍是累计 ACK 剪掉已送达旧范围后，备用路继续服务真正缺口。
- 标准 PTO、PTO 对冲和 abandoned 对冲均为 0，数据完整；PathIdle 和 3 PTO 清场没有缩短。

结果非常接近，但仍比严格门槛多 `6.353 ms`，所以 A 继续判失败。这个差值小于系统调度和队列抖动，既不能据此调低门槛，也不能靠重跑挑一次幸运值。下一步先用相同候选跑正向 1101 与反向 1101：前者历史上是“PTO 已触发但服务仍慢”，后者是“反馈修复后仍约 1.35 秒”。只有两个代表场没有暴露更大的结构性缺口，才值得研究如何为 1103 增加稳定余量。

### 数学魔法复核：先问清楚，再搬整车数据

1103 的 `1006.353 ms` 不是一个不可解释的总数，可以精确拆成三段：

```text
等待恢复动作：783.550 - 161.115 = 622.435 ms
取得累计反馈：865.403 - 783.550 =  81.853 ms
反馈后补齐缺口：1167.468 - 865.403 = 302.065 ms

总计：622.435 + 81.853 + 302.065 = 1006.353 ms
```

这说明继续优化“跨路 ACK 怎么返回”已经不是首要问题；它只用了约 `81.85 ms`。剩余风险主要在恢复动作开始前的等待，以及累计反馈回来后真正缺口的服务时间。

复核 RACK-TLP、Google Reactive、MPTCP TLP、FUSO、MPTCP Data ACK、MRC 和 Varuna 后，几项工作虽然场景不同，却共同指向同一个结构：

- RACK-TLP 每次只允许一个额外探针在途，并用探针带回的 SACK 证据判断更早的数据；它证明“先花一包取得信息”可以避免直接进入昂贵的 RTO 恢复。
- `Probe or Wait` 在 MPTCP 中把 TLP 的恢复包同时重注入备用子流，证明探针不应继续只走疑似故障的原路；但它仍面向尾部短流，不能直接解释持续流的无 ACK 进展状态。
- FUSO 在有备用拥塞窗口时优先复制最老的未确认数据，并限制每段最多被主动保护一次；它的目标是先修最可能阻塞交付的位置。不过 FUSO 只在没有新数据可发时使用空闲窗口，不能原样套到 FlowWeave 的持续流。
- MPTCP 的 Data ACK 是跨所有子流的累计数据级确认，值就是“接收端下一字节还期待什么”。这与前文的 `R(t)` 完全同构，能直接区分数据已经收到与子流 ACK 丢失。
- MRC 把可靠性探针做成独立原语：探针不消耗数据序号、不改变连接前进状态，只要求接收端返回当前 SACK；恢复又明确优先最早缺失序号。它给出了当前最清楚的架构参照。
- Varuna 使用接收端完成记录区分“操作已经执行、只是返回丢了”和“操作确实没执行”。它比字节流需要的机制更重，但再次证明恢复前先消除这类不确定性是有价值的。

因此下一代最小候选不应是“把完整恢复时钟提前几毫秒”，而应是两级动作：

```text
阶段一：有界可靠性探针
  只在备用路发送最多一包最早未确认 STREAM 数据，并请求立即反馈；
  不暂停原路，不改变 PATH_STATUS，不把整个 U_sender 重新排队。

阶段二：完整恢复
  到现有 ACK 进展 PTO 仍没有新进展时，才执行当前已经验证的完整跨路恢复；
  标准 PTO、PathIdle 和 3 PTO 状态保留继续不变。
```

一包探针有两种结果，两种都产生有效信息：

```text
探针范围属于 M_missing：它直接补上一个真实缺口；
探针范围属于 D_ack_lost：它虽然是副本，却逼出跨路径累计 PATH_ACK，剪掉大批无效旧范围。
```

可以把一包探针 `p` 的价值粗略写成：

```text
V(p) = P(p 属于 M_missing) × 队头修复收益
     + P(p 属于 D_ack_lost) × 不确定集合缩减收益
     - 一包额外流量成本
```

RACK-TLP 选择最高序号段，是为了让它的 SACK 对所有更早段形成时间证据；FUSO 选择最老未确认段，是为了直接修最早阻塞。FlowWeave 已经能在同一次响应里取得原路径完整 PATH_ACK 范围，因此第一候选倾向“最早未确认的一包”：最坏情况下它能直接补队头，反馈信息仍由 PATH_ACK 范围提供。这个选择必须先用确定性反例证明，不能只凭直觉接线。

探针时间 `tau_probe` 暂不选常数。对任一场景 `s`，令：

```text
R_s = 从探针发出到有序记录恢复的剩余时间
D   = 1000 ms 的严格断流预算

要达标必须满足：
tau_probe_s - gap_start_s < D - R_s
```

1103 当前 `R_1103 = 383.918 ms`，所以在这次样本里，探针最迟需要在故障后 `777.197 ms` 发出；现有完整恢复在 `783.550 ms`，恰好晚 `6.353 ms`。但单样本不能估计 `R_s` 的尾部，也不能据此硬编码 `777 ms`。先跑正向和反向 1101，是为了取得至少三种代表状态下的 `R_s`：

- 若三场的恢复余量都稳定且主要只差等待时间，优先实现“一包可靠性探针”；
- 若某场在取得累计确认后仍长期阻塞，说明发送端 PATH_ACK 仍不是充分信息，应升级为协商后的数据级 `STREAM_PROGRESS`，先只报告每条流的连续接收 offset；
- 若某场连备用路反馈本身都不稳定，先修反馈路径，不得用更早定时器掩盖。

`STREAM_PROGRESS` 的第一版只需表达 MPTCP Data ACK 同等级的信息：

```text
stream_id
contiguous_offset  # 接收端已经无洞连续收到的位置，等价于 R(t)
```

发送端只允许该值单调增加，且不得超过已经发送的最高 offset；它可以释放数据级重传负担，但原路径包号、拥塞和 RTT 账仍按 PATH_ACK 独立处理。只有代表场证明一包探针的信息仍不够，才值得承担这个新帧和协商状态的复杂度。

本轮明确排除三条伪捷径：不缩短 PathIdle，不缩短 3 PTO 清场，不用 FEC 掩盖 ACK/数据状态不确定性。FEC 可以降低未来新数据丢失，却不能告诉发送端过去的 `U_sender` 中哪些字节其实已经到达。

### 双向 1101 代表场否决“直接提前一包探针”

完全相同的 `CrossPathRecoveryWithAckEscape` 候选已复跑正向和反向种子 1101。原始结果保存在 `benchmark-results/2026-07-12-ack-escape-representative-diagnostic.csv`，SHA-256 为 `b8a3d5a6e844e119b8d45f2efd30879968e47a8400012c28ea006a5ded5809ee`；表头、正向行和反向行均为 116 列。

结果不是“都差一点”，而是两个完全不同的状态：

```text
正向最大断流：12025.460 ms，失败
反向最大断流：  381.864 ms，单场通过
```

正向时间线中，主路 PTO、恢复和对冲在故障后 `461.644 ms` 已发生，备用路同路 PATH_ACK 在约 `306.827 ms` 已出现，跨路径累计 PATH_ACK 在约 `545.139 ms` 已回来，备用路在 `1444.275 ms` 已开始发送新数据；最大断流却从 `2179.581 ms` 持续到 `14205.041 ms`。因此更早发一包最多只能优化首轮恢复，无法解释、更无法消除随后出现的 12 秒第二缺口。

首轮恢复与第二缺口之间还有两项结构性证据：

- 初次对冲载荷约 `659763` 字节，但恢复记录起始 offset 比黑洞前主路首次数据多约 `790958` 字节，第二缺口很可能落在初次 PTO 快照之后形成的范围。
- 备用路自身又丢失 79 个包，最终拥塞窗口降到约 `20076` 字节。恢复副本或其 ACK 再次丢失，可能把真正的队头缺口留在“已从首轮队列弹出、但尚未被确认”的状态。

这把下一步从猜机制改成了一个可证伪的状态识别问题。对唯一的持续数据流定义：

```text
R(t) = 接收端已经无洞连续消费到的 offset
H(t) = 接收端见过的最高 offset
A(t) = 发送端已经连续确认到的 offset
Q(t) = 发送端当前最低待重传 offset
Bq(t) = 当前待重传总字节数

接收端洞后缓存量：J(t) = H(t) - R(t)
反馈信息欠账：    I(t) = max(0, R(t) - A(t))
重传目标偏差：    E(t) = Q(t) - R(t)  # Q 存在时
```

三组符号对应三类不同修复，不能混做：

1. 若 `J >> 0` 且 `Q` 为空或 `E > 0`，接收端已经压着洞后数据，发送端却没有瞄准真实队头；需要类似 MPTCP Data ACK 的 `STREAM_PROGRESS`，最小只报告 `stream_id + contiguous_offset`。
2. 若 `J >> 0` 且 `Q <= R`，发送端正在补正确位置但 `R` 长时间不动；优先处理备用路上恢复副本再次丢失，而不是增加新线协议。
3. 若 `J ≈ 0` 且 `H` 也停止，说明备用路没有继续把后续数据送到接收端；应查发送或拥塞控制停滞，数据级进度帧不会凭空创造吞吐量。

2026 年 MRC 原文进一步确认了这个分解：它让接收端 SACK 同时携带累计位置、位图起点和洞后到达位图，接收端优先报告最老的不完整区域，发送端优先重传最早缺失 PSN；独立可靠性探针只负责索取这份状态，不消耗数据序号，也不改变连接前进状态。CSA-UD 则用接收端位图扫描缺失 PSN 并发选择性 NACK，同时把重传队列置于新数据之前。两者场景虽是数据中心 RDMA，但共同支持同一条架构原则：先把“哪里缺”变成显式证据，再决定补多少；不能只靠更早超时重复搬运同一份不确定集合。

- CSA-UD 的接收端位图、选择性 NACK 与重传优先队列：<https://arxiv.org/abs/2606.20582>

## 架构与安全边界

- 候选必须通过一个默认关闭的通用 NoQ 配置启用，以便同一版本公平比较官方基线；失败后连配置一起删除。
- 只复制已经由本地应用写入且已经加密发送过的 STREAM 范围，不产生新应用数据。
- 每次怀疑周期最多加入一轮在途范围；重复范围由区间集合合并。
- 备用路径的发送继续受拥塞控制约束，不能把 PTO 当作绕过拥塞窗口的理由。
- 对端故意不确认数据时，最多造成一轮受拥塞控制的跨路径副本，不能形成无限放大。
- 路径恢复后允许重新参与发送；不能把一次误判永久写入公开 `PATH_STATUS`。
- NoQ 补丁必须集中、可删除，不把 FlowWeave 上层实验名称写进协议线格式。

## 不属于本候选的内容

- 不引入机器学习、新依赖或跨连接历史。
