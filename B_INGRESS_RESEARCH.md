# B 组分离 ingress 可观测场：v1 预注册

## 目标与边界

现有 B 候选连续失败在同一个可观测性问题：普通 ACK 或某路径的完成量只能看到 `min(调度器投喂量, 路径服务量)`，不能证明路径已经被填满。`CAPACITY_PROBE_RESEARCH.md` 的短 Packet Train、Chirp 和 BBR3 只读资格也都已经按各自历史合同否决并删除，不能换名字复跑。

本 v1 只建立新的物理测量场，不实现调度器、不修改 NoQ、不产生可直接用于 B 晋级的容量数字。它必须先证明在真正分离的发送端/接收端网络命名空间中，接收侧 ingress 能稳定区分以下三种状态：

1. 应用欠载：接收速率跟随发送端投喂，明显低于线路上限；
2. 线路饱和：发送端持续超额投喂，接收 ingress 被物理整形上限截住；
3. 交叉流量：单个 FlowWeave 流的接收速率下降，但同一入口的总 ingress 仍接近线路上限。

只有这三种状态都能被同一测量器正确分账，后续候选才允许讨论“在预热阶段用有用应用数据受控填满路径，再把接收侧饱和证据反馈给发送端”。本文件不预注册那个候选，也不允许用本诊断的已知 `tc` 配置直接喂给正式调度器。

## 固定拓扑

- 外层使用 `unshare --user --map-root-user --net` 建立一次性发送命名空间；其网络命名空间 inode 必须不同于启动脚本的父命名空间。
- 在外层内部再建立一个独立接收网络命名空间；接收进程的网络命名空间 inode 必须与发送端不同。
- 两条路径使用两对独立 veth，不再复用同一个 loopback 设备：
  - 线路一：发送端 `fwbs1` `10.241.1.1/30`，接收端 `fwbr1` `10.241.1.2/30`；
  - 线路二：发送端 `fwbs2` `10.241.2.1/30`，接收端 `fwbr2` `10.241.2.2/30`。
- 两条发送侧 veth 的 MTU 固定为 1500。线路一使用 TBF `8 Mbit/s`、netem `15 ms / 0.1% / seed 1101`；线路二使用 TBF `25 Mbit/s`、netem `50 ms / 0.1% / seed 2201`。
- TBF 平均桶固定 `32 KiB`，峰值只比目标高 `1 Kbit/s`、`minburst 1600`、`latency 100 ms`，与最后一轮历史容量探针的整形纪律一致；差别是接收时间和计数发生在独立 veth ingress，而不是同一回环设备。
- 所有进程和接口随外层命名空间退出而销毁，不创建 `/run/netns` 持久对象，不修改真实网卡、路由或防火墙。

## 固定负载与分账

每个 UDP 载荷固定 1200 字节，使用版本化头、阶段号、流号、序号和固定内容护栏。发送端按 IPv4 + UDP 的 28 字节固定头计算目标线速；每个阶段先预热 1 秒，再持续测量 2 秒。接收端在本命名空间内从 `/proc/net/dev` 读取 veth `rx_bytes/rx_packets`，同时按三个独立 UDP socket 分账应用流量：线路一主流、线路二主流、线路二交叉流。

三个阶段固定为：

```text
阶段                    线路一主流       线路二主流       线路二交叉流
application-limited      4.0 Mbit/s       12.5 Mbit/s      0
line-limited            12.0 Mbit/s       37.5 Mbit/s      0
cross-traffic            0                10.0 Mbit/s      25.0 Mbit/s
```

发送端实际成功排队的包数、发送错误、接收端每流正确包数、格式/内容/来源错误、逐接口 ingress 字节/包、阶段实际时长、CPU 和 RSS 都必须写入 CSV。接口总 ingress 与单流接收不能混成一个数字；交叉流量也不能隐藏。

## 唯一 smoke 与门槛

预注册结果文件在实现前不存在：

```text
benchmark-results/2026-07-13-b-separated-ingress-observability-v1-smoke.csv
```

唯一 smoke 必须完整运行三个阶段，并同时满足：

- 安全护栏：父/发送/接收三个网络命名空间关系正确，两对指定 veth 与地址存在；所有发送错误、格式错误、内容错误、来源错误和未知阶段包均为零；
- 欠载阶段：两条接口 ingress 分别在 `4.0` 和 `12.5 Mbit/s` 的 `±10%` 内，并且各自低于物理上限的 75%，证明测量器不会把应用投喂误称为容量；
- 过载阶段：两条接口 ingress 分别在 `8` 和 `25 Mbit/s` 的 `±10%` 内；线路二/线路一比率相对 `25/8 = 3.125` 的误差不超过 10%；发送端实际投喂至少为各自上限的 125%；
- 交叉流量阶段：线路一 ingress `<0.1 Mbit/s`；线路二总 ingress 在 `25 Mbit/s` 的 `±10%` 内；线路二主流单独接收速率不得超过总 ingress 的 75%，交叉流接收速率必须高于 `0.1 Mbit/s`，且两条应用流合计与接口 ingress 的差异不超过 10%；
- 每阶段接收正确包的估算 IP/UDP 线速与对应接口 ingress 差异不超过 10%，避免只看接口噪声或只看 socket 丢包；
- CSV 三个阶段全部闭合，CPU、RSS、TBF/netem 和接口统计均可检查。

若 smoke 任一条件失败，保留 CSV，不重跑、不改容差、不换持续时间或速率；先根据哪一种状态不可辨认决定是修测量拓扑、停止 receiver-ingress 路线，还是提出全新的诊断。只有 smoke 全部通过，才允许为“有用数据预热 + ingress 饱和反馈”建立独立候选名、独立协议设计和独立 B 结果文件。

本诊断使用明文 UDP 是为了隔离物理测量器，不能作为加密、认证、代理或 Hysteria 对照证据。正式 FlowWeave 仍必须使用标准 TLS 1.3、精确证书校验和 BENCHMARK.md 的完整 B 合同。

## v1 唯一 smoke 结果：接收分账成立，题目生成器失败

v1 已按固定种子完整运行三个阶段，没有重跑。结果为：

```text
阶段                    offered 线路一/线路二/交叉    ingress 线路一/线路二
application-limited      3.998 / 12.501 / 0            4.045 / 12.694 Mbit/s
line-limited            11.999 / 37.496 / 0            6.841 / 15.673 Mbit/s
cross-traffic            0 / 10.000 / 25.001            2.976 / 16.925 Mbit/s
```

欠载阶段通过：接口 ingress、按流 socket 计数和发送投喂在 10% 内一致，证明独立接收命名空间与 `/proc/net/dev` 分账有效。所有发送、格式、内容、来源和未知阶段错误均为零，父/发送/接收三个网络命名空间也正确分离。

过载阶段失败不是 receiver 把应用受限误判成线路上限，而是 v1 的 TBF 峰值桶没有产生预注册的物理题目。`peakrate 8001/25001 Kbit/s + minburst 1600` 在当前内核计时粒度下只交付约 `6.841/15.673 Mbit/s`。线路二叶子 netem `limit 10000` 同时积压约 11 MiB；400 ms 阶段冷却无法排空，所以交叉流量阶段仍在接收上一阶段的旧包，新的阶段三流量被队头阻塞，线路一也残留 `2.976 Mbit/s`。这解释了阶段三按流计数全部为零而接口仍有流量，不是计数器漏包。

永久证据：

```text
benchmark-results/2026-07-13-b-separated-ingress-observability-v1-smoke.csv
4 行、1601 字节
SHA-256 6974ff37c95edbcc6d65132148cf9e8603dcc6279f028aef8af523e6d158be0e
```

v1 按合同否决，不重跑、不覆盖、不改变其 CSV。它不能证明 receiver-ingress 路线失败，因为已确认欠载分账正确，而过载的已知物理上限没有被整形器兑现。

## v2 预注册：去掉峰值桶并有界叶子队列

v1 后只在临时命名空间做了不写结果的整形器校准。相同 12/37.5 Mbit/s 超额投喂下：

```text
TBF rate + 32 KiB burst，无 peakrate：8.003578 / 25.013007 Mbit/s
HTB rate/ceil + 32 KiB burst：          7.999005 / 25.004103 Mbit/s
```

两者都能真实生成 8/25 Mbit/s；HTB 额外报告量子过大警告，因此 v2 固定选择改动更小的 TBF。新结果文件在预注册时不存在：

```text
benchmark-results/2026-07-13-b-separated-ingress-observability-v2-smoke.csv
```

v2 只允许两项基础设施修正：

- 从两条 TBF 删除 `peakrate` 与 `minburst`，保留 `rate 8/25 Mbit/s`、`burst 32 KiB`、`latency 100 ms`；
- 两条 netem 的 `limit` 从 10000 固定降到 256 包，使最坏积压可在既有 400 ms 阶段冷却内排空。

命名空间、veth、地址、MTU、延迟、丢包、种子、三个负载阶段、1 秒预热、2 秒测量、CSV 字段和全部门槛与 v1 完全相同。v2 不改发送速率、不延长测量、不放宽误差、不删除交叉流量。唯一 smoke 失败仍永久保留并停止；只有 v2 全部通过，才允许预注册真正的 B 传输候选。

## v2 唯一 smoke 结果：三种状态全部可辨认

v2 已完成唯一一次固定种子 smoke，三个阶段全部通过：

```text
阶段                    offered 线路一/线路二/交叉    ingress 线路一/线路二
application-limited      3.997 / 12.497 / 0            4.039 / 12.635 Mbit/s
line-limited            11.957 / 37.486 / 0            8.009 / 25.028 Mbit/s
cross-traffic            0 / 9.995 / 24.994             0 / 25.006 Mbit/s
```

过载阶段的 ingress 比率为 `25.028 / 8.009 = 3.125`，与真实 `25/8` 基本一致；两条发送投喂都超过物理上限 125%。交叉流量阶段，线路二主流单独只收到 `6.012 Mbit/s`，交叉流收到 `18.718 Mbit/s`，两者合计接近接口总 ingress `25.006 Mbit/s`。这直接证明“单流完成量低”不等于“线路容量低”，但饱和时同入口总服务量仍可观测。

三个阶段的接口与 socket 线速分账均在 10% 内，发送、格式、内容、来源和未知阶段错误全为零；父/发送/接收命名空间分离正确，阶段结束时两条 qdisc backlog 都为零。CPU 为 `10.411%`，控制器/接收器峰值 RSS 为 `2656/2596 KiB`。
