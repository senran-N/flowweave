# 项目地图

## 项目是做什么的

这是一个实验性的多路径传输项目，暂定名为 FlowWeave / 织流。研究方向是：把 Wi-Fi、移动网络或多条宽带连接编织成一条连接。

长期目标同时包含三件事：

- A：网络切换时，正在进行的连接不断开。
- B：多条线路可以共同传输，提高总速度。
- C：丢包严重时，实时数据仍尽量保持低延迟。

目前处于第二阶段 B 的候选调度筛选，不是可日常使用的代理软件，也没有宣称性能超过 Hysteria 2、VLESS 或其他现有方案。

## 当前技术选择

- 核心语言：Rust。
- MPQUIC 实验底座：NoQ 1.0.1 加 FlowWeave 的最小可审计补丁；NoQ 默认策略仍保持官方发包顺序，另有一个等待真实筛选的 ACK-ECF 候选。它是实验发动机，不是永久绑定的协议内核。
- 加密：标准 TLS 1.3；实验使用内存中临时生成、被客户端精确校验的证书。
- 普通功能实验：Linux 回环地址，只监听本机，外部设备无法连接。
- 坏网络实验：rootless 隔离网络命名空间加 Linux tc netem。网络限制随脚本退出而消失，不修改真实网卡。
- 依赖策略：没有下载新的第三方包；根项目直接引用已经锁定并本地替换的 `noq-proto` 1.0.1，只为使用调度配置类型。
- 公平基准：固定对手为 Hysteria 2.9.3；最终结论必须满足 BENCHMARK.md，不能根据跑分临时改规则。

## 当前进度

### 第一阶段：真正的 MPQUIC 双路径

2026-07-11 已完成并实际运行：

- 客户端与服务端成功协商 MPQUIC。
- 同一连接建立两条路径，不是两个互不相干的普通 QUIC 连接。
- 两条路径分别承载了实际数据。
- 主路径被正常关闭后，原连接继续通过第二路径传输。
- QUIC Datagram 能够往返。
- 第三条超限路径、格式错误的数据和超出实验上限的参数会被拒绝。

### 第二阶段 A：可重复坏网络实验场

2026-07-11 已建立并实际运行：

- 两条线路可以分别设置带宽、延迟和丢包。
- 每个对照项开始前都会重置网络队列；丢包模拟使用固定随机种子，减少“不同选手抽到不同随机题目”的偏差。
- 能对比仅线路一、仅线路二和 NoQ 默认多路径。
- 能记录断路恢复时间、流吞吐量、Datagram P50/P95/P99、应用层丢失、额外流量和每条路径的统计。
- 测试只允许在一次性隔离网络空间运行；即使手动伪造启动标记，只要仍在主网络空间就会拒绝修改队列。

连续三轮固定场景的结果：

- 20 ms / 0.1% 与 80 ms / 1% 两路同时可用时，默认多路径吞吐量约 10.15～10.45 Mbit/s，两条路径都显著承载了流量。
- 实验把单路径空闲判定上限设为 3 秒；主线路在传输中变为 100% 丢包后，原连接约 3.99～4.40 秒恢复，数据改走备用线路。
- 8% 与 2% 高丢包场景中，默认多路径吞吐量三轮约为 3.82、2.40 和 8.10 Mbit/s；相对当轮最佳单路径，两轮更快，一轮只剩约 46%。默认多路径的 Datagram 应用层丢失约为 13.89%～18.06%。
- 8 Mbit/s 低延迟线路与 25 Mbit/s 高延迟线路组合时，默认多路径相对当轮最佳单路径约在 -3.2%～+1.9% 之间，没有稳定聚合收益。

这些数字是实验基线，不是产品性能承诺。固定随机种子能减少丢包题目差异，但不能消除 QUIC 拥塞窗口、重传时机、操作系统调度和短测试数据量造成的波动。高丢包场景的三轮结果得出了不同的吞吐量排名，因此后续任何算法比较都必须看多轮中位数和波动范围，禁止挑选最好看的一次。

### 第二阶段 B：调度与公平基准

2026-07-11 已锁定公平验收合同：

- A、B、C 都有固定网络条件、测试时长和数字门槛。
- 每个场景至少运行 5 轮并保留全部原始结果。
- Hysteria 2.9.3 的 BBR 和 Brutal 两种模式都必须测试。
- 远程服务器只证明真实部署可用，不能单独证明两条物理线路的带宽聚合。
- NoQ Proto 1.0.1 已原样引入并单独提交，随后增加了可撤销的最小调度补丁。
- FlowWeave 当前对外提供 NoQ 默认基线和唯一候选 ACK-ECF；轮询、最低 RTT、预计最早送达和交付速率加权都已按数据删除，不留失败算法开关。
- ACK-ECF 已按 BLEST、ECF、QAware 和 DEMS 的启发实现：每条路径只从真实应用数据 ACK 学习交付率，以 `最小 RTT/2 + (在途字节 + 新包字节)/确认交付率` 预测到达时间；新路径使用最多 8 个正常应用包的有偿探测，预测差距明显时允许等待优选路径。
- 控制包不会训练交付率；路径迁移会清空旧 4-tuple 的速率样本；单路径、只有一条 Available 路和 NoQ 默认策略继续走原发送路径；ACK-ECF 的应用数据按单个 UDP 数据报重新评分，避免大 GSO 批次一次灌错几十包。
- 数学与集成测试覆盖 8:25 容量比例收敛、异构延迟到达顺序、速率变化、极值安全、等待/回退、Backup、单路径、控制包隔离和真实 ACK 学习。接入后 NoQ 396 项单元测试、3 项文档测试，FlowWeave 4 项普通测试和网络实验防伪回归全部通过，两边 Clippy 零警告。
- NoQ 路径统计已把“首次发送的 STREAM 数据”和“重传数据”分开，避免用重复流量冒充两路有效聚合。
- 首次数据统计有确定性测试；本次清场后 NoQ 380 项单元测试、3 项文档测试，FlowWeave 4 项普通测试和 1 项网络实验防伪回归均通过，两边 Clippy 零警告。

2026-07-11 的隔离坏网络单轮初筛结果：

- A 换网：四种策略都在原连接恢复，但仍约需 4.02～4.49 秒。根因仍是 3 秒路径空闲判定，不是数据调度顺序。
- B 异构线路：最佳单路为 8.38 Mbit/s；NoQ 默认 7.79、轮询 8.59、最低 RTT 8.45、预计最早送达 10.92 Mbit/s。预计最早送达本轮约高 30%，值得进入五种子复测，其余策略尚无稳定聚合证据。
- C 高丢包：预计最早送达本轮流吞吐量最高，为 7.19 Mbit/s；但四种多路径策略的 Datagram 丢失仍约 9.72%～15.28%，明显不合格，不能靠调度冒充冗余或 FEC。

这些都是单轮初筛数据，不是最终性能结论，因此随后按固定 5 个种子进行了候选筛选；失败候选必须删除，不能永久堆成开关。

随后完成的 2 MiB 五种子候选筛选，原始 60 轮保存在 `benchmark-results/2026-07-11-scheduler-screening.csv`：

- 平衡 15+15 Mbit/s：默认、轮询、最低 RTT、预计送达四种策略都是 5/5 轮比最佳单路快至少 15%，每条路首次数据占比均远高于 10%。
- 异构 8+25 Mbit/s：默认 0/5、轮询 3/5、最低 RTT 1/5、预计送达 3/5 达到“快 15%”。没有策略通过 4/5 阶段门槛。
- 最低 RTT 的证据最弱，已从 NoQ 补丁、FlowWeave 接口和实验矩阵完整删除。
- 轮询的异构中位数约为最佳单路的 122.2%，预计送达约为 116.2%；但两者最差轮分别只有 95.6% 和 91.9%，只能进入 20 秒/64 MiB 长时复赛，不能宣称稳定聚合。
- 所有多路径候选每轮较少线路都承载至少约 35.8% 的首次应用数据；这证明问题不在“第二条路没用上”，而在短流、拥塞窗口和异构线路调度效率。

随后按正式时长完成 release 长时复赛：每场单连接预热 2 秒，再计时至少 20 秒；50 场原始结果保存在 `benchmark-results/2026-07-11-scheduler-long.csv`：

- 平衡 15+15 Mbit/s：轮询 5/5 轮达到“比最佳单路快 15%”，吞吐量中位 18.76 Mbit/s，相对最佳单路中位 152.4%，每轮较少线路首次数据占比至少 30.4%。
- 平衡线路中的 NoQ 默认和预计送达都长期偏向一条路：相对最佳单路分别只有约 102.4% 和 105.3% 中位数。
- 异构 8+25 Mbit/s：轮询仅 1/5 轮过线，中位相对最佳单路 100.1%，最差 86.6%；预计送达 0/5 轮过线，中位 95.8%，最差 86.7%。
- 轮询的问题是按批次平均分配，无法按 8/25 Mbit/s 的容量比例分配，慢路会造成乱序和队头等待；预计送达的问题是形成路径偏置，缺乏持续探索，常退化成最佳单路甚至更慢。
- 两个候选都没有同时通过平衡和异构场景，必须删除。下一代候选应按每条路径的实际交付速率加权，并保证低频探索，不能继续微调这两个失败公式。
- 长测记录了 CPU、RSS、首次数据、重传和线速。CPU 可逐场比较；RSS 因同一进程的分配器会保留历史内存，目前只用于发现异常增长，最终跨程序内存比较必须隔离进程。

随后实验“交付速率加权”：用 `拥塞窗口 / 平滑 RTT` 作为权重，平滑按比例发包，并给慢路保留至少约 10% 探索机会。修复了路径建立控制包污染调度积分的问题后，全部确定性测试通过，再运行 40 场五种子短筛；原始结果保存在 `benchmark-results/2026-07-11-delivery-rate-weighted-screening.csv`：

- 平衡 15+15 Mbit/s：5/5 轮达到“比最佳单路快 15%”，吞吐量中位 20.65 Mbit/s，相对最佳单路中位 171.6%。
- 异构 8+25 Mbit/s：0/5 轮过线，吞吐量中位 12.06 Mbit/s，相对最佳单路中位 96.2%，最差 83.6%。
- 两条线路首次数据仍接近各占一半，证明 `拥塞窗口 / RTT` 没有学到 8/25 的真实容量比例；UDP 线速中位达到应用数据的 1.09 倍，额外重传也没有换来收益。
- 该候选没有通过短筛，因此没有浪费时间运行长测，并已从 FlowWeave 和 NoQ 分叉完整删除。
- ACK-ECF 不再使用瞬时拥塞窗口冒充线路带宽；它已经完成论文审计、确定性测试和真实 ACK 接线，但尚未运行固定五种子短筛，因此目前只能叫“候选”，不能宣称聚合成功。

## 这些结果说明什么

- NoQ 默认行为确实能让两条 Available 路径同时发送数据。
- “两路都有流量”不等于“带宽聚合”：异构线路中几乎没有超过最佳单路。
- 单次吞吐量变好不等于算法已经变好：高丢包排名会波动，而且 Datagram 丢失仍然严重。
- 约 4 秒的黑洞恢复证明 A 有可行底座，但距离优秀的换网体验仍有明显差距。
- B 已经淘汰四种自定义调度；ACK-ECF 是当前唯一等待同一五种子门槛检验的候选，仍没有稳定带宽聚合结论。
- A 的约 4 秒恢复瓶颈不在调度器，而在失效检测；必须单独做快速、抗误判的路径健康机制。
- C 仍需要关键 Datagram 双发或 FEC；普通路径调度无法把 10% 左右应用层丢失变成合格结果。

## 主要文件

- Cargo.toml：项目名称和经过核实、锁定版本的依赖。
- Cargo.lock：由 Cargo 生成，锁定所有间接依赖的准确版本。
- BENCHMARK.md：提前锁定怎样才算公平、稳定地优于 Hysteria 2。
- SCHEDULER_RESEARCH.md：记录 MPQUIC/MPTCP 调度论文、关键公式、失败算法的数学原因和下一候选 ACK-ECF 的设计边界。
- src/main.rs：普通实验的启动入口和中文结果展示入口。
- src/lib.rs：可复用的 MPQUIC 连接、传输校验、指标采集和断路实验逻辑；NoQ 相关操作集中在这里。
- src/scheduler.rs：FlowWeave 对外展示的调度名称；当前只有 NoQ 默认基线和 ACK-ECF 候选，失败候选不保留。
- benchmark-results：真实运行产生的逐轮原始数据和说明；失败轮次不删除。
- tests/network_lab.rs：四类坏网络场景、单路径对照和安全护栏。
- scripts/run_netem_lab.sh：进入一次性隔离网络空间，启动 smoke、screen 或 release 长时实验。
- third_party/README.md：记录第三方源码的版本、校验值、许可证和我们实际修改过什么。
- third_party/noq-proto：NoQ Proto 1.0.1 官方源码快照，加 4 个文件组成的最小测量补丁；当前没有自定义调度补丁。
- .gitignore：避免把编译产生的临时文件提交进 Git。

## 架构警戒线

后续每增加一层复杂度，必须先回答：

1. 它解决了哪个可以测量的问题？
2. 有没有更简单、已有标准的能力可以复用？
3. 是否让我们的协议错误地绑死在 NoQ 内部？
4. 正常、断路、乱序和恶意输入时分别会怎样？
5. 失败后能否回到最近一次可用提交？
6. 是否在没有基准数据时提前优化？
7. 主人能否用一句人话理解这次变化？

当前硬性边界：

- 不自创加密，继续使用标准 TLS 1.3。
- 没有真实调度收益，不宣称带宽聚合。
- 不扩大 NoQ fork；当前偏离严格记录在 `third_party/README.md`，升级时可逐文件撤销重放。
- B 的轮询、MinRTT、预计送达和交付速率加权都已按原始数据删除；ACK-ECF 已有论文依据和确定性测试，但必须经过同一五种子门槛才允许长期保留。
- ACK-ECF 使用 ACK 确认交付率、在途字节和最小 RTT 预测到达时间，并引入由 ACK、pacing 或 QUIC 丢包定时器唤醒的等待动作；完整推导见 `SCHEDULER_RESEARCH.md`。
- C 先验证关键 Datagram 双发，再决定是否引入 FEC；不能因为流吞吐量变好就忽略实时包仍大量丢失。

## 当前明确没有做

- 没有代理入口、TUN、SOCKS5 或用户界面。
- 没有真正完成带宽聚合。
- 没有选出最终调度算法；ACK-ECF 只是尚未通过真实五种子筛选的候选。
- 没有改善约 4 秒的断路检测时间。
- 没有关键数据双发或 FEC。
- 没有修改真实网络、部署服务器、上传文件或接触真实用户数据。

## 本机验收命令

普通功能实验：

~~~bash
cargo run
cargo test --all-targets
~~~

隔离坏网络实验：

~~~bash
./scripts/run_netem_lab.sh
./scripts/run_netem_lab.sh screen
./scripts/run_netem_lab.sh long
~~~

默认 `smoke` 坏网络实验会运行约一分钟；`screen` 会运行五种子短流筛选；`long` 使用 release 构建，运行约 19 分钟的正式时长 B 组复赛。看到最终 `test result: ok` 表示实验基础设施成功完成；指标不好不代表测试程序失败，它可能正是在如实揭示协议短板。
