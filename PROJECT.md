# 项目地图

## 项目是做什么的

这是一个实验性的多路径传输项目，暂定名为 FlowWeave / 织流。研究方向是：把 Wi-Fi、移动网络或多条宽带连接编织成一条连接。

长期目标同时包含三件事：

- A：网络切换时，正在进行的连接不断开。
- B：多条线路可以共同传输，提高总速度。
- C：丢包严重时，实时数据仍尽量保持低延迟。

目前第二阶段 A、B、C 都已按预注册公平合同最终胜过固定的 Hysteria 2.9.3 对手。A 的 FlowWeave v6.9 正反向各十场均数据完整、最终响应闭环且逐场严格 `<1 s`，而 Hysteria BBR/Brutal 正反向均为 `0/5` 原业务连接连续。B 的 Cubic + NoQ 默认持续单流正式门控在平衡/异构取得 `26.580/27.509 Mbit/s` 中位；Hysteria 最强中位分别只有 `13.664/23.336 Mbit/s`，FlowWeave 相对为 `194.5%/117.9%`。C 的 v12 逐消息 2-of-3 小分片在五种子 formal 中 P95 中位 `23.275 ms`、最差 `23.429 ms`、总丢失 `45`、UDP 中位 `2.153667×`；Hysteria 最强 P95 中位 `29.657 ms`，FlowWeave 低 `21.519%`，同时通过可靠性、P99、线速和安全护栏。最小固定目标 TCP 代理已实现标准 TLS 1.3、严格证书名称校验、文件令牌、MPQUIC 多路径、双向半关闭转发、无重启共享令牌轮换/撤销、配置/权限拒绝和 systemd 部署样例；本地真实 TLS 双路径、并发流和轮换测试通过，同一物理出口的 30 分钟真实公网双接口 soak 也已通过。三个目标和最小部署入口现已闭环；独立运营商出口、多小时/多天 soak、升级策略和生产 SLA 仍是更强部署验证范围，不能由当前证据外推。

## 当前技术选择

- 核心语言：Rust。
- MPQUIC 实验底座：本地固定的 NoQ 1.0.1 与 NoQ Proto 1.0.1，加 FlowWeave 的最小可审计测量、A 组恢复和通用定向 DATAGRAM 补丁；失败的 B 组调度已全部删除，普通数据仍保持 NoQ 官方发包顺序。它是实验发动机，不是永久绑定的协议内核。
- 加密：标准 TLS 1.3；实验使用内存中临时生成并被客户端精确校验的证书，产品代理从磁盘读取单张 DER 叶证书、PKCS#8 DER 私钥和独立 CA DER，并执行标准 `server_name` 校验，没有 insecure verifier。
- 普通功能实验：Linux 回环地址，只监听本机，外部设备无法连接。
- 坏网络实验：rootless 隔离网络命名空间加 Linux tc netem。网络限制随脚本退出而消失，不修改真实网卡。
- Linux VPN 数据边界直接使用锁定的 `libc 0.2.186`：包桥接只执行 `fcntl(O_NONBLOCK)`；独立附着层在非 root、无 `CAP_NET_ADMIN` 且启用 `NoNewPrivileges` 后，只对已存在、已启用、MTU 精确匹配的接口执行 `TUNSETIFF/TUNGETIFF` 复核。当前没有创建 TUN、地址、路由、NAT、forwarding 或 DNS 修改代码。
- 依赖策略：根项目通过 `[patch.crates-io]` 同时引用本地 vendor 的 `noq` 1.0.1 和 `noq-proto` 1.0.1；官方包校验值、上游提交、许可证和每个偏离上游的文件都记录在 `third_party/README.md`。
- 公平基准：固定对手为 Hysteria 2.9.3；最终结论必须满足 BENCHMARK.md，不能根据跑分临时改规则。

## 当前进度

### 第一阶段：真正的 MPQUIC 双路径

2026-07-11 已完成并实际运行：

- 客户端与服务端成功协商 MPQUIC。
- 同一连接建立两条路径，不是两个互不相干的普通 QUIC 连接。
- 两条路径分别承载了实际数据。
- 主路径被正常关闭后，原连接继续通过第二路径传输。
- QUIC Datagram 能够往返。
- 第三条超限路径、格式错误的数据和超出实验上限的参数会被拒绝。

### 第二阶段 A：可重复坏网络实验场

2026-07-11 已建立并实际运行：

- 两条线路可以分别设置带宽、延迟和丢包。
- 每个对照项开始前都会重置网络队列；丢包模拟使用固定随机种子，减少“不同选手抽到不同随机题目”的偏差。
- 能对比仅线路一、仅线路二和 NoQ 默认多路径。
- 能记录断路恢复时间、流吞吐量、Datagram P50/P95/P99、应用层丢失、额外流量和每条路径的统计。
- 测试只允许在一次性隔离网络空间运行；即使手动伪造启动标记，只要仍在主网络空间就会拒绝修改队列。

连续三轮固定场景的结果：

- 20 ms / 0.1% 与 80 ms / 1% 两路同时可用时，默认多路径吞吐量约 10.15～10.45 Mbit/s，两条路径都显著承载了流量。
- 实验把单路径空闲判定上限设为 3 秒；主线路在传输中变为 100% 丢包后，原连接约 3.99～4.40 秒恢复，数据改走备用线路。
- 8% 与 2% 高丢包场景中，默认多路径吞吐量三轮约为 3.82、2.40 和 8.10 Mbit/s；相对当轮最佳单路径，两轮更快，一轮只剩约 46%。默认多路径的 Datagram 应用层丢失约为 13.89%～18.06%。
- 8 Mbit/s 低延迟线路与 25 Mbit/s 高延迟线路组合时，默认多路径相对当轮最佳单路径约在 -3.2%～+1.9% 之间，没有稳定聚合收益。

这些数字是实验基线，不是产品性能承诺。固定随机种子能减少丢包题目差异，但不能消除 QUIC 拥塞窗口、重传时机、操作系统调度和短测试数据量造成的波动。高丢包场景的三轮结果得出了不同的吞吐量排名，因此后续任何算法比较都必须看多轮中位数和波动范围，禁止挑选最好看的一次。

### 第二阶段 A：PTO 跨路径对冲候选

2026-07-12 已完成并实际运行：

- 最新 MPQUIC 草案允许逐路径 PTO 后在其他可用路径重传延迟包内容；SingleR 和 MRC 进一步支持“一个时刻补一次”以及“快速恢复数据、保守判断路径”的分离设计。
- NoQ 增加默认关闭的通用配置。官方基线与候选使用同一版本、同一 TLS 1.3、同一网络种子；A 组恢复机制没有混入 B 组调度枚举。
- 重复、重叠、乱序 ACK 只扣一次未确认字节；任一副本被确认后，尚未发送的无用重复范围会立即取消。
- 触发对冲时记录本路径下一包号；黑洞前已经在路上的旧 ACK 不能错误恢复主路，只有对冲后发出的探针被确认才恢复普通数据调度。
- 确定性测试覆盖默认关闭、单路径不触发、首次和重复 PTO、备用路拥塞窗口、恢复、纯 FIN、两种副本 ACK 顺序和旧 ACK 边界。
- 五种子原始结果保存在 `benchmark-results/2026-07-12-pto-hedge-screening.csv`。

短筛结果：

- 黑洞首字节恢复 5/5 低于 1 秒，P95（五样本即最慢值）为 393.00 ms；NoQ 默认基线约为 3.47～3.61 秒。
- 完整 256 KiB 候选校验仍需约 1.13～3.63 秒，因为 80 ms 冷备用路需要慢启动；该数字与“断流多久”分开报告，不能混用。
- 正常网络候选吞吐相对基线中位 99.45%，UDP 总量中位 100.68%，两条路径 5/5 保持开放。
- 高丢包候选 UDP 总量相对基线中位 100.40%，5/5 数据正确并至少保留一条路径。
- 候选五轮共触发 9 次对冲，新加入重传队列的 STREAM 载荷 413,248 字节。

候选因此通过了预先锁定的三组短筛并进入正式 A 组；后续持续流结果证明，短筛只看“故障后第一个有效字节”会漏掉后续长停顿，不能继续作为晋级结论。

### 第二阶段 A：30 秒双向正式实验

2026-07-12 已完成 release 构建下的 20 场正式实验：同一条持久 QUIC 流持续发送 30 秒，第 10 秒把主路改为 100% 丢包，正向和反向各使用 5 对固定种子，并交替运行 NoQ 基线与 PTO 候选。接收端逐条校验序号、载荷和摘要，断流指标取故障后最大的相邻正确记录到达间隔。

完整原始结果保存在 `benchmark-results/2026-07-12-pto-hedge-formal-a-v2.csv`：

- 正向候选 5/5 数据完整，但断流中位 `5570.67 ms`、P95/最差 `10830.88 ms`，远高于 `<1000 ms` 门槛；NoQ 基线中位 `6156.74 ms`、最差 `8963.60 ms`。
- 正向 5 轮只有 3 轮在故障后触发对冲，共重新入队 `1,789,216` 字节；其余 2 轮完全没有对冲。即使触发，对冲后的断流仍为 `2.80～5.57 s`，说明“一次搬运当前仍在途的 STREAM 帧”不能覆盖持续流的全部阻塞状态。
- 正向候选故障后 UDP 相对同轮基线中位 `101.18%`，额外流量不是本次失败的主要矛盾。
- 反向候选与 NoQ 基线均为 `0/5` 完整。发送端在主路黑洞后没有从健康备用路收到足够反馈，最终等待记录载荷超时；这暴露了数据能换路但 PATH_ACK/ACK 可能仍经故障主路返回的反馈面缺口。
- 反向候选还在故障前误触发过 1 次、4 字节的对冲，证明正常期误判也必须继续计账。

`benchmark-results/2026-07-12-pto-hedge-formal-a.csv` 是第一次诊断运行的 5 条不完整记录；当时 12 秒收尾上限会误杀仍在恢复的场次。它保留作实验器缺陷证据，不与修正后的完整 v2 数据混算。

正式结果否决了当前候选，但实验场本身有效：它同时抓到了“有对冲仍长停顿”“完全不对冲”和“反向反馈链路失效”三类短筛看不到的问题。下一轮不得调低门槛或只重跑好看的种子，必须先增加事件时间线并分别修复数据面与反馈面。

随后增加了不改变算法的三场时间线诊断，原始数据保存在 `benchmark-results/2026-07-12-pto-hedge-diagnostic.csv`：

- 正向种子 1101 在故障后约 `304 ms` 已完成 PTO 与对冲，但主路约 `4009 ms` 才关闭，备用路约 `4519 ms` 才第一次从本路径发回 PATH_ACK，完整记录直到约 `5348 ms` 才恢复。长停顿主要是反馈绕回黑洞主路，不是第一次 PTO 太慢。
- 正向种子 1103 没有触发任何主路 PTO，直接等到约 `5488 ms` 关闭主路，最大断流约 `10.86 s`。这是 ACK 修复后要单独复查的第二个问题。
- 反向种子 1101 在故障发生时备用路已经关闭。此前它发送过约 `1.19 MiB` 新数据，但确认被主路带回，发送端把这条健康但“只出不进”的路误判为空闲。
- NoQ 确定性测试已复现 `PATH_ACK[path 1]` 实际从 path 0 发出的行为。诊断统计现在能区分 PATH_ACK 确认的路径与实际发送路径，并记录首次 PTO、对冲、同路 ACK、路径关闭和最大断流窗口的毫秒时间线。

架构结论：多路径不是“数据能从两条路发出”就完成了；拥塞窗口、PTO 和路径存活都依赖 ACK 反馈。第一修复优先级必须是反馈面路径韧性，暂不增加 FEC、窗口预留或新的失效检测器。

### 第二阶段 A：PATH_ACK 同路反馈修复

2026-07-12 已实现并实际验证：

- 开放路径优先在本路径发送确认该路径的 PATH_ACK；Backup 路径可以发送 ACK-only 包，不再因应用数据调度偏好而失去反馈。
- 只有被确认路径已经 abandoned 或没有可用远端 CID 时，才从另一条开放路径跨路回退；关闭路径会把已有延迟 ACK 立即转为可发送，abandoned 路径不会继续发送自己的 ACK。
- 两项确定性测试分别证明“开放 Backup 同路确认”和“路径关闭后跨路回退”；NoQ 396 项测试、FlowWeave 8 项库测试以及两边 Clippy 均通过。
- 三场修复后原始数据保存在 `benchmark-results/2026-07-12-path-ack-affinity-diagnostic.csv`，旧诊断文件没有覆盖。

修复后代表结果：

- 正向种子 1101：备用路首次同路 PATH_ACK 约 `59.90 ms`，不再等待约 4.5 秒；但最大断流仍为 `5872.79 ms`。主路约 `477.50 ms` 已 PTO 并对冲 `664,037` 字节，说明剩余长停顿主要不是反馈黑洞，而是最早有序数据缺口没有被一次对冲完整清除。
- 正向种子 1103：备用路约 `204.82 ms` 开始发包、约 `312.05 ms` 已同路确认，但主路故障后仍完全没有 PTO，直到约 `14905.58 ms` 才关闭；最大断流 `14739.06 ms`。这稳定复现了独立的检测缺口。
- 反向种子 1101：故障时两条路径都保持开放，备用路约 `6.21 ms` 开始发包、约 `12.08 ms` 同路确认，传输由修复前失败变为数据完整；最大断流降为 `1349.29 ms`，但仍略高于 `<1 s` 门槛。

这项修复解决了协议正确性问题，但没有让 A 达标。反馈等待 `h_feedback` 已不再是主导项；剩余工作严格分成“无 PTO 的健康检测”和“最早有序缺失范围优先恢复”，不得混在一次参数调整里。

### 第二阶段 A：无 PTO 状态机根因

2026-07-12 已用正向种子 1103 单独复现，原始数据保存在 `benchmark-results/2026-07-12-no-pto-state-diagnostic.csv`。这轮只增加只读状态，未改变调度、PTO 或路径空闲上限；NoQ 397 项测试、FlowWeave 8 项库测试以及两边 Clippy 均通过。

- 故障瞬间主路并非“没有在途包”：有 `794` 个需确认包、约 `1.15 MiB` 在途字节，PTO 定时器已武装；故障后最高达到 `1305` 个需确认在途包、约 `1.87 MiB`。
- 主路故障时 RTT 约 `465.95 ms`、PTO 约 `540.14 ms`。它仍零星发送需确认包直到 `3216.63 ms`，所以按“最后一次需确认发包 + PTO”计算的截止至少在 `3756.77 ms`。
- 主路最后一次收到 UDP 是 `248.56 ms`；3 秒路径空闲约在 `3250.14 ms` 先触发，使路径进入 abandoned 并停止普通 PTO。普通时限丢包计时器为 0，包表没有提前被清空。
- MPQUIC 草案要求收到 PATH_ABANDON 后继续保留路径编号空间约 3 PTO，以处理乱序包和防止误发 Stateless Reset。该轮结束时主路 PTO 已升到约 `3825.13 ms`，3 PTO 约 `11.48 s`；路径状态在 `14888.37 ms` 才删除，和观测到的约 `11.64 s` 清场间隔一致。
- 备用路在 `6.67 ms` 已发包、`115.09 ms` 已同路确认，但最早阻塞记录直到 `14985.51 ms` 才恢复。最大断流 `14747.71 ms`，说明数据被卡在“路径已 abandoned、但还没最终判丢”的保留期，而不是备用路不可用。

架构结论：标准要求保留路径状态，不等于必须把业务恢复也拖到 3 PTO 之后。下一候选要分开处理：路径编号空间继续按标准保留；未确认 STREAM 数据在进入 abandoned 时立即做一次去重安全的跨路对冲；更早的黑洞检测则使用以 ACK 进展为锚点、不会被后续发包反复推迟的独立恢复时钟。禁止直接缩短 3 PTO 保留期或 3 秒路径空闲上限来掩盖问题。

该候选已完成确定性实现和真实单场验证，但仍未达标：

- 新开关默认关闭，PTO 对冲与 abandoned 即时对冲可以分别复现；历史诊断仍使用 PTO-only 配置。
- 启用后，进入 abandoned 会把当前路径包表中的 STREAM 范围加入全局重传队列一次；原包号状态和晚到 ACK 处理保持到正常 3 PTO 清场。
- 关闭开关时不会改变上游行为；启用时仍要求存在另一条已验证路径，并继续遵守目标路径拥塞窗口和 pacing。
- 确定性测试证明数据在路径状态删除前已经从 Backup 送达，同时原路径包表仍保留；NoQ 399 项测试、FlowWeave 8 项库测试以及两边 Clippy 均通过。
- 真实正向种子 1103 的原始数据保存在 `benchmark-results/2026-07-12-abandon-reinjection-diagnostic.csv`：路径进入 abandoned 时一次对冲 `1,224,458` 字节，最大断流从上一轮 `14747.71 ms` 降到 `3712.71 ms`。
- 主路仍到 `15291.20 ms` 才最终删除，但最大断流已在 `3878.45 ms` 结束，直接证明业务恢复和 3 PTO 路径状态保留已经解耦。
- 剩余断流起于故障后约 `165.73 ms`、止于 `3878.45 ms`；LossDetection 在约 `3388.26 ms` 停止后约 490 ms 恢复，说明现在的主导项就是 3 秒 PathIdle，而不是路径清场。

### 第二阶段 A：ACK 进展恢复时钟候选

2026-07-12 已完成默认关闭的确定性实现和正向种子 1103 真实诊断：

- 标准 QUIC PTO 与 LossDetection 完全保留；新增时钟只在当前路径仍压着 STREAM 数据、且存在另一条已验证路径时工作。
- 纪元从首个在途 STREAM 包开始，真正新确认的 ack-eliciting 包会重置；继续发新包不会移动截止。触发预算仍使用当前路径的 QUIC Data PTO，不引入固定 RTT 魔数或在线概率模型。
- 每个无 ACK 进展纪元最多对冲一次，并在原路径发送一个带立即确认请求的 PING。原路径只暂停普通数据，不改变公开 PATH_STATUS；只有恢复边界之后的探针 ACK 才恢复普通调度。
- 逐路径在途 STREAM 计数为 O(1)，没有在每次 ACK 时扫描整张包表；PTO、abandoned 与 ACK 进展三类统计完全分账。

### 2026-07-12：数学路线审计

继续复核 CloudBurst、FlEC、QUIRL、AC-RLNC、编码多路径队列、Whack-a-Mole、Queueing Bandits、LSQ 和 Coupled BBR 等工作后，方向进一步收敛：

以下条目保留 2026-07-12 当时的预注册判断；后续 B 的持续单流正式门控和 C v5-v12 已继续验证、否决并最终收口，当前结论以本页前面的 B/C 最终门控和 `MATH_RESEARCH.md` 文末架构决定为准。

- 普通 ACK 只能看到 `min(投喂量, 真实容量)`；路径未被填满时，真实容量在数学上不可辨认。下一代 B 可以不再输出容量数字，而研究“某路径确认多少新数据，就归还多少本路径新数据信用”的有界完成驱动。
- B 的异构门槛要求 `容量利用率 × 编码有效率 >= 87.12%`；即使每 16 份原数据只加 1 份校验，也要求两路总容量利用率至少 92.57%。因此重型 FEC 不可能替代正确的 B 聚合。
- C 的两路独立双发理论丢失率为 `8% × 2% = 0.16%`，可达到 99.84%；真正风险是 200 字节小包的包头。四条消息一批可把估算线速压到约 2.18 倍，并只增加最多约 30 ms 凑批等待。
- `(2,3)` XOR 虽更省流量，但离 99.5% 门槛太近，不作为第一候选。第一候选仍是可审计的批量双发；只有实测过不了流量门槛才升级真正编码。
- FALCON 需要 729 个预训练粗粒度模型；GPR-MPQUIC 主要依赖神经网络和仿真；QUIRL 的 Tetrys 实现还存在知识产权公开限制。当前都不进入实现。

完整公式、论文适用边界和下一步实验顺序见 `MATH_RESEARCH.md`。

## 这些结果说明什么

- NoQ 默认行为确实能让两条 Available 路径同时发送数据。
- “两路都有流量”不等于“带宽聚合”：异构线路中几乎没有超过最佳单路。
- 单次吞吐量变好不等于算法已经变好：高丢包排名会波动，而且 Datagram 丢失仍然严重。
- A 的早期 PATH_ACK、abandoned、ACK 进展、流控与应用义务反例已由 v6.9 收敛；正式正反向各十场均完整闭环并逐场严格低于 1 秒，而 Hysteria 2.9.3 的 BBR/Brutal 正反向都无法保持原业务连接，A 已按预注册分支最终胜出。
- B 淘汰多代自定义调度和容量传感器后，最终由未修改调度的 Cubic + NoQ 默认持续单流合同在平衡/异构达到 `5/5` 与 `4/5`，并以 `194.5%/117.9%` 的中位吞吐最终胜过两场 Hysteria 最强项；失败候选没有被包装成产品算法。
- A 的恢复机制与 B 的容量调度必须保持分离；旧 ACK、纯 FIN、拥塞窗口和额外 UDP 都已经证明是不可省略的边界。
- C 的 v2/v4 分别暴露批处理等待和完整冗余日程尾部，v5-v11 又依次否决 GSO 相关副本、边缘 P95、块内自由度、种子敏感、pair 等待和 DATAGRAM 两级 FIFO；v12 最终以 P95 中位 `23.275 ms`、总丢失 `45`、UDP 中位 `2.153667×` 胜过 Hysteria 最强项，失败候选没有遗留为产品默认能力。

## 架构警戒线

后续每增加一层复杂度，必须先回答：

1. 它解决了哪个可以测量的问题？
2. 有没有更简单、已有标准的能力可以复用？
3. 是否让我们的协议错误地绑死在 NoQ 内部？
4. 正常、断路、乱序和恶意输入时分别会怎样？
5. 失败后能否回到最近一次可用提交？
6. 是否在没有基准数据时提前优化？
7. 主人能否用一句人话理解这次变化？

当前硬性边界：

- 不自创加密，继续使用标准 TLS 1.3。
- 只按各自预注册正式合同宣称已经测得的 A/B/C 收益；不把失败的自定义调度、其他负载或未测公网环境外推为普遍性能承诺。
- 不扩大 NoQ fork；当前偏离严格记录在 `third_party/README.md`，升级时可逐文件撤销重放。
- B 的轮询、MinRTT、预计送达、交付速率加权、ACK-ECF、`qualified-service-gso-v1` 和 staged 候选都已按原始数据删除；最终胜出路径保持 NoQ 官方低 Path ID 顺序和 Cubic，不复活失败调度。
- B ingress v2 与 declared epoch v1 已分别证明外部饱和服务和 MPQUIC 原来源服务可观察；禁止读取 `tc` 配置作为协议输入、把应用受限 ACK 或 pacing 当容量、复活历史候选，或复用任何历史 CSV。
- A 已最终胜过 Hysteria 2.9.3；后续产品化不能缩短 PathIdle 或 MPQUIC 3 PTO 保留，也不能删除已验证的精确证书校验与标准 TLS 1.3 边界。
- C 最终只保留 v12 合同：逐消息三片均受主路 BBR3 的 cwnd/pacing/MTU/反放大约束，三条全局方程受备用路同样约束；固定 `7/100/107` 字节、no-GSO、独立 UDP 边界和完整计费不得放松。v1-v11 的失败文件冻结，专用 priority 等失败机制不得复活。

## 当前明确没有做

- 已有固定目标 TCP 最小代理、JSONL 健康门控、本地/同出口公网 soak 和共享令牌轮换；VPN 逐客户端身份、代际、撤销、服务端逐身份数据句柄、客户端 `ACCEPT` 工厂、协商包长、真实重组、策略、NoQ DATAGRAM 运行器和预附着 packet fd 桥接现已由单客户端连接与 Endpoint 生命周期完成真实 UDP/DNS/显式双源路径/mTLS/FWC1/双向 IPv4/IPv6 loopback 组合合同，并验证身份拒绝后继续接入、活动期忙时拒绝、地址漂移与错误源地址拒绝、握手截止、停止释放、唯一 TUN 读者和连续代际设备复用。隔离双 network namespace 门控又把两端真实 TUN、veth 外层和同一 Endpoint 核心同场组合，UDP、TCP、双栈 ICMP、分片、精确/超 MTU、三代际、外层失联收敛及 `SIGKILL` 后重附着均通过。仍没有产品 VPN 命令、可安装的 root oneshot、真实宿主机默认路由/NAT/DNS、内部长期重连与多路径恢复、完整故障矩阵、SOCKS5、动态目标、管理界面或自动更新。
- 默认 NoQ 已在持续单流五种子合同中通过 B 自身门槛，并在两个场景的吞吐和额外线速上最终胜过 Hysteria 2.9.3。
- 没有需要产品化的新自定义调度算法；当前没有通过筛选的自定义候选，B 最终使用 NoQ 默认调度。
- A、B、C 均已完成 Hysteria 2.9.3 公平对照并在各自预注册合同中最终胜出；同出口 30 分钟公网双接口 soak 不能替代独立运营商、多小时/多天、跨版本升级或生产 SLA 证据。
- 已有通用可靠 TCP 代理数据面和 C 组专用的 v12 XOR/GF(256) 实验编码器，但没有通用 FEC 协议或生产级实时媒体接口。
- 没有修改主网络空间、部署服务器、上传文件或接触真实用户数据；TUN 创建只发生在脚本的一次性隔离网络空间中并随进程退出消失。

## 本机验收命令

普通功能实验：

~~~bash
cargo run
cargo test --all-targets
~~~

隔离真实 TUN 附着门控（不会修改主网络空间）：

~~~bash
./scripts/run_vpn_tun_lab.sh
~~~

隔离坏网络实验：

~~~bash
./scripts/run_netem_lab.sh
./scripts/run_netem_lab.sh failover
./scripts/run_netem_lab.sh formal-a
./scripts/run_netem_lab.sh diagnose-a
./scripts/run_netem_lab.sh diagnose-no-pto
./scripts/run_netem_lab.sh diagnose-abandon
./scripts/run_netem_lab.sh diagnose-ack-escape-representative
./scripts/run_netem_lab.sh diagnose-second-gap
./scripts/run_netem_lab.sh diagnose-feedback-handoff
./scripts/run_netem_lab.sh diagnose-feedback-handoff-1103
./scripts/run_netem_lab.sh screen
./scripts/run_netem_lab.sh long
~~~

默认 `smoke` 坏网络实验会运行约一分钟；`failover` 运行 A 组 30 场五种子短筛；`formal-a` 使用 release 构建运行 20 场、约 11 分钟的双向持续换网正式实验；`diagnose-a` 使用三个代表场景运行约 2 分钟的 A 组事件时间线；`diagnose-no-pto` 只复跑正向种子 1103 的 PTO-only 历史状态；`diagnose-abandon` 用相同种子验证 abandoned 即时对冲；`diagnose-ack-escape-representative` 用旧 ACK 逃生候选复跑正反向 1101；`diagnose-second-gap` 只复跑正向 1101 并输出逐流 `R/H/A/Q`；`diagnose-feedback-handoff` 用新候选复跑正反向 1101；`diagnose-feedback-handoff-1103` 只运行一次正向 1103，并写入独立摘要和 `R/H/A/Q` 时间线。这些诊断都不修改门槛，并记录 RTT/PTO、在途包、包表、最后一次需确认发包和定时器状态；`screen` 运行 B 组五种子短流筛选；`long` 使用 release 构建，运行约 19 分钟的正式时长 B 组复赛。看到最终 `test result: ok` 表示实验基础设施成功完成；指标不好不代表测试程序失败，它可能正是在如实揭示协议短板。
