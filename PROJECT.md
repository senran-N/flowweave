# 项目地图

## 项目是做什么的

这是一个实验性的多路径传输项目，暂定研究方向是：把 Wi-Fi、移动网络或多条宽带连接编织成一条连接。

长期目标包含三件事：

- A：网络切换时，正在进行的连接不断开。
- B：多条线路可以共同传输，提高总速度。
- C：丢包严重时，实时数据仍尽量保持低延迟。

目前只处于第一阶段可行性验证，不是可日常使用的代理软件。

## 当前技术选择

- 核心语言：Rust。
- MPQUIC 实验底座：NoQ 1.0.1。
- 加密：NoQ 提供的 TLS 1.3；实验使用内存中临时生成且被客户端精确校验的证书。
- 实验环境：Linux 回环地址。它模拟两张网卡，但不会修改电脑的真实网络设置。
- 服务端只监听本机回环地址，外部设备无法连接这个实验服务。

## 当前进度

2026-07-11 已完成并实际运行第一阶段实验：

- 客户端与服务端成功协商 MPQUIC。
- 同一连接使用相同服务端地址、两个不同客户端地址建立了两条路径。
- 主路径和第二路径分别传输了超过 256 KiB、512 KiB 的测试数据。
- 主路径关闭后，原连接继续通过第二路径传输。
- 24 个 QUIC Datagram 全部成功往返；回环环境延迟只用于确认功能，不代表真实网络性能。
- 第三条超限路径和格式错误的数据都被明确拒绝。
- `cargo test` 当前为 2 项通过、0 项失败。

## 主要文件

- `Cargo.toml`：记录项目名称和经过核实、锁定版本的依赖。
- `Cargo.lock`：由 Cargo 生成，锁定所有间接依赖的准确版本。
- `src/main.rs`：第一阶段 MPQUIC 双路径实验和自动测试。
- `.gitignore`：避免把编译产生的大量临时文件提交进 Git。

## 第一阶段要证明什么

1. 客户端和服务端确实协商启用了 MPQUIC。
2. 同一个连接能够建立两条不同的网络路径。
3. 两条路径都能实际承载数据。
4. 主路径关闭后，原连接仍能通过第二条路径继续传输。
5. QUIC Datagram 实时数据通道在剩余路径上仍可工作。
6. 超过路径上限或发送格式错误的数据时，程序会明确拒绝，而不是崩溃。

## 当前明确没有做

- 没有代理入口、TUN、SOCKS5 或用户界面。
- 没有真正的带宽聚合调度算法。
- 没有 FEC 前向纠错。
- 没有修改真实网络、部署服务器或发布任何东西。

## 本机验收命令

```bash
cargo run
cargo test
```
