# 数学路线审计：从“猜带宽”转向“排队与有效信息”

## 一句话结论

数学没有要求 FlowWeave 立刻引入神经网络或庞大的 FEC 框架，反而给出了三个更简单的方向：

- A 保留“一次 PTO 跨路径对冲”这个恢复原语，但先修 PATH_ACK 反馈路径；当前候选已经被正式持续流淘汰。
- B 不再反复猜线路上限，改研究“哪条路径完成了工作，哪条路径就获得下一份工作”的有界 ACK 信用。
- C 先做四条消息一组的跨路径批量双发；只有它过不了流量门槛，才升级到真正的纠删码。

这里的“有效信息”也叫自由度：它不是某个指定包，而是任何一份能让接收端多恢复一份原始数据的信息。

## 现有失败背后的数学原因

### B 的容量为什么不可直接从普通 ACK 看出来

用一个最小模型就能说明问题。设：

- `x_i`：调度器主动喂给路径 `i` 的速率；
- `C_i`：路径真正最多能承载的速率；
- `y_i`：接收端通过 ACK 看到的交付速率。

在没有其他瓶颈时：

```text
y_i = min(x_i, C_i)
```

如果观察到 `y_i = x_i`，那真实容量可能是 `x_i`、`2x_i`，甚至 `100x_i`。这些不同的真实世界会产生完全相同的 ACK 观测。

因此，只要路径没有被确实填满，任何算法都不可能只靠这份观测唯一推回 `C_i`。这不是平滑参数没调好，而是信息本身不够。要识别容量，必须满足至少一个条件：

1. 受控地把路径填满；
2. 获得 ECN、队列、物理层速率等额外信号；
3. 不再要求先估出容量，直接换一种调度结构。

FlowWeave 已经真实验证了前两类方法在当前短流合同中的困难，因此下一代优先研究第三类。

## 先进论文带来的真正启发

| 工作 | 数学或系统思想 | 对 FlowWeave 有用的部分 | 不能直接照搬的原因 |
|---|---|---|---|
| CloudBurst（2021） | 把消息编码后撒到多条路，收到最先到达的足量符号就恢复 | 不必提前知道哪条路最快；把“等最慢包”改成“等第 K 个有效包” | 面向数据中心短消息，不做拥塞控制，还依赖小交换机队列 |
| FlEC（2022） | 应用把截止时间告诉传输层，冗余只在最后仍来得及恢复时发送 | 冗余必须服从拥塞窗口；低延迟时自动退回普通重传 | 通用框架约 8200 行 C，远超当前项目需要 |
| QUIRL（2024） | 普通重传与 FEC 按应用延迟要求切换 | “能等重传就不编码，来不及才编码” | 公开仓库说明 Tetrys 实现因知识产权问题未公开，不适合作为地基 |
| 有限滑窗 RLNC（2018） | 用滑动窗口消除固定分块等待，并给出异构延迟下的缓冲区公式 | 编码窗口和接收缓冲必须显式计账 | 假设路径速率已知，并使用 GF(2^8) 编码 |
| AC-RLNC（2020） | 跟踪“缺少的自由度”和“补上的自由度”，前馈与反馈冗余结合 | C 可以按缺口补信息，而不是盲目重复某个包 | 模型主要是等时隙擦除信道，不等于公网带宽异构路径 |
| 编码多路径的抢占研究（2023） | 把系统建模成 `D/M/(K,N)/L` 分叉汇合队列 | 队列无限长会放大延迟；严格只留最新一批又会损害可靠性，短队列更稳 | 指数服务时间和同步到达是理论简化 |
| Whack-a-Mole（2025） | 用位反转序列把任意连续区间内的分配误差限制在 `O(log m)` | 一旦目标比例已知，可以稳定、低开销地撒包；高延迟路应在批次末尾提前停发 | 它保证“严格执行比例”，但不负责找出正确比例 |
| Transport Coding（2025） | 编码把完成时间从最大值 `T_(k:k)` 变成第 K 个到达值 `T_(k:n)` | 清楚展示冗余降低尾延迟、但也提高负载的矛盾 | 只模拟相同、指数分布路径，不能作为真实 WAN 证明 |
| Queueing Bandits（2026） | 在未知服务率下同时学习和排队，用队列遗憾衡量错误的长期代价 | 证明“边学边调度”必须为探索和积压付账 | 理论状态和计算量不适合短流实时发包 |
| LSQ / Pull-based load balancing | 服务器完成工作后主动拉取下一份任务，有界队列误差仍可保持稳定 | B 可以让路径用实际完成事件争取下一份数据，不先输出一个容量数字 | 服务器队列理论不能直接当作 QUIC 性能证明，只能指导状态机 |
| Coupled BBR + AR&P（2021） | BBR 提供平滑速率，再按预计到达时间排包；坏路转为承载重复数据 | 拥塞控制和调度不能各做各的；长期饱和证据比普通 ACK 平滑可靠 | 其耦合公式故意把总速率限制为最佳单路，按设计无法通过 FlowWeave 的 B 聚合门槛 |
| FALCON（2022）、GPR-MPQUIC（2026） | 离线模型加在线学习，或神经网络预测路径状态 | 提醒我们动态无线最终可能需要上下文 | FALCON 预设 729 个粗粒度模型；GPR 主要是 ns-3 仿真。当前引入会成为难审计黑盒 |

## A：真正缺的是反馈闭环，不是更多副本

原 A 候选只用了最优停止问题的简化形式：

```text
T(h) = min(T_original, h + T_backup)
```

它在第一次逐路径 PTO 时只补发一次，把快速恢复和最终判死分开。SingleR 的一次补发预算、MPQUIC 草案的跨路径重传，以及 MRC 的数据面/控制面分离仍支持这个原语，但正式持续流证明该式漏掉了返回确认：

```text
G >= h_detect + h_feedback + S_backup(B_blocking)
```

诊断种子 1101 的真实时间线为：

- 主路黑洞后约 `304 ms` 已触发 PTO、进入恢复并重新入队约 `450 KiB`；
- 主路约 `4009 ms` 才关闭；
- 接收端约 `4519 ms` 才第一次从备用路本身发回同路 PATH_ACK；
- 第一条解除有序阻塞的完整记录约在 `5348 ms` 到达。

因此这轮 `5.264 s` 断流不是“对冲太晚”，而是备用路的确认最初被调度到黑洞主路，备用路拥塞窗口得不到归还。可近似分解为：

```text
h_detect ~= 0.304 s
h_feedback ~= 4.519 - 0.304 = 4.215 s
S_backup   ~= 5.348 - 4.519 = 0.829 s
```

反向诊断更直接：备用路在故障前已经发送约 `1.19 MiB`，但因为其 PATH_ACK 一直经主路返回，发送端把这条“只出不进”的路径按空闲规则提前关闭；故障发生时备用路已经不存在。这个结果与 MPQUIC draft 21 第 5.4/5.5 节一致：PATH_ACK 可以跨路径，但 RTT 和控制环会同时包含数据路径与 ACK 路径，策略必须稳定。

A 的下一步不是增加 FEC 或拥塞窗口预留，而是让开放路径优先在本路径返回自己的 PATH_ACK；只有路径已关闭或不可发送时才跨路回退。修完后仍要复查另一个独立现象：某些持续流故障点没有任何 PTO，直接等到路径空闲关闭。

## B：从容量估计改成 ACK 信用

### 核心区别

失败的 ACK-ECF 做的是：

```text
ACK 速率 -> 估计容量 -> 计算下一轮分配比例
```

下一候选研究的是：

```text
某路径确认完成 X 字节 -> 该路径获得最多 X 字节的新数据信用
```

它不把 ACK 换算成“这条路有多少 Mbit/s”，因此不会拿调度器自己制造的投喂量冒充线路上限。更快的路径在相同时间里归还更多信用，自然得到更多新工作；停止确认的路径也会自然停止获得新工作。

这只是候选原理，不是已经证明成功的实现。必须处理：

- 每条路的初始信用和拥塞窗口上限；
- ACK 是确认原副本还是跨路径重传副本；
- 丢包、重复 ACK 和乱序 ACK 不能重复发放信用；
- 控制帧、重传数据和新应用数据必须分账；
- 所有路径都暂时没有信用时不能死锁；
- 路径恢复后只能小步重新加入，不能突然灌入大量旧信用。

### B 不能使用重型冗余

异构场景总物理容量是：

```text
8 + 25 = 33 Mbit/s
```

阶段门槛要求至少达到：

```text
1.15 × 25 = 28.75 Mbit/s
```

设：

- `U`：两路总容量实际被利用的比例；
- `R`：编码有效率，原始数据 / 总编码数据。

即使不算其他协议开销，也必须满足：

```text
33 × U × R >= 28.75
U × R >= 0.8712
```

如果每 16 份原数据加 1 份校验，`R = 16/17 = 94.12%`，仍要求总容量利用率至少 `92.57%`。若使用 1.5 倍流量的重型编码，理论有效上限只有 `22 Mbit/s`，不可能通过异构 B 门槛。

所以 B 的第一候选必须几乎不加冗余。编码只能以后作为很轻的丢包/尾部修复层，不能代替正确聚合。

### 异构延迟所需缓冲并没有想象中大

有限滑窗 RLNC 论文给出的无编码近似缓冲关系是：

```text
buffer ~= low_delay_path_rate × delay_difference
```

对锁定的 8 Mbit/s、15 ms 与 25 Mbit/s、50 ms 场景：

```text
8 Mbit/s × (50 - 15) ms = 35,000 字节
```

这说明异构路径不是天然不能聚合；真正问题是分配错误和无界排队。未来候选应设置明确的重排缓冲上限并记录峰值，不能靠无限内存掩盖队头阻塞。

## C：先用概率算清最小方案

### 直接双发为什么在可靠性上足够

锁定场景中两路独立丢包率为 `8%` 和 `2%`。同一批数据各走一份时，两份同时丢失的概率是：

```text
0.08 × 0.02 = 0.0016 = 0.16%
```

理论有效到达率为：

```text
1 - 0.0016 = 99.84%
```

它高于 `99.5%` 阶段门槛，并且完成时间取两路中先到的一份：

```text
T = min(T_path1, T_path2)
```

真正的风险不是可靠性，而是 200 字节小消息的包头开销可能把两份副本推到 `2.2×` 线速上限之外。

### 批处理可以把包头摊薄

设：

- 每条应用消息 `M = 200` 字节；
- 每批 `g` 条消息；
- 每条消息在 QUIC DATAGRAM 帧中的小额固定开销为 `F`；
- 每个实际 UDP 包的其余固定开销为 `H`；
- 整批在两条路径各发送一次。

总流量比例约为：

```text
ratio = 2 × [g × (M + F) + H] / (g × M)
```

以保守的 `F ≈ 3`、`H ≈ 60` 字节估算，`g = 4` 时：

```text
ratio ≈ 2.18
```

四条消息在每秒 100 条的负载下，最多增加约 `30 ms` 的凑批等待，仍给 `150 ms` 的 P95 门槛留下充足余量。

这只是设计估算。正式候选必须使用 NoQ 实际 UDP 发送字节和系统层包长共同复核，不能用公式替代测量。

### 为什么不先上最省流量的 `(2,3)` XOR

可以把两条原消息和一条 XOR 校验分别作为三个包发送。若它们的丢包率为 `8%`、`2%`、`2%`，至少丢两个包、导致整组无法完整恢复的概率约为：

```text
0.3536%
```

平均值看似也能达到 `99.5%`，但余量很窄。按“坏组里的两条消息都算丢失”的保守模型，单次 60 秒实验越过 `0.5%` 丢失线的概率约为 `7.28%`。它适合以后作为省流量备选，不适合作为第一候选。

### C 第一候选的边界

- 每四条 200 字节消息组成一批，批内保留每条消息的序号和产生时间。
- 同一批明确指定两条不同路径各发送一次，不能让默认调度碰巧把两份都放在同一路。
- 第一份正确副本恢复四条消息；后一份只计作重复，不再次交付。
- 批次超过截止时间后丢弃并记作迟到，不能靠无限等待提高到达率。
- 两份副本都必须受各自路径的拥塞窗口和 pacing 约束。
- 先测实际包头与批大小；如果四条一批仍超过 `2.2×`，候选直接失败，不偷偷改计费口径。

## 编码知识产权与依赖边界

真正的 RLNC、Raptor、Tetrys 和部分滑窗编码实现存在专利、授权或实现未公开问题。QUIRL 仓库明确说明 Tetrys 实现因知识产权披露而保持私有；其配套 Rust `networkcoding` 仓库也主动标注了相关专利情况。

因此当前决定是：

- 不复制这些实现；
- 不因为论文效果好就立即安装编码依赖；
- 第一候选只使用普通副本和项目现有 QUIC/TLS 能力；
- 以后若需要 Reed-Solomon 或其他编码，先核实许可证、专利状态、维护情况和 CPU 成本，再单独获批引入。

## 架构决定

三个目标仍然全部保留，但不能塞进一个万能调度器：

```text
A：PTO 一次跨路径对冲
   解决“路径突然黑洞后，旧数据怎样尽快恢复”

B：有界 ACK 信用 / 服务完成驱动
   解决“未知容量下，新数据怎样自然按完成能力分配”

C：截止时间约束下的批量双发，必要时再升级编码
   解决“实时消息怎样用受控冗余降低丢失和尾延迟”
```

它们共享路径状态、拥塞控制和统计，但状态机、指标与失败条件保持分离。A 的 ACK 不能给 B 发新数据信用；C 的重复副本也不能被统计为 B 的有效聚合。

## 下一步实验顺序

1. 先实现并验证 PATH_ACK 同路优先、失效后跨路回退，重跑正式 A 组。
2. 若仍出现“没有 PTO、只等空闲关闭”，再单独修复无在途包时的健康探测，不与 ACK 修复混成一项。
3. 建立正式 C 组 200 字节、100 次/秒、60 秒的计时和线速计费场，先跑无冗余基线。
4. 只增加一个最小、通用的“指定本地路径发送 Datagram”能力，验证四消息批量双发。
5. C 通过后，再为 B 单独设计 ACK 信用的确定性状态机；不复活已经删除的 ACK-ECF 速率估计器。
6. 只有普通副本无法满足 C，或 B 的轻量尾部修复确有数据需求时，才进入编码方案选型。

代码实现虽然按依赖顺序小步推进，A、B、C 三条研究线都保持活跃；“同时进行”指三项目标不互相牺牲，不是把三套机制混成无法审计的一团。
