# 数学路线审计：从“猜带宽”转向“排队与有效信息”

## 一句话结论

数学没有要求 FlowWeave 引入神经网络或庞大的通用 FEC 框架；完整实验链最终把三个方向都收敛到可审计、可回退的机制：

- A 的反馈、数据义务与流控缺口已经由 v6.9 收敛，正式正反向各十场均完整闭环并严格 `<1 s`；Hysteria 2.9.3 的 BBR/Brutal 正反向五种子均为 `0/5` 连续，A 已通过最终连接连续性分支。
- B 的容量估计与自定义调度候选按证据逐个删除，最终不再猜容量：Cubic + NoQ 默认调度在持续积压单流合同中取得平衡/异构 `26.580/27.509 Mbit/s` 中位，Hysteria 最强项为 `13.664/23.336 Mbit/s`，FlowWeave 相对为 `194.5%/117.9%`，并通过额外线速护栏。数学结论不是“某个估计器终于猜对”，而是公平负载合同让现有逐路径拥塞控制和默认调度已经足够聚合。
- C 从 v5 的 GSO 相关同失、v6 的 no-GSO 独立性、v7 的六成直接副本、v9 的种子敏感 P95、v10 的成对等待和 v11 的两级 FIFO 失败继续收敛到 v12：每条消息用同刻 `2-of-3` 小分片本地恢复，再用 `40/3` 全局方程兜底。v12 五轮 P95 中位/最差为 `23.275/23.429 ms`、总丢失 `45`、UDP 中位 `2.153667×`；相对 Hysteria 最强 `29.657 ms` 低 `21.519%`，C 已最终通过。

下文按实验发生顺序保留推导和当时的“下一步”判断；这些阶段性判断由后续章节继续验证或否决，当前结论以上述三项和文末架构决定为准。

这里的“有效信息”也叫自由度：它不是某个指定包，而是任何一份能让接收端多恢复一份原始数据的信息。

## 现有失败背后的数学原因

### B 的容量为什么不可直接从普通 ACK 看出来

用一个最小模型就能说明问题。设：

- `x_i`：调度器主动喂给路径 `i` 的速率；
- `C_i`：路径真正最多能承载的速率；
- `y_i`：接收端通过 ACK 看到的交付速率。

在没有其他瓶颈时：

```text
y_i = min(x_i, C_i)
```

如果观察到 `y_i = x_i`，那真实容量可能是 `x_i`、`2x_i`，甚至 `100x_i`。这些不同的真实世界会产生完全相同的 ACK 观测。

因此，只要路径没有被确实填满，任何算法都不可能只靠这份观测唯一推回 `C_i`。这不是平滑参数没调好，而是信息本身不够。要识别容量，必须满足至少一个条件：

1. 受控地把路径填满；
2. 获得 ECN、队列、物理层速率等额外信号；
3. 不再要求先估出容量，直接换一种调度结构。

FlowWeave 已经真实验证了前两类方法在当前短流合同中的困难，因此下一代优先研究第三类。

## A：真正缺的是反馈闭环，不是更多副本

原 A 候选只用了最优停止问题的简化形式：

```text
T(h) = min(T_original, h + T_backup)
```

它在第一次逐路径 PTO 时只补发一次，把快速恢复和最终判死分开。SingleR 的一次补发预算、MPQUIC 草案的跨路径重传，以及 MRC 的数据面/控制面分离仍支持这个原语，但正式持续流证明该式漏掉了返回确认：

```text
G >= h_detect + h_feedback + S_backup(B_blocking)
```

诊断种子 1101 的真实时间线为：

- 主路黑洞后约 `304 ms` 已触发 PTO、进入恢复并重新入队约 `450 KiB`；
- 主路约 `4009 ms` 才关闭；
- 接收端约 `4519 ms` 才第一次从备用路本身发回同路 PATH_ACK；
- 第一条解除有序阻塞的完整记录约在 `5348 ms` 到达。

因此这轮 `5.264 s` 断流不是“对冲太晚”，而是备用路的确认最初被调度到黑洞主路，备用路拥塞窗口得不到归还。可近似分解为：

```text
h_detect ~= 0.304 s
h_feedback ~= 4.519 - 0.304 = 4.215 s
S_backup   ~= 5.348 - 4.519 = 0.829 s
```

反向诊断更直接：备用路在故障前已经发送约 `1.19 MiB`，但因为其 PATH_ACK 一直经主路返回，发送端把这条“只出不进”的路径按空闲规则提前关闭；故障发生时备用路已经不存在。这个结果与 MPQUIC draft 21 第 5.4/5.5 节一致：PATH_ACK 可以跨路径，但 RTT 和控制环会同时包含数据路径与 ACK 路径，策略必须稳定。

A 的下一步不是增加 FEC 或拥塞窗口预留，而是让开放路径优先在本路径返回自己的 PATH_ACK；只有路径已关闭或不可发送时才跨路回退。修完后仍要复查另一个独立现象：某些持续流故障点没有任何 PTO，直接等到路径空闲关闭。

### 数学修正：路径健康不是一个数，而是一张有向反馈矩阵

把数据从逻辑路径 `i` 发出、再把它的 PATH_ACK 从逻辑路径 `j` 返回，实际形成的是一条组合控制环 `(i, j)`。设：

- `d_i→`：数据在路径 `i` 去程的单向延迟；
- `d_j←`：确认在路径 `j` 回程的单向延迟；
- `a`：接收端等待和处理 ACK 的时间；
- `L_ij`：这个组合控制环的反馈延迟。

则：

```text
L_ij = d_i→ + a + d_j←
```

这正是 MPQUIC draft 21 第 5.4 节所描述的 RTT 含义。重要的是，收到 `PATH_ACK[i]` 只能证明“数据去程 `i` 与 ACK 回程 `j` 的组合成功”，不能证明 `i` 的两个方向都健康。于是：

- 同路 ACK `(i, i)` 给出稳定、容易解释的闭环，也会实际检验路径 `i` 的返回方向；
- 跨路 ACK `(i, j)` 在路径关闭或非对称故障时可以救活连接，但会隐藏路径 `i` 返回方向已经失效的事实；
- 只记录“path i 健康/不健康”会丢失关键状态，至少要分开记录数据去程证据、ACK 实际回程和组合反馈年龄。

A 的反馈不变量因此写成：

```text
对每条仍承载数据的开放路径 i，
必须存在一条已验证可用的返回路径 j，
使 PATH_ACK[i] 在限定时间 Δ_i 内回到发送端。
```

当前最小策略是词典序而不是复杂打分器：

1. `i` 仍开放且有可用 CID 时，优先 `j = i`，保持控制环稳定；
2. `i` 已 abandoned 或无法发送时，立即从剩余开放路径跨路回退；
3. 路径关闭会停止自己的 MaxAckDelay 定时器，因此所有已经积累的待发 ACK 必须在关闭瞬间转成“立即发送”，否则它会永久留在队列里；
4. 若以后真实复现“去程通、同路回程断”，再单独增加有界的跨路逃生或反向健康探测，不能现在无条件复制所有 ACK。

MRC 的 2026 生产经验支持第 4 点：它为控制包保留独立的反向路径集合，入站活跃但没有出站数据时每个 RTT 发送探测，并用成功响应更新反向路径。FlowWeave 可以借它的状态分离思想，但公网版本必须先测探测频率、额外流量和误判，不能照搬数据中心参数。

### 修复后的分解：反馈项已经退出主导位置

同路优先修复后的三个代表场景保存在 `benchmark-results/2026-07-12-path-ack-affinity-diagnostic.csv`：

- 正向 1101：备用路首次同路 PATH_ACK `59.904 ms`，主路 PTO/对冲 `477.503 ms`，最大断流 `5872.791 ms`；
- 正向 1103：备用路首次同路 PATH_ACK `312.045 ms`，主路 PTO/对冲均为 0，最大断流 `14739.058 ms`；
- 反向 1101：备用路首次同路 PATH_ACK `12.078 ms`，数据完整，最大断流 `1349.289 ms`。

与修复前备用路约 `4519 ms` 才首次同路确认相比，`h_feedback` 已下降两个数量级左右。因此剩余断流不能再归因于“ACK 还在绕主路”，而应写成两个互相独立的缺口：

```text
G >= h_detect + S_backup(B_blocking)

h_detect 的缺口：ack-eliciting 包持续零星发出时，现有 PTO 截止会被反复向后推。
B_blocking 的缺口：一次搬运 sent_packets 快照，不保证包含接收端最早未交付范围。
```

新的 1103 只读状态诊断把第一类反例进一步定量化。设：

- `s_k`：黑洞后第 `k` 个 ack-eliciting 包的发送时刻；
- `P(t)`：当时路径 PTO；
- `D_pto(t) = s_last(t) + P(t)`：现有 PTO 截止；
- `r_last`：最后一次收到主路 UDP 的时刻；
- `I = 3 s`：实验路径空闲下限；
- `D_idle = r_last + I`：路径空闲截止。

实测：

```text
s_last = 3.216634 s
P(0)   = 0.540144 s
D_pto  >= 3.756778 s

r_last = 0.248558 s
D_idle ~= 3.248558 s
实际进入 abandoned ~= 3.250141 s
```

所以并不是 PTO 没有武装，而是：

```text
D_idle < D_pto
```

只要新包到达发送器的间隔持续小于 PTO，而且累计未确认字节还没有填满较大的拥塞窗口，`s_last` 会不断前移，PTO 可以长期不触发。1103 故障时已经有 `794` 个需确认在途包，峰值 `1305` 个；“没有在途包”的假设被真实数据否定。

进入 abandoned 后，标准要求路径编号空间继续保留约 3 PTO。结束快照 `P_drain = 3.825131 s`，因此：

```text
3 * P_drain = 11.475393 s
```

从 LossDetection 停止到路径最终删除实测约 `11.638 s`，与 PATH_ABANDON 往返加 3 PTO 保留一致。备用路在 `6.673 ms` 已发包、`115.092 ms` 已同路确认，却要等到 `14.985508 s` 才解除有序阻塞。数学上这是两个串联等待：

```text
h_detect = min(D_idle, D_pto)                 # 本轮由 D_idle 先发生
h_release = PATH_ABANDON 往返 + 3 * P_drain  # 数据仍留在旧包表
G >= h_detect + h_release + S_backup(B_blocking)
```

正确拆法不是缩短标准保留期，而是令业务恢复与路径状态生命周期并行：

```text
T_state  = h_detect + 3 * P_drain             # 路径状态何时安全删除
T_data   = h_suspect + S_backup(B_blocking)    # 用户数据何时恢复
用户断流只应受 T_data 约束，不应等待 T_state。
```

第一步在路径进入 abandoned 时立即对冲未确认 STREAM 范围，但继续保留包号空间；第二步用 ACK 进展而不是最后发包时刻定义 `h_suspect`，后续新发包不得把恢复截止向后推。种子 1101 仍是另一类反例：`477.503 ms` 已触发对冲并搬运 `664,037` 字节，最大断流仍为 `5.87 s`。总补发量不是充分统计量，后续至少要观察：

第一步的真实结果已经符合模型预测：

```text
G_before = 14.747711 s
G_after  =  3.712711 s
差值     = 11.035000 s
```

主路状态仍到 `15.291197 s` 才删除，而有序交付在 `3.878446 s` 已恢复。因此 `T_data` 已不再等待 `T_state`。路径进入 abandoned 时对冲 `1,224,458` 字节，剩余约 3.71 秒断流可以近似写成：

```text
G_after ~= h_idle + S_backup(B_blocking)
h_idle  ~= 3.39 s
S_backup 的可见尾部 ~= 0.49 s
```

这里 `h_idle` 取 LossDetection 停止时刻 `3.388261 s` 附近；两项会有采样和记录边界重叠，不能机械相加当作精确等式。但主导关系已经清楚：下一步只有把 `h_suspect` 提前到 ACK 无进展后的一个有界窗口，才可能进入 `<1 s`。

```text
R(t)：接收端下一条可有序交付记录所需的最早 STREAM offset
H(t)：接收端已经看到的最高 STREAM offset
W(t)：R(t) 附近仍缺失的范围集合
```

恢复策略的目标应是最小化 `R(t)` 停滞时间，而不是最大化某轮复制的总字节数。这与 MPTCP 的 Buffer Blocking Protection、BLEST 的队头风险模型，以及 MRC 优先恢复最早缺失序号的做法一致。

### ACK 进展年龄：把“最后发了什么”改成“最后完成了什么”

种子 1103 暴露的是两个不同的随机过程：

```text
发送过程：S(t)，只表示本地调度器还在投喂路径
完成过程：C(t)，只在 ACK 真正确认新 ack-eliciting 包时增长
```

黑洞后 `S(t)` 仍增长而 `C(t)` 停止。用最后发送时刻计时等于让被监测对象自己不断延后检测；更有意义的充分状态是完成过程的年龄：

```text
A_i(t) = t - e_i
```

`e_i` 在首个包含 STREAM 的包进入在途集合时建立，只在 `C(t)` 真正增长且仍有 STREAM 在途时刷新；最后一个 STREAM 范围离开包表后清空。恢复触发条件写成：

```text
A_i(t) >= PTO_i
AND 存在另一条已验证路径
AND 当前路径仍压着未确认 STREAM 范围
AND 本纪元尚未触发过恢复
```

这与 RFC 6298 的 TCP RTO 计时边界同构，却只触发一次可撤销的多路径对冲，不替换 QUIC PTO、不提前宣告丢包，也不缩短 MPQUIC 的 3 PTO 状态保留。RFC 7765 说明还可以把截止锚定到最早未确认包，使部分 ACK 后不重新等待完整 RTO；第一版暂不采用这层更激进规则，先用“新 ACK 后重新给一个 PTO”控制误触发。

若以后需要概率化，可以把健康路径的无进展生存函数记为 `S_i(a)`，得到 φ-accrual 式怀疑度：

```text
φ_i(a) = -log10 S_i(a)
```

但当前没有稳定、跨换网可迁移的 `S_i`，在线拟合还可能把已经退化的样本学成“正常”。因此先把 `A_i/PTO_i` 作为可审计、无量纲的诊断量；正常网络和高丢包的额外 UDP 预算就是它的真实假阳性检验。

## B：从容量估计改成 ACK 信用

### 核心区别

失败的 ACK-ECF 做的是：

```text
ACK 速率 -> 估计容量 -> 计算下一轮分配比例
```

下一候选研究的是：

```text
某路径确认完成 X 字节 -> 该路径获得最多 X 字节的新数据信用
```

它不把 ACK 换算成“这条路有多少 Mbit/s”，因此不会拿调度器自己制造的投喂量冒充线路上限。更快的路径在相同时间里归还更多信用，自然得到更多新工作；停止确认的路径也会自然停止获得新工作。

这只是候选原理，不是已经证明成功的实现。必须处理：

- 每条路的初始信用和拥塞窗口上限；
- ACK 是确认原副本还是跨路径重传副本；
- 丢包、重复 ACK 和乱序 ACK 不能重复发放信用；
- 控制帧、重传数据和新应用数据必须分账；
- 所有路径都暂时没有信用时不能死锁；
- 路径恢复后只能小步重新加入，不能突然灌入大量旧信用。

### B 不能使用重型冗余

异构场景总物理容量是：

```text
8 + 25 = 33 Mbit/s
```

阶段门槛要求至少达到：

```text
1.15 × 25 = 28.75 Mbit/s
```

设：

- `U`：两路总容量实际被利用的比例；
- `R`：编码有效率，原始数据 / 总编码数据。

即使不算其他协议开销，也必须满足：

```text
33 × U × R >= 28.75
U × R >= 0.8712
```

如果每 16 份原数据加 1 份校验，`R = 16/17 = 94.12%`，仍要求总容量利用率至少 `92.57%`。若使用 1.5 倍流量的重型编码，理论有效上限只有 `22 Mbit/s`，不可能通过异构 B 门槛。

所以 B 的第一候选必须几乎不加冗余。编码只能以后作为很轻的丢包/尾部修复层，不能代替正确聚合。

### 异构延迟所需缓冲并没有想象中大

有限滑窗 RLNC 论文给出的无编码近似缓冲关系是：

```text
buffer ~= low_delay_path_rate × delay_difference
```

对锁定的 8 Mbit/s、15 ms 与 25 Mbit/s、50 ms 场景：

```text
8 Mbit/s × (50 - 15) ms = 35,000 字节
```

这说明异构路径不是天然不能聚合；真正问题是分配错误和无界排队。未来候选应设置明确的重排缓冲上限并记录峰值，不能靠无限内存掩盖队头阻塞。

### ACK 信用 v1 的真实反例

第一版实现严格遵守了上面的候选原理：首次 STREAM 字节扣信用，只有全局第一次确认的 STREAM 字节归还；重传、控制帧与 DATAGRAM 不扣信用，拥塞窗口只作为上限，全部路径归零时最多允许一个 MTU 的受控救援。确定性测试证明重复 ACK 不会重复发放、两条真实路径都能归还唯一完成量，因而烟测失败不能归因于状态机记账错误。

异构 `8+25 Mbit/s` 的两个固定种子结果为：

```text
1101/2201：最佳单路 10.76，NoQ 12.81，ACK 信用 11.62 Mbit/s
           ACK 信用 / 最佳单路 = 108.0%，CPU 45.7%（NoQ 24.4%）

1102/2202：最佳单路 11.50，NoQ 15.15，ACK 信用 12.02 Mbit/s
           ACK 信用 / 最佳单路 = 104.5%，CPU 48.0%（NoQ 27.1%）
```

两轮首次数据分别为 `1,148,187 / 948,973 B` 与 `1,093,420 / 1,003,740 B`，仍接近 `1:1`；唯一确认量也接近该比例。救援次数均为零，所以系统没有卡在信用耗尽，而是完成驱动本身在 2 MiB 内没有形成真实 `8:25` 的服务比例。

原因是“完成量”仍不是无条件的容量信号。对路径 `i`，短窗口内观察到的完成量近似：

```text
D_i(T) = min(F_i(T), C_i × T) - startup_i(T) - loss_i(T)
```

其中 `F_i` 仍由调度器决定。初始信用接近、慢启动尚未收敛且 RTT 不同时，`D_i` 会同时包含投喂上限、ACK 返回时延和启动损失。把 `D_i` 原样归还并不能证明 `F_i` 已足够大到暴露 `C_i`，因此比 ACK-ECF 少了一层显式速率估计，却没有消除“未饱和路径容量不可辨认”的根问题。

此外 v1 每生成一个数据报都重新扫描、排序和选择路径，破坏了 NoQ 原有 GSO 批处理，CPU 接近翻倍。增加有界调度量子可以修复这项实现成本，但不会自动把 `1:1` 变成 `8:25`，所以不能单独作为算法 v2。

原始证据 `benchmark-results/2026-07-12-ack-credit-v1-heterogeneous-smoke-2.csv` 为 37 列、9 行，SHA-256 `fcef37b41729828141a94e9aa8f215e071fec71e50e2c84029fb58e1e6570b6c`。该候选按预注册条件未晋级五种子或长测，源码实现已经删除。下一次 B 实现必须使用独立名称、独立预注册文件，并优先解决物理可观测性；当前首选是分离 veth/双网络命名空间的 ingress 测量场。

### 分离 ingress v2：可观测性前置条件已经成立

新的测量场把发送端与接收端放进不同网络命名空间，两条路径使用独立 veth，并在接收侧同时记录接口总 ingress 与三个按流 UDP 计数。v1 因 TBF 峰值桶只生成 `6.841/15.673 Mbit/s` 且留下跨阶段积压而失败；不覆盖 v1 后，v2 去掉峰值桶并把 netem 叶子队列限制为 256 包，保持所有负载与门槛不变。

v2 的三个真实状态为：

```text
欠载：O = 3.997 / 12.497，Y = 4.039 / 12.635 Mbit/s
过载：O = 11.957 / 37.486，Y = 8.009 / 25.028 Mbit/s
交叉：单流 6.012，交叉流 18.718，总 ingress 25.006 Mbit/s
```

这里 `O_i` 是发送投喂，`Y_i` 是接收入口服务量。实测确认：

```text
O_i < C_i  => Y_i ~= O_i       应用受限，不能估 C_i
O_i > C_i  => Y_i ~= C_i       持续饱和后，服务上限可辨认
单流 y_i < C_i 但 Σy_i ~= C_i  交叉流量下不能把单流完成量冒充容量
```

这没有神奇地让普通 ACK 变成容量传感器；它补上的是资格条件。真实 MPQUIC 下一候选必须在已有 2 秒预热内用首次应用 STREAM 数据有界填满每条路径，并同时证明该路径有持续投喂、拥塞/排队压力和新确认进展。只有这种窗口的完成量才可以更新权重；应用受限或调度器没有喂满的窗口只能报告下界。预热后的发送还必须使用有界量子，避免 ACK 信用 v1 的逐包排序破坏 GSO。

### BBR3 最小门控：拥塞控制不能创造调度机会

NoQ 默认 + BBR3 的唯一首种子 smoke 把“控制器”和“调度器”分开了。BBR3 四个单路项相对 Cubic 的保留率为 `118.4%/122.0%/105.5%/162.3%`，说明标准控制器在这些单路样本中没有退化；但 BBR3 双路结果为：

```text
平衡：2.446 / 2.287 = 1.06926，较少路径首次数据 1.155922%
异构：2.515 / 2.476 = 1.01570，较少路径首次数据 2.083079%
```

设路径 `i` 的控制器在被调度后允许发送量为 `A_i(t)`，调度器给它的构包机会为 `G_i(t)`，则实际首次数据仍满足：

```text
F_i(t) <= min(A_i(t), G_i(t))
```

BBR3 能扩大或收紧 `A_i`，却不能在 NoQ 每次从最低 Path ID 开始、首条路即可构造 GSO batch 时自动扩大第二路的 `G_i`。实测第二路只拿到 `1.2%/2.1%` 首次数据，正是 `G_2` 成为主瓶颈。因此读取更多 BBR3 内部状态、调 Startup 参数或延长同一 smoke 都不回答问题，formal 按合同停止。

下一候选 `qualified-service-gso-v1` 不直接估不可辨认的容量上限，而只接受合格窗口的完成量。每个 250 ms 窗口必须同时有首次 STREAM 投喂、cwnd/pacing 压力和全局首次确认进展；样本以固定 `1/2` EWMA 更新。训练和无样本回退按 GSO batch 轮换，已有样本后按 `[10%, 90%]` 有界权重做平滑 batch 调度。这样把“制造可辨认样本”和“使用样本”放进同一闭环，同时把重评分频率从每 UDP 包降到每个最多 10 段的 GSO batch。

唯一 smoke 否决了这个资格定义。平衡场景候选只达到最佳单路的 `108.1%`；异构场景服务率估计为线路一/二 `175456/19650 B/s`，权重随即反向收敛到 `89.9%/10.1%`，候选吞吐仅 `1.741 Mbit/s`，是最佳单路的 `68.6%`。较少路径仍有 `15.3%` 首次数据，说明失败不是完全没探索，而是错误服务率主动把大部分 batch 给了慢路。

数学错误有两层。首先，pacing block 满足的是控制器的时间约束：

```text
t < next_send_i  =>  暂时不可发
```

它不推出 `offered_i > C_i`，因此不能替代 ingress v2 中 `O_i > C_i` 的饱和证据。其次，若在到达窗 `k` 统计 ACK，观测量实际是多个发送 cohort 的卷积：

```text
A_i[k] = Σ_j F_i[j] × P_i(ACK 落入 k | 发送于 j)
```

不同 RTT 会改变条件分布 `P_i`。把 `A_i[k] / 250 ms` 当作发送窗 `k` 的服务率，会把确认相位和前窗在途数据误归到当前窗；50 ms 路径因此可以在相同真实服务下表现得比 15 ms 路径更差。下一传感器必须给首次 STREAM 字节绑定发送 cohort，只把全局首次 ACK 归还到原 cohort，并在真实 cwnd full 后做有界 drain 再结算。v1 证据 SHA-256 为 `0eda38fc88b61125362f7389a33abbccc85e471f1b8854fff6798bf22dcabe2b`，代码已删除。

`cwnd-cohort-sensor-v1` 已在代码前预注册为纯观测门控。对发送 cohort `k`、路径 `i`，令 `F_i[k]` 为该窗首次发送并绑定来源的 STREAM 字节，`Q_i[k]` 表示同窗是否在仍有 STREAM offered load 时触发真实 cwnd 阻塞，`U_i[k]` 为这些来源字节在窗结束后 `1 s` drain 截止前得到的全局首次确认量。唯一允许进入估计的量为：

```text
eligible_i[k] = (F_i[k] > 0) and Q_i[k]
S_i = sum(U_i[k] * eligible_i[k]) / (250 ms * sum(eligible_i[k]))
```

ACK 的到达窗、返回路径和确认原包还是重传都不出现在公式中；同一字节只在其首次发送来源上贡献一次。唯一 smoke 先用持续单流分别测两条线路，并要求 `S_i` 与同项端到端吞吐相差不超过 `15%`、平衡 `min/max >= 85%`、异构快路/慢路至少 `2.0x`。在该门控通过前，`S_i` 不能成为任何调度权重。

唯一 smoke 否决了资格变量 `Q_i[k]` 的完备性。四项业务吞吐为平衡 `15.504/14.731 Mbit/s`、异构 `8.022/24.134 Mbit/s`，说明持续单流已经把线路服务事实跑出来；但只有平衡线路一出现 cwnd-full，其他三项全部为零。逻辑上只能推出：

```text
cwnd_full_i[k]  =>  offered load 曾超过当时 cwnd 许可
not cwnd_full_i[k]  !=>  application limited
```

Cubic 也有 pacing/ACK-clock 稳态；线路可以持续满速而 `bytes_in_flight < cwnd`。因此 cwnd full 是饱和的充分证据，不是必要证据。下一传感器若要覆盖 paced 路径，必须把“应用/STREAM 队列在整个观测区间持续有界积压”作为独立 offered-load 证明，再用发送 cohort 的唯一确认量描述控制器实际服务；pacing 只能证明服务时钟在工作，仍不能单独推出物理 `C_i`。本轮证据 SHA-256 为 `c6e7af58010893c027f92c83fc2be535b2c5691a3f24e25d25639b54e0d4861a`，代码按合同删除。

`persistent-backlog-cohort-v1` 已在代码前预注册。把每个 `250 ms` 发送 cohort 分成四个等长分区，令 `B_i[k,q]` 表示路径 `i` 在分区 `q` 的正常数据构包检查中至少一次观察到 STREAM 待发，资格改为：

```text
eligible_i[k] = (F_i[k] > 0) and product(q=0..3, B_i[k,q]) = 1
S_i = sum(U_i[k] * eligible_i[k]) / (250 ms * sum(eligible_i[k]))
```

四分区覆盖是 offered-load 证据；pacing/cwnd 不出现在乘积中。它不等于物理容量 `C_i`，而是“在持续积压下，当前拥塞控制和网络共同提供的实际服务”——这正是后续调度可消费、又不需要猜未观测容量的量。唯一 smoke 仍先用独立单路业务吞吐作为外部事实校验，未通过前不得进入双路权重。

唯一 smoke 证明这个乘积仍是内生筛选。平衡线路二只有 `5/20` 个 cohort 满足四分区覆盖，入选窗口服务率为 `19.975 Mbit/s`，但完整业务时间平均只有 `14.523 Mbit/s`；ACK 覆盖率为 `100%`，所以偏差不是确认丢失，而是：

```text
E[U_i[k] | product(B_i[k,q]) = 1] != E_time[U_i[k]]
```

`B_i` 与发送突发、协议缓冲补货和服务结果相关，按它过滤会偏向高活动窗口。异构线路一/二仍测得 `7.913/24.069 Mbit/s`，说明来源归属是有信息的，但资格不能用于无偏时间平均。下一门控必须用外生、预先声明的 backlogged epoch；所有完整内部 cohort 都进入分母，不能依据任何本窗网络/协议结果删样本。证据 SHA-256 为 `ab20e8eb7fb08a5f70d89883f79364e75f5cf18335602a0a0d7e7493f7de6130`，代码按合同删除。

`declared-backlogged-epoch-v1` 因此在代码前固定为无条件时间平均。应用在任何 cohort 观测前声明 `E=[t0,t0+5 s)`，并让一个确定性无限记录 writer 在整个区间保持未完成。令 `K={0,...,19}` 为由 `t0` 锚定的全部 `250 ms` cohort，`F_i[k]` 与 `U_i[k]` 仍分别表示首次发送来源字节和固定 `1 s` drain 内的全局首次确认字节，则：

```text
P(k in K) = 1                         for every declared cohort
S_i = sum(k in K, U_i[k]) / (|K| * 250 ms)
```

这里没有随机资格变量，也没有 `F_i[k]>0` 条件。即使某窗没有发送或确认，它仍以零贡献保留在相同分母中；writer 是否完整存活只决定整项实验有效与否，不能改变 `K`。因此估计回答的是“在应用预先承诺持续投喂的这段墙钟时间里，路径实际完成了多少原来源服务”，而不是“挑出协议看起来最忙的窗口后，其条件均值是多少”。唯一 smoke 仍要求 `S_i` 与同项应用吞吐相差不超过 `15%`、平衡两路 `min/max>=85%`、异构快/慢路至少 `2.0x`；通过前不得进入双路权重。

唯一 smoke 的四项 `S_i/业务` 为 `100.0%/100.1%/93.5%/100.7%`，20 个 cohort 全部进入分母且确认覆盖率均为 `100%`；平衡服务率比 `1.000x`，异构快慢比 `2.921x`。因此这一轮没有再观察到 `E[U|内生资格]` 的条件偏差，证据 SHA-256 为 `44de34a0fca26f72791359260c8b88cd51b4501085fb58c483660b68e21a7f2a`。

下一候选 `staged-fresh-service-gso-v1` 使用固定训练设计矩阵而不是随机资格。令八个 `250 ms` 槽的路径指示为：

```text
z = [1,2,2,1,2,1,1,2]
T_i = sum(fresh bytes in slots where z=i) / 1 s
w_i = clamp(T_i / (T_1 + T_2), 0.10, 0.90)
```

每条路径的暴露时长严格相同，槽集合在任何网络结果前固定；阻塞槽保持零贡献而不是补样。测量期冻结 `w_i` 并只在 GSO batch 边界做有界平滑加权轮转。这个设计需要同时通过训练方向和最终吞吐门槛：仅测出 `T_2>T_1` 不够，候选还必须在两场都达到最佳单路 `115%`、两路各至少 `10%` 首次数据和 `<=1.10x` 线速。

唯一 smoke 表明“外生”只保证取样不依赖本窗结果，不保证过程已经平稳。平衡 `T_1/T_2=16.815/18.925 Mbit/s` 尚可；异构只有 `8.801/10.991`，而同轮独立单路稳态为 `7.077/23.114`。短槽观测可写成：

```text
T_i = (1 / 1 s) * integral over fixed slots R_i(t, cc_state_i(t), ack_phase_i(t)) dt
```

它是固定训练日程下的瞬态平均，不是 `lim_{T->steady} integral R_i(t)dt/T`。Cubic 状态和 50 ms 反馈相位跨槽演化，快路四个离散 250 ms 槽没有达到稳态，因此 `T_2/T_1=1.249x` 而不是至少 `2x`。候选最终只到 NoQ 的 `93.8%`，按合同失败。证据 SHA-256 为 `eeb7a38564d6f087021606d8e65a5fbdb0e24c2ad78a3cc3628e0deaa134bc64`，不得通过延长同名训练修补。

反而，同一公平连续 writer 中的 Cubic NoQ 基线在平衡/异构得到 `27.757/30.821 Mbit/s`，说明在持续单流而非逐请求/响应负载下，默认 work-conserving 路径遍历可能已经满足 B 的首种子绝对门槛。这个观察仍需独立五种子合同验证，不能与失败候选混成结论。

因此下一步不再估计容量或改变权重，而是隔离负载模型。固定的 `cubic-noq-continuous-formal-v1` 令接收端完成记录数为 `N(t)`，记录载荷为 `L=16 KiB`，测量窗为 `[t_0,t_1)`、`t_1-t_0>=20 s`。稳定吞吐只定义为：

```text
G = 8 * L * |{record r : receive_complete(r) in [t_0,t_1)}| / (t_1 - t_0)
```

发送端 `write_all` 返回、发送缓冲入队、epoch 外 drain 和最终 FIN 都不进入 `G`。这避免把“已经交给本地 QUIC”误当“测量窗内已经交付给对端应用”。同一条 STREAM 在 `t_0` 前已经持续运行 2 秒且在 `t_1` 时仍未完成，所以比较对象是稳态单流服务，而不是重复事务的启动时间。

每轮聚合判定为 `G_mp / max(G_1,G_2) >= 1.15`，同时以测量窗内逐路径首次 STREAM 字节 `F_i` 要求 `min(F_1,F_2)/(F_1+F_2) >= 0.10`。线速护栏为 `UDP_bytes/(L * completed_records) <= 1.10`。每个场景必须至少 `4/5` 轮三项同时成立；两个场景通过后才进入 Hysteria B 对照。该门控不引入任何调度自由度，因此若通过，最小解释是“默认 NoQ 在持续积压单流合同下已经满足 B 自身绝对门槛”；若失败，才有证据继续研究新调度，而不是继续修补短训练估计器。

正式结果验证了负载模型假设，但也保留了默认调度的真实方差。平衡五轮全部通过，双路中位 `26.580 Mbit/s`、逐轮相对最佳单路中位 `1.885`。异构五轮的相对比为约 `0.808, 2.083, 1.199, 1.207, 1.328`，因此恰好 `4/5` 通过，中位 `1.207`。第 1 轮失败说明默认遍历并非容量最优；合同通过说明在固定种子分布和持续单流语义下，它仍达到预注册的稳定性门槛。两场较少路径份额最差分别为 `42.4%/23.4%`，线速均接近 `1.02x`，所以通过不是靠退化单路或异常重传获得。证据 SHA-256 为 `674d3591c99c805d4ac4fee9187c95637a2a4a172cdb7bfb658eda583a65b4df`。

Hysteria B 对照把最终相对比锁定为平衡 `26.580/13.664=1.945`、异构 `27.509/23.336=1.179`，两个场景都严格大于 `1.15`。若定义额外线速率 `E=max(U-A,0)/A`，平衡 FlowWeave/Hysteria 为 `0.0220/0.0241`，异构为 `0.0254/0.0311`；两场均满足 `E_F <= 2E_H`。因此 B 的吞吐优势与线速约束同时成立，正式对照 SHA-256 为 `34fbada609bdf4449377bbd819f7ffbd9117bfd769e4cf94c967c379f95007bd`。

## C：先用概率算清最小方案

### 直接双发为什么在可靠性上足够

锁定场景中两路独立丢包率为 `8%` 和 `2%`。同一批数据各走一份时，两份同时丢失的概率是：

```text
0.08 × 0.02 = 0.0016 = 0.16%
```

理论有效到达率为：

```text
1 - 0.0016 = 99.84%
```

它高于 `99.5%` 阶段门槛，并且完成时间取两路中先到的一份：

```text
T = min(T_path1, T_path2)
```

真正的风险不是可靠性，而是 200 字节小消息的包头开销可能把两份副本推到 `2.2×` 线速上限之外。

### 批处理可以把包头摊薄

设：

- 每条应用消息 `M = 200` 字节；
- 每批 `g` 条消息；
- 每条消息在 QUIC DATAGRAM 帧中的小额固定开销为 `F`；
- 每个实际 UDP 包的其余固定开销为 `H`；
- 整批在两条路径各发送一次。

总流量比例约为：

```text
ratio = 2 × [g × (M + F) + H] / (g × M)
```

以保守的 `F ≈ 3`、`H ≈ 60` 字节估算，`g = 4` 时：

```text
ratio ≈ 2.18
```

四条消息在每秒 100 条的负载下，最多增加约 `30 ms` 的凑批等待，仍给 `150 ms` 的 P95 门槛留下充足余量。

这只是设计估算。后续实测给出了一个重要反例：四消息 v1 的真实比率为 `2.210795×`，说明约 `2.18×` 的近似不足以判定通过，最终必须以 NoQ 实际 UDP 发送字节计费，不能用公式替代测量。

### 为什么不先上最省流量的 `(2,3)` XOR

可以把两条原消息和一条 XOR 校验分别作为三个包发送。若它们的丢包率为 `8%`、`2%`、`2%`，至少丢两个包、导致整组无法完整恢复的概率约为：

```text
0.3536%
```

平均值看似也能达到 `99.5%`，但余量很窄。按“坏组里的两条消息都算丢失”的保守模型，单次 60 秒实验越过 `0.5%` 丢失线的概率约为 `7.28%`。它适合以后作为省流量备选，不适合作为第一候选。

### C v1 的固定边界

- 每四条 200 字节消息组成一批，批内保留每条消息的序号和产生时间。
- 同一批明确指定两条不同路径各发送一次，不能让默认调度碰巧把两份都放在同一路。
- 第一份正确副本恢复四条消息；后一份只计作重复，不再次交付。
- 批次超过截止时间后丢弃并记作迟到，不能靠无限等待提高到达率。
- 两份副本都必须受各自路径的拥塞窗口和 pacing 约束。
- 先测实际包头与批大小；如果四条一批仍超过 `2.2×`，候选直接失败，不偷偷改计费口径。

### C v1 失败与五消息 v2 的实测闭合

四消息 v1 使用固定种子 `1101/2201` 运行 10 秒，共 1000 条逻辑消息：

- 有效 996、迟到 0、丢失 4，有效到达率 `99.600%`；
- P95 `63.568 ms`、P99 `73.551 ms`；
- 两条路径各发送 250 个 DATAGRAM，格式、序号、时间戳、摘要和内容错误均为零；
- UDP 总字节 `442159`，逻辑应用字节 `200000`，比率 `2.210795×`。

前三项达到绝对门槛，但最后一项严格超过 `2.2×`。因此 v1 按预注册合同否决，没有运行正式矩阵，也没有通过四舍五入、删字段或换种子挽救。证据 `benchmark-results/2026-07-13-c-batched-duplication-v1-smoke.csv` 的 SHA-256 为 `156c63e33ff7b14d9fdf7034577fd0532584a5e8626140ae70f9334968aca78c`。

v2 只把批大小从四条增至五条，不改变两路精确落位、网络条件、消息速率、消息大小或计费口径。每批 DATAGRAM 应用负载固定 `1048` 字节，最大凑批等待从 `30 ms` 增至 `40 ms`。烟测以有效到达率 `99.500%`、P95 `74.483 ms`、P99 `85.227 ms`、UDP 比率 `2.192455×` 通过。

随后五组固定种子、每轮 60 秒的正式矩阵逐轮通过：

```text
有效到达率：99.833%、99.750%、99.833%、100.000%、99.833%
P95：        72.235、72.290、72.802、72.509、72.615 ms
P99：        83.706、83.421、83.372、83.652、83.735 ms
UDP 比率：   2.164510×、2.161001×、2.163358×、2.162185×、2.164507×
```

有效到达率中位 `99.833%`、最差 `99.750%`；P95 中位 `72.509 ms`、最差 `72.802 ms`；P99 中位 `83.652 ms`、最差 `83.735 ms`；UDP 比率中位 `2.163358×`、最高 `2.164510×`。每轮两条路径各发送 1200 个 DATAGRAM，五轮迟到和所有错误均为零；30000 条逻辑消息中有效 29955、丢失 45。

正式证据 `benchmark-results/2026-07-13-c-batched-duplication-v2-five-message-formal-5.csv` 的 SHA-256 为 `e4c8e8726a23c35ff8e93906be4dd6193d0e49044b7a5f9f42fab5f958209c56`。数学上，增大 `g` 摊薄固定包头的方向被实测确认；工程上，v1 的失败也证明必须保留严格的不等式和实际线速计费。

Hysteria 2.9.3 的正式对照进一步暴露了批处理目标的冲突。对手最低五轮总丢失为 567，FlowWeave 的 45 少 `92.063%`；但对手最低 P95 中位只有 `29.657 ms`，要求 FlowWeave `<23.726 ms`。五消息 v2 的 `72.509 ms` 比对手高 `144.492%`。因此 `g = 5` 解决了固定包头，却把最多 40 ms 的凑批等待变成最终相对门槛无法接受的下界；下一步不能继续增大 `g`。

### 低等待 v3：局部快速方程与全局可靠性方程分层

逐消息双发把载荷推到 `2×` 后，只剩 `0.2×` 给所有 QUIC/UDP 包头，真实短包很难过线。更合适的做法不是让每条消息都有第二份完整副本，而是让一个校验方程同时保护多个消息，并把“赶上 P95 的快速方程”和“300 ms 内保证最终有效率的全局方程”分开。

预注册的 `pair-xor-global-10-2-v3` 对每 10 条消息发送：10 个线路一原件、5 个线路一成对 XOR、2 个线路二 GF(256) 全局校验。编码载荷因此为 `1.7×`；固定 20 字节候选头后为 `1.87×`。用 v2 正式实测的每数据 DATAGRAM 约 `33.09` 字节其余成本外推，10200 个独立 datagram 的总比率约为：

```text
(10200 × 220 + 10200 × 33.09) / 1200000 ≈ 2.151
```

这个值只用于决定值得做一次门控，不能替代实际线速。

设线路一丢包 `p = 0.08`。一对两个原件加一个 XOR 后，剩余未知消息数的概率为 `0.981824 / 0.011776 / 0.0064`，分别对应 0、1、2 个。五对合成 10 条块，再由线路二两个独立、到达率 `0.98` 的全局方程补足。保守地在未知数多于方程时不做部分猜测，枚举可得预计丢失率 `0.0854919%`，每 6000 条约丢 5.13 条；卷积 600 个块后，单轮超过绝对门槛 30 条的模型概率约 `4.44e-6`。

P95 只依赖线路一。每对第二条消息在原件丢失时，只要第一条原件和 XOR 到达就能立即恢复；因此快速概率为：

```text
0.92 + 0.5 × 0.08 × 0.92² = 0.953856
```

除以预计最终有效率 `0.999145` 后，快速消息占 `95.4672%`。这个余量足够把候选送入真实门控，但不足以掩盖实现抖动：先测单条立即原件的真实 P95，若不严格 `<23.726 ms` 就停止；随后再测精确 v3 日程的真实线速与独立构包，只有两项都通过才实现在线解码和正式矩阵。

真实门控否决了该候选。v3 的 1700 个帧全部独立构包、精确落位且零错误，UDP 比率 `2.180400×` 通过；但主路原件 P50 已为 `24.238 ms`，P95 `998.867 ms`、P99 `2000.550 ms`，10 秒负载实际耗时 `97.437 s`，CPU `96.144%`。因此失败发生在在线解码之前的“小包 + 8% 丢包 + 当前发送/独立构包确认实现”基线，概率模型没有把该服务时间排队写进去。高 CPU 表明逐帧轮询确认也可能放大尾部，不能把全部现象直接归因于 Cubic。v3 不进入在线解码；下一轮用新文件只发立即原件，对 Cubic/BBR3 使用普通异步背压，先隔离基础拥塞控制与发送节奏。

隔离门控证明基础路线仍可行：Cubic 原件 P95 `23.062 ms`，BBR3 `22.274 ms`，CPU 都为 `3.393%`，两者严格过线；BBR3 的 P99 也从 Cubic 的 `34.313 ms` 降到 `23.381 ms`。因此下一候选保留原 `10 + 5 + 2` 方程结构，选择 BBR3，并把“独立丢包”实现为协议构包边界而不是应用层忙等。该边界只改变应用 DATAGRAM 是否共用一个 QUIC 包，不改变拥塞控制账本或线速计费。

完整 v4 烟测随后给出了这个联合设计的真实代价。1000 条消息全部有效，丢失、迟到、重复和错误均为零；UDP 比率 `2.144800×`、P99 `119.662 ms`、CPU `3.393355%`，说明恢复自由度、线速预算、独立包边界和非忙等实现都按设计工作。但 P95 为 `33.142 ms`，相对严格目标多 `9.416 ms`，因此候选按合同停止，未运行五种子正式矩阵。

把它与同种子的 BBR3 原件门控比较：

```text
                         P50       P95
只发立即原件            20.860    22.274 ms
完整 10 + 5 + 2 日程     21.039    33.142 ms
增量                     0.179    10.868 ms
```

中位附近几乎不变而 95 分位明显上移，说明失败不是固定的 10 ms 编码等待，也不是 v3 那种全局忙等；额外校验、独立小包、8% 主路丢包下的拥塞/排队以及部分消息等待恢复共同作用在尾部。单轮结果不能唯一分解这些因素，但预注册判断不需要这种事后分解：作为整体，v4 没有保住基础门控仅有的 `23.726 - 22.274 = 1.452 ms` 余量。今后若提出 C v5，必须先给出降低这 `10.868 ms` 尾部增量的可检验机制，而不是只继续提高解析恢复概率。

### C v5-v12：从物理独立性到逐消息 `2-of-3`

v5 把 v4 的五个成对 XOR 换成五份同刻直接副本，独立模型给出的快速到达概率是：

```text
0.5 × (1 - 0.08²) + 0.5 × (1 - 0.08) = 95.68%
```

但 500 个 direct copy 中虽有 448 个正确到达，只有 1 个比对应 original 更早成为首次交付；最终只交付 `968/1000`，P95 `62.979 ms`。这证明“不同 QUIC/UDP datagram”并不自动等于 netem 看到的独立丢失事件：相邻 datagram 仍可能共享一个 GSO skb，并被一起处理或丢弃。

v6 唯一地关闭 segmentation offload，其余 `10+5+2` 日程不变。direct copy 首次交付从 1 次升到 38 次，`1000/1000` 全部有效，说明 v5 的相关同失确有 GSO 这一关键组成；但 P95 `24.641 ms` 仍比门槛高 `0.915 ms`。v7 用 12 字节紧凑头把直接副本增至六成，第一次同时取得 P95 `23.340 ms` 与 UDP `2.193015×`，却只交付 `994/1000`，比有效率门槛差 1 条。数学上的剩余问题由此从“快速集合不够大”变成“每个 10 条块的两个全局自由度不能跨块借用”。

v8 把相邻两块合并为 20 条超块并池化四条 GF(256) 方程，独立模型把预计消息丢失率降到 `0.0038002%`。唯一 smoke 暴露两个必须保留的反例：非零超块 accounting 仍错误乘以历史块长 10，导致除 block 0 外的帧都被拒绝；发送侧 UDP `2.209005×` 也已越线。因此不能把公式正确误当成实现正确。v9 用 10 字节等价头和非零超块确定性测试修正这两点，smoke 以 P95 `23.062 ms`、UDP `2.176575×` 通过，五轮 formal 也只丢 9 条；但第 4、5 轮 P95 为 `25.367/24.509 ms`，说明接近 95% 边界的结构对种子和快速集合尾部仍敏感。

v10 在相同 `32/4` 帧预算下把 12 个直接副本改成 4 个副本加 8 个成对 XOR。独立近似把进入全局层的未知消息从 v9 的 `3.584%` 降到 `1.11104%`，再加需要等待下一条消息才能形成 XOR 的第一成员 `2.70848%`，合计约 `3.81952%`。实际 P95 却为 `26.608 ms`：这个推导错误地把“慢恢复比例低于 5%”当成充分条件，没有给本来已经在 22～23 ms 的快速集合尾部留下余量。

v11 恢复 v9 的编码布局，只让 original 在定向 DATAGRAM 队列里优先于冗余。P95 反而从同种子 v9 的 `23.062 ms` 增至 `25.891 ms`；优先原件会让直接副本更晚获得发送机会，并未减少需要块级恢复的约 `3.584%`。因此专用 priority API、队列字段和测试全部删除，失败机制没有进入产品传输面。

v12 不再把一部分消息设为“有副本”、另一部分设为“只能慢恢复”。每条 200 字节消息立即拆成 Data0、Data1 两个 100 字节数据片，并同时生成 `LocalParity = Data0 xor Data1`；三片都走主路、独立 no-GSO 包，任意两片到达即可交付。主路独立丢失率 `p=0.08` 时，本地成功率为：

```text
P(local success)
= 1 - [3 × p² × (1-p) + p³]
= 98.1824%
```

一条消息进入 20 条超块的未知数据自由度分布为：

```text
P(U=0) = 0.981824
P(U=1) = 0.011776
P(U=2) = 0.006400
```

其中 `U=1` 对应只收到 Data0 或只收到 Data1，`U=2` 对应只收到 LocalParity 或三片全失。对 20 条消息做生成函数卷积，再令备用路三条 Vandermonde 方程各以 `0.98` 概率到达，并保守要求收到的方程数不少于未知自由度才恢复，预计消息丢失率为 `0.133247%`：1000 条约丢 `1.33`，五轮 30000 条约丢 `39.97`。同时只有 `1.8176%` 消息需要离开本地层，明显低于 v9/v11 的约 `3.584%`，因此 P95 不必落到块级全局等待。

五轮正式实测闭合了模型没有覆盖的相关丢包、拥塞控制和排队：

```text
轮次  有效       丢失  P95 (ms)  P99 (ms)  UDP 比率
1     5996/6000  4     23.429    122.038   2.156541×
2     5987/6000  13    23.417    103.430   2.153667×
3     5988/6000  12    23.275    130.585   2.151944×
4     5991/6000  9     22.834    131.514   2.152650×
5     5993/6000  7     22.707    91.991    2.155026×
```

总丢失 `45 <= 283`，P95 中位/最差 `23.275/23.429 ms`，UDP 中位 `2.153667×`，所有错误为零。Hysteria 2.9.3 最强 C 对照 P95 中位为 `29.657 ms`，其固定 80% 门槛是 `23.725600 ms`；v12 为对照的 `78.481%`，即低 `21.519%`。所以 v12 同时通过绝对可靠性、P99、线速、独立性和相对延迟门槛，C 的数学路线在该预注册实验合同内完成。

## 编码知识产权与依赖边界

真正的 RLNC、Raptor、Tetrys 和部分滑窗编码实现存在专利、授权或实现未公开问题。QUIRL 仓库明确说明 Tetrys 实现因知识产权披露而保持私有；其配套 Rust `networkcoding` 仓库也主动标注了相关专利情况。

因此当前决定是：

- 不复制这些实现；
- 不因为论文效果好就立即安装编码依赖；
- v12 只实现固定规模、可逐行审计的 XOR 与 GF(256) Vandermonde 消元，没有引入 RLNC、Raptor、Tetrys 或外部编码依赖；
- 7 字节头中的 16 位 digest 只做分片语义/accounting 校验，网络攻击下的机密性和完整性仍由标准 TLS 1.3 / QUIC AEAD 提供；
- 以后若需要通用 Reed-Solomon、滑窗编码或生产级 FEC，仍须先核实许可证、专利状态、维护情况和 CPU 成本，再用独立候选、依赖记录和新结果文件引入。

## 架构决定

三个目标都已通过，但不能因为最终胜出就塞进一个万能调度器：

```text
A：v6.9 有界跨路径选择性恢复
   解决“路径黑洞后，反馈、旧数据义务与流控怎样在 1 秒内闭环”

B：Cubic + NoQ 默认调度 + 持续积压单流合同
   不输出容量估计，直接由逐路径拥塞控制和现有调度完成异构带宽聚合

C：v12 逐消息 2-of-3 + 40 数据符号 / 3 全局方程
   本地层压低 P95，全局层在 300 ms 内补最终可靠性，全部冗余仍受逐路径拥塞控制
```

它们共享 NoQ MPQUIC、标准 TLS 1.3、路径状态和统计接口，但状态机、指标与失败条件保持分离。A 的 ACK 不能给 B 发新数据信用；C 的冗余片不能被统计为 B 的首次有效聚合；产品代理也不默认启用 C 的实验实时编码。底层偏离集中在已记录的定向 DATAGRAM、恢复和可观测接口，产品入口通过公开封装调用，后续可逐项替换 NoQ 而不改代理授权或应用协议。

## 当前证据与部署边界

1. A v6.9、B 持续单流、C v1-v12 与 Hysteria A/B/C 的历史 CSV、固定种子、哈希和门槛全部冻结；失败候选不补行、不挑种子、不在原名称下修补。
2. 当前胜出路径只使用 A 恢复、B 默认调度、C v12 编解码和最小固定目标代理；v1-v4 的版本化复现入口仍保留，v5-v11 的失败候选专用实现与 v11 DATAGRAM priority 能力已经清场。
3. 最小代理完成标准 TLS 1.3、证书名称校验、文件令牌、固定目标、双向半关闭、并发隔离和 systemd 部署材料，但它仍不是 TUN、SOCKS5、UDP 转发或动态目标代理。
4. 实验室结果证明预注册拓扑和持续时间内的相对胜出，不证明真实公网双接口的长期 soak、跨版本升级、运营监控或生产 SLA；这些属于部署后的独立验收。

因此当前数学结论不是继续寻找第四套调度器，而是冻结三条已通过的最小机制，并把后续工作限定为实现回归、真实部署观测和不扩大安全边界的可替换性维护。
