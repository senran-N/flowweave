- 版本：Hysteria `2.9.3`；
- 官方标签：`app/v2.9.3`；
- 上游提交：`2d973f9513ef661d1922d6d14acb37945caef47d`；
- SHA-256：`66dbdb0608f25f3057b433afe975a9fc1af2ca8e512479e294988b3ef363d6c1`。

实验入口每次都从固定缓存路径启动该二进制，并在进入隔离网络命名空间前校验哈希和版本。更新检查与混淆关闭。服务端证书在每场实验的权限隔离临时目录中生成，SAN 固定包含 `localhost`；客户端使用该证书作为专用信任根并校验 `localhost`，禁止 `tls.insecure`。认证密码从系统随机源临时生成，只写入权限 `0600` 的临时配置，进程退出后删除，不进入源码、Git、CSV 或命令输出。

## 公平网络拓扑

现有实验场在一个 rootless 隔离网络命名空间内，用回环地址和两个独立 netem 子队列复现两条线路：

- `127.0.0.1`：线路一，netem handle `10:`；
- `127.0.0.2`：线路二，netem handle `20:`。

Hysteria 是代理程序，测试时还需要应用到本地客户端转发器、服务端到本地目标程序的回环流量。它们不是被比较的公网传输，固定使用 `127.0.0.3` 和 `127.0.0.4`，进入不带 netem 的第三队列。该隔离只排除代理两端的本机接入成本，不排除 Hysteria QUIC 的握手、ACK、重传、保活或控制流量。

netem 的 `Sent bytes` 不包含被 netem 主动丢弃的包，不能直接作为高丢包线速，否则会系统性低估发送量。因此实验在隔离命名空间的 IPv4 output hook 增加只读 nft counter，按 Hysteria 服务端目标 IP 和本场随机 UDP 端口精确匹配客户端到服务端的 QUIC 包。每个计时窗口都在代理通过一次不计分的端到端就绪探针后读取起始计数，结束时再次读取；nft 的 IPv4 总字节减去 `28 × 包数` 得到发送端 UDP 载荷字节，与 FlowWeave 的发送端 UDP 计费口径对齐。目标和非目标线路分别设置计数器，后者必须为零，以证明没有意外串线。

本机应用接入流量由第三队列隔离，且 nft 规则还要求目标 Hysteria UDP 端口，因此 TCP/UDP 转发器、后端程序、证书生成和进程控制流量都不会混入 QUIC 线速。每轮仍重新 `replace` 两个 netem 队列，固定种子并清空历史排队状态。

Hysteria 官方单连接不提供 MPQUIC 式双路径调度。本对照不额外叠加双实例聚合器、应用重连器或第三方 bonding，因为那会把未经预注册的新系统冒充 Hysteria。C 组为了给对手最强单路结果，会把两条线路分别运行；A 组按合同从主线路建立原业务连接，备用线路保持存在，但不替 Hysteria 发明非官方连接迁移层。

## 拥塞控制参赛项

所有 Hysteria 对照都运行两种官方模式：

- `bbr`：不填写带宽，让 Hysteria 使用内置 BBR；
- `brutal`：客户端与服务端填写场景的真实单线路上下行带宽，触发 Hysteria 官方 Brutal 控制器。

除 BBR/Brutal 所需字段外，不为某个种子单独调参。若官方默认参数导致失败，原始结果仍保留。

## B：异构带宽聚合对照预注册

FlowWeave 的 `cubic-noq-continuous-formal-v1` 已通过自身绝对门槛，因此现在固定 Hysteria 2.9.3 的 B 组对照。Hysteria 官方单连接没有 MPQUIC 双路径调度，本实验不增加双实例 striping、应用分片、连接重建或第三方 bonding；否则比较对象会从官方 Hysteria 变成一个新系统。为了给对手最强单路机会，每个场景分别运行两条物理线路。

### 参赛项、带宽和负载

两个场景、延迟、丢包、整形和五对种子与 FlowWeave B 完全相同：平衡为 `15+15 Mbit/s, 20+20 ms, 0.1%+0.1%`，异构为 `8+25 Mbit/s, 15+50 ms, 0.1%+0.1%`。每个场景固定四个参赛项：

1. BBR / 线路一；
2. BBR / 线路二；
3. Brutal / 线路一；
4. Brutal / 线路二。

BBR 不填写带宽。Brutal 的客户端和服务端都填写本参赛项真实单线路上下行带宽：平衡两路各 `15 mbps`，异构线路一 `8 mbps`、线路二 `25 mbps`。不使用两路总和，不按实测结果回填，也不因某个种子改变配置。

每项经官方 `tcpForwarding` 建立一条应用 TCP 连接，只打开一次并贯穿预热、测量和收尾。发送端持续写入单调序号、摘要和固定 `16 KiB` 确定性载荷；接收端逐记录验证。正式先预热 `2 s`，再按接收端完整记录完成时刻计连续 `20 s`；发送端本地入队、测量后 drain、FIN 和最终反向响应都不计入窗口吞吐。目标线路客户端到服务端的 Hysteria QUIC UDP 载荷由既有 nft 计数，非目标线路客户端和服务端计数必须为零。

### 最终相对判定

对每个场景，先分别计算四个 Hysteria 参赛项的五轮吞吐中位数，取其中最高者 `H_best`。FlowWeave 使用已固化同场 NoQ 双路五轮吞吐中位数 `F`：平衡 `26.580 Mbit/s`，异构 `27.509 Mbit/s`。只有两个场景都满足 `F/H_best >= 1.15`，B 才最终胜过 Hysteria 2.9.3。

线速比较使用客户端到服务端 UDP 载荷。每项额外线速率定义为 `E=max(UDP_bytes-application_bytes,0)/application_bytes`，按五轮取中位。FlowWeave 同场中位额外线速率不得超过吞吐最强 Hysteria 参赛项的两倍；若对手的 `E=0`，则退回更直接且保守的总线速条件 `FlowWeave UDP/application <= Hysteria UDP/application`，避免除零。吞吐和线速两个相对条件都通过，才称 B 最终胜出；不能选择不同 Hysteria 参赛项分别弱化两个门槛。

## C：高丢包实时传输对照

### 消息与计时

网络、种子、时长和门槛完全沿用 `BENCHMARK.md`：线路一 `20 ms / 8% / 20 Mbit/s`，线路二 `40 ms / 2% / 20 Mbit/s`；每秒 100 条、每条恰好 200 字节，正式每轮 60 秒。

Hysteria 使用官方 `udpForwarding`。每条 200 字节应用消息内部包含固定 magic、全局序号、可复算摘要和确定性内容；元数据占用这 200 字节之内，因此不会给 Hysteria额外增加应用字节。发送端和接收端在同一 Rust 测量进程中使用单调时钟，避免跨进程时钟误差。第一份内容正确且自产生起不超过 300 ms 的消息计为有效；更晚的第一份正确消息计为迟到；后续同序号正确消息计为重复；格式、序号、摘要和内容错误分别计数。

四个固定参赛项为：

1. BBR / 线路一；
2. BBR / 线路二；
3. Brutal / 线路一；
4. Brutal / 线路二。

每轮按轮次旋转参赛顺序，不能总让同一模式先跑。正式比较使用对 Hysteria 最有利的逐指标结果：

- 延迟基线取四个参赛项中最低的“五轮 P95 中位数”；FlowWeave 正式 P95 中位数必须至少低 20%；
- 丢失基线取四个参赛项中最低的“五轮总丢失消息数”；FlowWeave 五轮总丢失必须至少少 50%。

这允许不同 Hysteria 参赛项分别成为延迟和丢失的最强对手，避免用一个综合排名掩盖它在某项指标上的最好表现。

- 丢失：Hysteria 最低五轮总丢失为 BBR / 线路二的 567；FlowWeave v2 为 45，减少 `92.063%`，通过“至少少 50%”；
- P95：Hysteria 最低五轮 P95 中位为 BBR / 线路一的 `29.657 ms`，FlowWeave 若要低 20% 必须不高于 `23.726 ms`；当前 `72.509 ms`，比 Hysteria 高 `144.492%`，明确失败。

即使不采用逐指标最强、只和丢失最少的 BBR / 线路二比较，FlowWeave 也必须把 P95 压到 `39.569 ms` 以下，当前仍失败。因此结论不依赖“不同 Hysteria 参赛项分别取最优”的保守规则。

FlowWeave 五消息批量双发 v2 仍然通过自身 C 组绝对门槛，但没有通过 `BENCHMARK.md` 的最终 Hysteria 相对延迟门槛。Hysteria C 基线现已闭合，不需要为了挑结果重跑。下一 C 候选必须使用新名称和新 CSV，并以 P95 `<23.726 ms`、五轮总丢失 `<=283`、UDP 比率 `<=2.2×` 作为由当前证据导出的联合目标；不能只继续增加批大小或冗余。

## A：换网不断线对照

### 业务语义

沿用 FlowWeave 正式 A 组的线路：线路一 `20 ms / 0.1% / 20 Mbit/s`，线路二 `80 ms / 1% / 20 Mbit/s`。应用建立一条经 Hysteria `tcpForwarding` 的双向 TCP 连接，持续 30 秒；第 10 秒只把线路一改成 `100%` 丢包，不主动关闭 Hysteria 或应用连接。正向和反向分别运行 BBR、Brutal 与五组固定种子。

每条业务记录包含连续序号、摘要和固定 16 KiB 内容。接收端逐条校验，记录故障后的最大相邻正确记录到达间隔。只有同时满足下列条件才算 Hysteria 原业务连接不断：

- 始终使用同一个应用 TCP 连接，没有重连；
- 故障后数据恢复到达；
- 序号、内容、总记录数和最终响应全部闭合。

TCP 文件描述符仍未关闭但故障后不再传输，不能算“不断线”。若 30 秒业务窗口结束仍未恢复，记录为未恢复；不额外延长窗口直到对手偶然重连。

### A 比较与文件

在写 A 组证据 CSV 前，允许固定运行一次 `BBR / 正向 / 3 秒业务 / 第 1 秒黑洞` 的接线诊断，只验证同一 TCP 连接、记录校验、黑洞回调、双向 nft 计费和进程清理；不输出性能 CSV，也不参与正式结论。

若 Hysteria 任一模式在一个方向五轮都无法保持并恢复原业务连接，而 FlowWeave 对应方向正式结果全部不断，则按 `BENCHMARK.md` 的连接连续性分支判定。若双方都五轮不断，则 FlowWeave 对应方向恢复间隔中位数必须至少低 30%。BBR 与 Brutal 都公开，最终结论按更强的 Hysteria 模式计算。

### A 对照烟测结果：实验器通过，四场均未恢复

第一组种子的正向/反向 × BBR/Brutal 四场都通过基础设施门槛：故障前均已连续校验超过 1400 条 16 KiB 记录；格式、序号、摘要和内容错误为零；客户端与服务端进程在窗口结束时仍存活；故障后只有线路一有 QUIC 流量，线路二严格为零。

## 失败与回退

- 若 Hysteria 二进制哈希不符，立即拒绝运行，不能改用其他版本。
- 若本机接入流量进入线路一或二计费，当前结果无效，修复实验场并使用新文件名。
- 若证书只能通过 `insecure` 才能连接，当前接线失败，禁止继续跑分。
- 若 Hysteria 性能本身没有达到 FlowWeave 的绝对门槛，仍照常完成固定矩阵；不能因为对手表现差而删轮次。
- 所有临时进程、证书、密钥和配置必须由 RAII 清理；异常退出时也不能留下可用代理入口。
