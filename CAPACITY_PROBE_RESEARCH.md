# 路径容量传感器论文审计与失败实验

## 真正要解决的问题

ACK-ECF 失败后，问题已经不是“调度公式还差哪个权重”，而是：

> 在不免费预热、不过度挤占线路的前提下，怎样知道每条路径最多还能稳定交付多少数据？

普通 ACK 速率只能看到调度器已经喂给某条路径的流量。路径未被喂满时，这个数只是容量下界；若拿它反过来决定下一轮投喂比例，就会形成自证循环。因此本实验先把容量测量从 NoQ 和正式 MPQUIC 调度中拆出来，独立验证“传感器”本身。

提前锁定的初筛门槛是：

- 固定五个种子中，25 Mbit/s 路径必须 5/5 次排在 8 Mbit/s 路径之前。
- 每轮估计出的容量比例，相对真实比例 `25 / 8 = 3.125` 的误差不超过 20%。
- 同一轮比较两个候选、两条线路的探测总量不超过 2 MiB 正式短流的 5%。
- 探测耗时、字节、CPU、丢包、乱序和时间戳异常全部计账。
- 读数证据不足时必须拒绝输出，不能用一个看似精确的数字掩盖“不知道”。

## 论文给出的数学工具

### Packet Train：整列包的平均离散度

对背靠背发送的等长包，取第 `i` 与第 `j` 个有效接收时间戳：

```text
C_hat = (j - i) × L × 8 / (a_j - a_i)
```

其中 `L` 是估算的 IPv4 + UDP 包长，`a` 是接收时刻。相比只看一对包，整列平均可以减少单个时间戳噪声；但经典 packet-dispersion 研究已经指出，它更接近瓶颈容量或平均离散速率，不等同于任意交叉流量下的可用带宽。

### Chirp：从排队曲线拟合容量

Chirp 让相邻探测速率按指数上升。第 `i` 个包的相对排队延迟为：

```text
q_i = (a_i - a_0) - (s_i - s_0)
```

不需要两端时钟同步，只需要各自时钟在一个很短的 chirp 内保持稳定。

把候选路径视为服务速率 `C` 的流体队列，理论排队曲线递推为：

```text
Q_0(C) = 0
Q_i(C) = max(0, Q_(i-1)(C) + L×8/C - (s_i - s_(i-1)))
```

第一版没有用脆弱的“第一个上升点”阈值，而是遍历候选容量，选择整条理论曲线与观测曲线均方误差最小的容量：

```text
C_hat = arg min_C Σ_i (q_i - Q_i(C))²
```

这吸收了 pathChirp 的自致拥塞、WebRTC GCC 的收发间隔差，以及 PathKatana 的整条理论曲线最小二乘拟合思想。

### 时间戳与应用受限边界

2026 年 3 月的 `draft-ietf-quic-receive-ts-02` 已为普通 QUIC 和 MPQUIC 定义多包接收时间戳，并明确点名 pathChirp 与 GCC。草案同时警告：用户态调度导致时间戳不准时应该省略样本，而不是假装精确。

2026 年 BBR 草案把交付率限定为：

```text
delivery_rate = min(send_rate, ack_rate)
```

并明确标记 application-limited 样本。BBR 只有在主动提高发送率、连续三个非应用受限轮次增长不足 25% 后，才认为已经填满路径。这说明可靠容量样本必须伴随“确实尝试过填满”的状态证据。

## 实验实现

实验曾实现两个完全独立于 NoQ 的 UDP 候选：

- 每条路径 16 个、每个 1200 字节 UDP 载荷的背靠背 Packet Train。
- 每条路径 16 个同尺寸包的指数 Chirp，初始 2 Mbit/s，相邻速率倍率 1.25，最高约 45.5 Mbit/s。
- 每个候选同时测两条路约 39,296 个估算线速字节，占 2 MiB 的 1.87%；两个候选合计约 78,592 字节，占 3.75%。
- 先使用用户态时间戳，随后改为 Linux `SO_TIMESTAMPNS` 内核接收时间戳。
- 记录丢包、乱序、连续小于 30 微秒的批处理时间戳、Packet Train 包间隔变异系数，以及 Chirp 的 RMSE、归一化拟合误差和排队信号。
- 数学极值、缺失时间戳、没有形成排队信号和 8/25 Mbit/s 合成数据均有确定性测试。

这些测试只能证明代码正确实现了所写模型，不能证明真实网络服从该模型；真实五种子实验正是用来检查这个差别。

## 四轮真实诊断

### 1. 用户态时间戳 + 原有 netem rate

- Packet Train 可用读数 `0/10`，时间戳异常 65 次。
- Chirp 可用读数 `4/10`，时间戳异常 42 次。
- 25 Mbit/s 路径曾被误报为约 69～91 Mbit/s。

结论：接收线程一次取出多个已排队 UDP 包时，程序看到的时间不是包真正到达时间。拒绝机制成功拦截了多数假答案。

### 2. Linux 内核接收时间戳 + 原有 netem rate

- Packet Train 可用读数 `0/10`，时间戳异常 76 次。
- Chirp 可用读数 `3/10`，时间戳异常 31 次。

结论：内核时间戳排除了用户态唤醒延迟，但 `netem rate` 在回环接口上本身会成批释放高速包。这里看到的是模拟器的真实突发，不是时间戳 API 出错。

### 3. 独立 TBF 限速 + netem 只负责延迟和丢包

首次 TBF 配置使用 1600 字节 burst：

- 时间戳批处理降为 0。
- Packet Train 可用读数 `2/10`。
- Chirp 可用读数 `8/10`，快慢排名 `3/5`，容量比例误差门槛 `2/5`。

结论：方向上消除了批处理，但 TBF 的平均桶小于 `rate / 内核时钟频率` 所需容量，25 Mbit/s 被整形器本身压成更低且波动的速率。

### 4. 足量平均桶 + 单 MTU 峰值桶

最后一次把平均 burst 提至 32 KiB，并用只比目标速率高 1 Kbit/s、`minburst=1600` 的峰值桶阻止开局突发。完整逐轮原始数据保存在：

```text
benchmark-results/2026-07-11-capacity-probe-screening.csv
```

结果：

```text
Packet Train：
可用读数 4/10
快慢排名 1/5
容量比例误差门槛 1/5

Chirp 曲线拟合：
可用读数 9/10
快慢排名 4/5
容量比例误差门槛 1/5
容量比例误差中位 49.62%
范围 0.17%～55.58%
```

最危险的证据不是“拟合误差很大”，而是相反：多轮错误答案的归一化拟合误差仍然很小。例如快路被估成 7.88、10.85 或 14.31 Mbit/s 时，曲线仍可以看起来拟合良好。这证明当前置信度只衡量“某条错误模型曲线能否解释观测”，没有衡量“真实路径是否服从这个模型”。

CPU 通过 `/proc/self/stat` 的 10 毫秒级进程时钟记录，因此大量短探针显示为 0；这个精度只足以发现异常高 CPU，不足以比较微小差异。

## 为什么最终失败

1. 16 包微型探针太短，起始令牌、定时器、队列初态和偶发调度延迟足以支配整条曲线。
2. Packet Train 测到的是这一次短列包经历的平均服务，不是稳定可用容量；短列中的一个突发或停顿会严重改变结果。
3. Chirp 的流体 FIFO 模型没有包含令牌桶、成批释放和多级队列。模型失配时，最小二乘仍会给出一个数学上漂亮、物理上错误的最优值。
4. “低 RMSE”不是充分置信度。还必须证明路径在多个独立时间窗口中被受控饱和，并且不同探测方向或不同模型给出的区间相互一致。
5. 本地回环实验适合验证安全护栏和调度相对比较，但不适合单独认证亚毫秒级单向包间隔传感器。

因此两个候选均未通过门槛，不接入 NoQ，不实现 QUIC Receive Timestamps 扩展，也不据此产生第六个调度器。失败实现按项目规则删除，只保留论文推导和最终原始 CSV。

## 后续路线 A：直接读取 NoQ BBR3 的容量证据

2026-07-12 继续验证了一个更保守的方向：不再额外发微型探针，而是只读 NoQ 已有 BBR3 模型。每条 MPQUIC 路径本来就有独立拥塞控制器，BBR3 内部已经维护：

- 窗口最大交付率 `max_bw`；
- 最新 delivery-rate 样本；
- 最新样本是否 application-limited；
- 是否正在 Startup / ProbeBW_UP 主动增加负载；
- 是否通过连续非应用受限轮次完成 filled-pipe 判断。

实验实现了一个不绑定 BBR3 私有状态名的只读 `BandwidthEstimate`：Cubic 与 NewReno 明确返回“没有容量模型”，BBR3 同时报告数值与资格。只有 `full_bw_reached` 为真时，调用者才允许把 `max_bw` 当作合格容量；cwnd 和 pacing rate 仍不能冒充容量。

固定五种子继续使用原来的 2 MiB 正常应用数据，没有额外探针，也没有改变 NoQ 官方选路顺序。原始数据保存在：

```text
benchmark-results/2026-07-12-bbr-capacity-screening.csv
```

结果：

```text
单路 8 Mbit/s：合格 1/5
单路 25 Mbit/s：合格 0/5
单路成对排名：0/5（因为没有成对合格样本）

同一 MPQUIC 连接两路同时合格：0/5
多路径成对排名：0/5
```

未取得资格不等于 BBR3 没有学习。单路原始 `max_bw` 中，8 Mbit/s 线路大约为 9.37～12.29 Mbit/s，25 Mbit/s 线路大约为 15.16～27.83 Mbit/s；多数轮次已经能粗略区分快慢。但这些连接仍处于 Startup，BBR3 没有观察到连续三个非应用受限、增长不足 25% 的平台轮次。若此时为了跑分直接使用原始 `max_bw`，等于删除了本路线最关键的“确实填满路径”证据。

多路径情况更差：默认 NoQ 调度经常让第二条路只承担少量甚至 0 字节首次数据，第二条路的 `max_bw` 可低至约 0.13 Mbit/s。BBR3 能诚实标记“尚未合格”，但不能凭空解决调度器没有给路径足够数据的问题。

这说明：

1. BBR3 的通用只读接口在架构上成立，资格信号也能阻止错误使用。
2. 但 2 MiB 短流在当前 RTT、随机丢包和实现噪声下，通常来不及完成 filled-pipe 验证。
3. 先跑长流再把结果免费送给短流会违反基准合同；短筛失败后也不能改成长测来保留候选。
4. 因此该接口没有通过当前可测目标，实验快照之后应从 NoQ 和 FlowWeave 完整删除，只保留 CSV 与结论。

## 三次停手后的可选出路

### A. 读取 BBR3 已有的受控饱和证据（已验证，短流失败）

NoQ 已内置 BBR3。下一步可先审计并通过一个很窄、可替换的 FlowWeave 适配层读取每条路径的：

- `max_bw` 或等价窗口最大交付率；
- 当前样本是否 application-limited；
- 是否处于 Startup / ProbeBW_UP 等主动探测阶段；
- 样本对应的轮次、丢包和 inflight 状态。

该方向已按上述五种子数据验证。接口能诚实区分“有数值”和“有资格”，但当前 2 MiB 短流中几乎没有路径及时取得资格，因此不能作为下一代调度输入。

### B. 建立真正分离的发送端/接收端测量场

使用 veth + 两个网络命名空间，或两台独立主机，让接收时间戳发生在真实 ingress，而不是同一回环设备；再用多次 chirp、置信区间和交叉流量验证。它更接近论文实验条件，但工程量更大，而且探测字节可能超过短流 5% 预算。

### C. 暂停 B，先推进 A 或 C

当前 B 已连续碰到可观测性墙。可以先解决 A 的约 4 秒黑洞恢复，或 C 的关键 Datagram 双发/FEC，再回到带宽聚合。这样不会浪费已有 MPQUIC 底座，但带宽聚合目标会暂时停滞。
